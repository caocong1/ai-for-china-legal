# 监管合规实践配置文件契约（Practice Profile Schema）

> 本文件定义 regulatory-legal 插件的实践配置文件：**是什么、在哪、技能怎么读**。
> 监管合规版改编自 criminal-compliance/skills/_shared/practice-profile-schema.md，结合监管合规语境特殊化（监管合规立场护栏、不提供逃避监管方法护栏、行刑衔接强制升级契约、监管合规边界契约、行政程序时限契约）。

---

## 一、配置文件是什么 / 真相来源

**唯一真相来源：`regulatory-legal/CLAUDE.md`（散文模板）**

- 本阶段（10-01）**首次新建**，由 `/regulatory-legal:cold-start-interview` 冷启动访谈填充和更新
- 律师/合规官/法务可直接编辑该文件，无需重新运行访谈
- 所有 regulatory-legal 技能从 `regulatory-legal/CLAUDE.md` 读取实践配置，不从其他来源读取

**架构决策（不引入冲突新格式）**：

> 本插件沿用 CLAUDE.md 散文模板作为单一真相来源，不引入独立 JSON 或 YAML 格式的配置文件。理由：
> 1. 散文模板对监管合规律师/合规官/法务/当事人更直观、可读，直接编辑体验好
> 2. 与 commercial-legal / construction-legal / employment-legal / family-legal / criminal-compliance 等插件架构一致，维护成本低
> 3. 避免 CLAUDE.md 与独立 YAML/JSON 文件的双重真相来源冲突

**纠正现有访谈表述**：

现有 `regulatory-legal/skills/cold-start-interview/SKILL.md` 末尾「生成 监管合规 实践配置文件（YAML 格式）」的表述应理解为**填充 `regulatory-legal/CLAUDE.md` 对应章节**，而非另存独立 YAML 文件。技能和用户都应将访谈结果直接写入 CLAUDE.md，不得创建单独的 `practice-profile.yaml` 等文件。

## 三、技能读取契约

### 3.1 标准读取流程

每个 regulatory-legal 技能在执行实质性工作前，**必须**按以下顺序读取配置：

1. **读取 `regulatory-legal/CLAUDE.md`**
2. **检查关键字段是否含 `[PLACEHOLDER]`**：
   - `## 我们是谁` 的「执业角色」字段
   - `## 我们是谁` 的「所属行业」与「主管监管机关」字段（**监管合规核心变量**）
   - `## 合规与监管立场` 的「整体服务立场」字段
   - `## 合规与监管立场` > `### 监管关系` 字段
   - `## 谁在使用` 的「角色」字段
3. **据此选择运行模式**（正常模式 or 临时模式，见下节）
4. **检查行政程序时限**：如识别到处罚应对立场或已收到处罚决定，立即触发时限护栏

### 3.2 配置缺失/含占位符时的标准提示

当 `regulatory-legal/CLAUDE.md` 不存在或关键字段含 `[PLACEHOLDER]` 时，技能输出以下标准提示：

```
> **配置提示**：未检测到完整的监管合规实践配置（regulatory-legal/CLAUDE.md 中关键字段含占位符或文件不存在）。
>
> 如需个性化配置，请运行 `/regulatory-legal:cold-start-interview` 完成冷启动访谈，将结果填入 regulatory-legal/CLAUDE.md。
>
> 或者，告诉我你的服务立场（监管动态监控/行政处罚应对/合规体系建设/监管报告起草）和所属行业，我将以「临时模式」继续当前任务。
>
> **临时模式安全警示**：临时模式下，行刑衔接强制升级护栏与监管合规边界护栏默认最高。如涉及重大处罚、行刑衔接或人身自由风险，请立即联系执业律师。
```

### 3.3 临时模式默认值表

| 配置字段 | 临时模式默认值 | 说明 |
|---------|-------------|------|
| 整体服务立场 | **按用户任务判断**，默认监管动态监控 | 监控是最安全的默认立场；处罚应对须用户明确说明 |
| 执业角色 | 律师（最高权限） | 降低误升级风险；非律师须用户明确说明 |
| 合规严格度 | 保守型 | 临时模式下所有疑点从严标红 |
| 所属行业与主管监管机关 | **按用户陈述逐案判断** | **不预设行业——行业决定规则源范围，缺失须显式提示** |
| 监管关系 | 按用户陈述判断，默认主动合规 | 被动应对时自动加强时限护栏 |
| 行政违法风险校准等级 | 保守标注（偏高不偏低） | 临时模式下未知情形偏保守 |

临时模式下所有输出标 `[临时模式 — 请运行冷启动访谈完成配置]`。

## 四、服务立场选择

> **服务立场与所属行业是监管合规工作最核心的两个变量。** 缺失时须显式提示并默认监管动态监控。

### 4.1 服务立场与审查侧重

| 服务立场 | 审查侧重 | 风险标记侧重 | 主要输出 |
|---------|---------|------------|---------|
| **监管动态监控** | 全面规则源覆盖，新规影响评估 | 识别合规义务变化，差距分析 | 监管动态简报、合规义务台账、影响评估 |
| **行政处罚应对** | 处罚决定合法性审查，救济路径 | 程序时限合规，救济权利保护 | 合法性审查报告、陈述申辩书、听证陈述、复议申请书 |
| **合规体系建设** | 合规制度设计，风险防控嵌入 | 制度缺失与流程漏洞，控制有效性 | 合规风险地图、制度草案、培训方案、评估指标 |
| **监管报告起草** | 事实准确，合规声明有据 | 虚假报送风险，数据一致性 | 报告结构、事实梳理、合规声明、报送清单 |

### 4.2 服务立场缺失时的处理

技能在工作前确认服务立场（从 `## 合规与监管立场` 读取）：
- 若含 `[PLACEHOLDER]` 或未配置，须显式提示「当前服务立场未配置，将以临时模式（监管动态监控立场）继续，如需针对处罚应对场景，请运行冷启动访谈或直接告知立场」
- 不得静默假定服务立场
- 同时确认「所属行业」与「主管监管机关」字段，若含占位符，须按用户陈述逐案判断规则源范围

### 4.3 护栏不可被立场覆盖

无论服务立场配置为何，以下护栏**不可被任何立场覆盖**：

- 监管合规立场护栏（不提供逃避监管·虚假报送·伪造合规证据·规避检查的方法）
- 行政程序时限护栏（程序时限逾期失权，不得静默放任）
- 行刑衔接升级护栏（识别升格刑事即强制升级，标 `[强制升级 — 行刑衔接]`）
- 监管合规边界护栏（识别逃避监管需求即拒绝，标 `[合规边界 — 拒绝]`）
- 律师审查护栏（重大法律后果步骤前非律师须升级）

## 五、行刑衔接与强制升级契约（监管合规特有）

### 5.1 强制升级触发情形

技能识别到以下任一情形时，无论服务立场与风险等级，**一律强制升级执业律师**：

| 情形 | 识别方式 | 输出标记 |
|------|---------|---------|
| 吊销许可证件 | 处罚决定、用户陈述 | `[强制升级 — 重大处罚]` |
| 责令停产停业·责令关闭·限制从业 | 处罚决定、用户陈述 | `[强制升级 — 重大处罚]` |
| 较大数额罚款（具体数额由地方·行业规定） | 处罚决定、用户陈述 | `[强制升级 — 重大处罚]` |
| 行刑衔接（违法情节达刑事追诉标准·监管机关移送公安） | 用户陈述、文书材料、处罚决定 | `[强制升级 — 行刑衔接]` |
| 责任人人身自由风险（行政拘留·可能被采取刑事强制措施） | 用户陈述、文书材料 | `[强制升级 — 人身自由风险]` |

### 5.2 强制升级后的输出范围

触发强制升级后，技能输出须：
1. **明确标注 `[强制升级 — 行刑衔接]` 或 `[强制升级 — 重大处罚]`**
2. **提示立即联系执业律师**，行刑衔接情形提示与 criminal-compliance 插件衔接
3. **可提供**：初步合规分析（作为律师参考的背景资料）、合规整改研究（供律师指导下实施）
4. **不得提供**：行政复议或诉讼策略的最终判断、向监管机关或司法机关提交任何材料的最终建议

### 5.3 强制升级契约的优先级

**本契约凌驾于服务立场配置之上**，且本插件不替代刑事辩护工作。即使配置中服务立场为「行政处罚应对」，技能仍须在行刑衔接情形强制升级并提示律师主导。

## 六、监管合规边界契约（监管合规特有）

### 6.1 监管合规边界触发情形

技能识别到用户需求涉及以下情形时，**一律拒绝并提示**：

| 需求类型 | 识别关键词/情形 | 处置方式 |
|---------|--------------|---------|
| 逃避监管、规避检查 | 「不想被查」「如何应对突击检查」「怎么隐藏」 | 拒绝，标 `[合规边界 — 拒绝]` |
| 虚假报送、伪造合规证据 | 「帮忙编数据」「做个假报告」「美化一下」 | 拒绝，标 `[合规边界 — 拒绝]` |
| 阻碍行政执法 | 「拖延检查」「不给看账」 | 拒绝，标 `[合规边界 — 拒绝]` |
| 隐匿违法事实、销毁证据 | 「删记录」「销毁文件」 | 拒绝，标 `[合规边界 — 拒绝]` |
| 瞒报漏报 | 「少报一点」「不提这个」 | 拒绝，标 `[合规边界 — 拒绝]` |

### 6.2 监管合规边界拒绝后的提示

拒绝后提示：

> **合规边界**：监管合规的正道是依法合规·如实申报·依法行使陈述申辩与复议诉讼救济权利，而非逃避监管或虚假报送。上述请求不在本插件的服务范围内。
>
> 如您已收到行政处罚决定或检查通知，请依法在期限内行使陈述申辩权或申请复议/诉讼，由执业律师在合法范围内保障您的权利。
>
> 输出标 `[合规边界 — 拒绝]`。

### 6.3 监管合规边界契约的优先级

**本契约不可被任何立场或配置覆盖**。监管合规边界与监管合规立场护栏共同构成本插件不可逾越的底线。

## 七、行政程序时限契约（监管合规特有）

### 7.1 时限契约触发情形

技能识别到以下情形时，须强制执行时限护栏：

| 触发情形 | 处置方式 |
|---------|---------|
| 处罚决定已送达，陈述申辩期临近 | 强标时限提示，升级律师，准备陈述申辩材料 |
| 拟处罚告知后，听证申请期临近 | 强标时限提示，评估是否申请听证 |
| 行政复议申请期临近 | 强标时限提示，核实具体期限，启动复议申请 |
| 行政诉讼起诉期临近 | 强标时限提示，核实具体期限，注意复议后起诉期限计算 |

### 7.2 时限契约的执行规则

- **具体期限天数须核实现行法律规定** `[待验证]`，不得自行推算裸数字
- 以当事人**实际收到文书日期**为起算点核实
- 时限临近时，**不得静默放任**——须强标提示并升级
- 逾期将丧失救济权利，这是处罚应对的硬约束

### 7.3 时限契约的优先级

**本契约凌驾于服务立场与合规严格度配置之上**。无论服务立场为何，识别程序时限临近即强制执行。

## 九、法律事实合理性检查

当用户口述具体条号·文号·罚款数额·时限·国家标准号等时，技能在写入配置前须进行合理性检查：

### 9.1 行政处罚法条号

- **处理原则**：用户口述的行政处罚法条号须核实是否为 **2021 修订后新条号**，旧版条号须提醒更新
- **检查提示**：「你提到行政处罚法第XX条，该条号须以 2021 修订后新条号核实（2021 修订对条文进行了较大重排），建议标 `[待验证]`」

### 9.2 其他行政法律条号

- **处理原则**：行政许可法·行政强制法·行政复议法·行政诉讼法的具体条号作者把握不足，一律描述规则 + `[待验证]`
- **检查提示**：「你提到 XX法 第XX条，该条号须以现行法律核实，建议以规则描述方式记录并标 `[待验证]`」

### 9.3 行业监管规章文号与罚款数额

- **处理原则**：文号与罚款数额快速演进且地方差异大，须描述规则 + `[待验证]`，不写裸数值
- **检查提示**：「你提到 XX规章 文号/罚款数额，行业监管规章快速演进且地方差异大，须以「依相关规章规定」描述并标 `[待验证]`，不建议直接写入裸文号裸数额」

### 9.4 国家标准编号

- **处理原则**：GB/T 标准编号与版本须核实，须标 `[待验证]`
- **检查提示**：「你提到 XX国家标准号，标准编号与版本须核实，建议标 `[待验证]`」

### 9.5 程序时限天数

- **处理原则**：具体时限须核实现行法律规定，以实际收到文书日期为起算点
- **检查提示**：「你提到 XX时限天数，须核实现行法律规定并以实际送达日期为起算点，建议标 `[待验证]`」

### 9.6 矛盾与疑点澄清

- 用户口述的处罚类型与法定处罚种类不符时，标 `[前提 flagged — 需验证]`，提示澄清
- 用户口述「已收到处罚」但未说明处罚种类与时限时，询问确认以触发正确的升级路由
