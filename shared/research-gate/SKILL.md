# 中国法研究闸门

**本 skill 防止 Unsupported 的中国大陆法律结论。在生成任何法律文书前运行，除非用户明确豁免联网检索。**

## 适用场景

当以下任一条件满足时，必须先运行本研究闸门：

- 用户要求生成诉讼方案、起诉状、答辩状、代理意见、证据目录
- 用户要求审查、起草或修订合同
- 用户要求出具法律意见书、备忘录、合规意见
- 用户要求发送律师函、催告函、声明
- 涉及中国法律结论但用户未提供充分法律依据
- 案件涉及保函、担保、融资担保公司、特殊主体资格

## 强制工作流

### 第一步：清点用户材料

**先读用户提供的文件。** 如果用户提供目录或说 `本目录/全部材料/所有文件`，必须**递归清点整个目录**，包括根目录和子目录。

列出每个 PDF、DOCX、图片、表格和文本文件的状态，标记为 `已读取/无法读取/待读取`。

从材料中提取：
- 当事人及诉讼地位
- 关键日期、金额、文件
- 争议焦点
- 用户要求的输出
- 缺失事实

### 第二步：构建中国法搜索词

为每个法律问题构建三类搜索词：

1. **正向依据** — 支持预期路线的法律、司法解释、案例
2. **反向依据** — 可能推翻或限制该路线的不利案例、少数观点、程序陷阱
3. **程序陷阱** — 案由、主体资格、管辖、时效、举证责任、效力

### 第三步：按序检索

按以下优先级检索：

1. 用户提供的本地材料
2. 国家法律法规数据库、最高人民法院官网、司法解释、部门规章
3. 人民法院案例库、指导性案例、公报案例、最高法典型案例
4. 中国裁判文书网、地方法院公开判决
5. 监管机构材料
6. 法律公众号、律所文章、实务评论

> ⚠️ **两段式检索协议**：由于 `flk.npc.gov.cn` 是动态 SPA，headless 环境不可直接抓取，必须使用 **WebSearch 定位 → WebFetch 全文** 的两段式。详见 `references/authoritative-sources.md`。

### 第四步：动态网页处理

对于微信公众号、裁判文书网、人民法院案例库等需要 JS 渲染或登录态的页面，使用 `references/authoritative-sources.md` 中的 fallback 链。

### 第五步：输出研究底稿

在进入文书起草前，必须生成以下研究底稿：

| 输出 | 说明 |
|------|------|
| `法律依据表` | 法条、司法解释、案例 + 精确条文号 + 来源链接 |
| `类案/反例表` | 正向案例 + 不利案例 + 少数观点 |
| `争议焦点` | 案件核心争议及各方可能主张 |
| `风险提示` | 程序陷阱、主体资格、管辖、时效风险 |
| `来源清单` | 每个结论的来源层级（L1-L5） |
| `无法获取的来源及原因` | 标注需要律师人工补检索的内容 |

## 引用规则

- 法规和案例必须引用**精确名称、条文号、案号、法院、日期**
- 标注来源层级：L1 法律/司法解释、L2 入库案例/指导案例、L3 普通判决、L4 监管材料、L5 实务文章
- 如果权威来源冲突，**必须陈述冲突并给出更稳妥路线**，不得隐藏不利案例
- 如果关键结论无可靠权威来源，标注 `待律师复核`，**不得呈现为已定论**
- **禁止**仅引"百度百科"、"知乎"、个人博客作为法律依据

## 来源分级

| 级别 | 类型 | 约束 |
|------|------|------|
| L1 | 法律/司法解释全文 | 必须引用，诉讼方案中不得仅引 L5 转述 |
| L2 | 入库案例/指导案例/公报案例 | 可作为裁判思路依据 |
| L3 | 普通判决 | 可作为地方实务参考 |
| L4 | 监管文件 | 主体资格、行业规则依据 |
| L5 | 律所文章/学术/公众号 | **仅作发现案号、检索词、争议点的线索**，不得作绑定性依据 |

## 输出规则

研究底稿完成后，进入专项文书 skill 生成具体文书。详见：

- `references/source-policy.md` — 引用策略和失败回退链
- `references/quality-gates.md` — 研究质量闸门
- `references/search-playbooks.md` — 专项搜索策略（保函、管辖、主体资格等）
- `references/authoritative-sources.md` — 实测可访问的权威来源 URL 及抓取方式
