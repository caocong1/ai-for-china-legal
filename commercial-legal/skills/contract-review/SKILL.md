## 目的地检查

在生成输出之前，检查输出目的地。如果用户指定了目的地（群、分发列表、对方当事人等），询问是否在律师-当事人特权圈内。公司全员群、对方当事人、供应商等会使特权保护失效。当目的地看起来在特权圈之外时，标记并提供选项：(a) 仅法务的 privileged 版本，(b) 面向更广渠道的脱敏版本，或 (c) 两者。

## 目的

阅读合同，对照团队实际使用的审查立场（存储在实践配置文件中），找出每一个偏离审查立场的条款，告诉律师每个问题该怎么处理——给出具体的修改语言，而不是模糊的"建议修改"。

输出是一份审查备忘录，律师可以一次性完成所有操作。每个问题都有风险等级、商业影响解释、建议修改方案和升级通知（如果需要）。

## 主体资格预检

**在审查合同前，如果合同涉及以下任一情形，必须先核查主体资格：**

| 情形 | 核查内容 |
|------|----------|
| 担保条款/保证条款 | 担保人主体资格（上市公司决议、国资审批、融资担保公司权限等） |
| 上市公司作为一方 | 对外担保是否经董事会/股东会决议 |
| 国企/事业单位作为一方 | 是否有国资委审批或内部决策程序 |
| 银行分支机构作为担保方 | 是否有总行授权文件 |
| 保险公司（保证保险） | 区分保险责任和保证责任 |

如发现主体资格问题，在备忘录顶部显著标记，并评估对合同效力的影响。

## 前置条件：加载审查立场

**在阅读合同之前，读取实践配置文件。** 如果配置文件缺失或仍有占位符，显示以下提示：

> 我注意到你还没有配置你的实践档案——这是让我根据你的执业立场、升级规则和文书风格来定制输出的关键。
>
> **两个选择：**
> - 运行 `/commercial-legal:cold-start-interview`（约 2 分钟）来配置你的档案，然后我会根据你的真实审查立场进行审查。
> - 说 **"临时模式"**，我会使用通用默认值——中国法律体系、中等风险偏好、律师角色、无特定审查立场（从第一性原理标记常见合同风险）——并在每个输出上标记 `[临时模式 — 配置你的档案以获得定制输出]`。

### 临时模式

如果用户说"临时模式"，正常运行审查，使用以下通用默认值：中等风险偏好，律师角色，中国法律体系，无特定审查立场（标记常见的甲方/乙方风险——无限责任、无数据泄露例外、不明确的违约金、自动续约无通知等）。在审阅者注释和每个发现块上标记 `[临时模式]`。在输出末尾附加：

> "这是一次通用默认值运行。运行 `/commercial-legal:cold-start-interview` 来获得根据你的真实审查立场、管辖法律、风险偏好定制的输出。约 2 分钟。"

**哪一方？** 在应用审查立场之前，确定公司在该合同中的立场。通常很明显：如果你是买方/接受服务方，你是"甲方"立场；如果你是卖方/提供服务方，你是"乙方"立场。如果不明显（经销协议、合作协议、分成协议），询问："在这个协议中，[公司] 是哪一方——甲方（买方/委托方）还是乙方（卖方/受托方）？" 读取配置文件中匹配的审查立场部分。在输出中标注应用了哪一方的立场，以便审阅者知道。

实践配置文件是真相来源。它告诉你：
- 团队的标准审查立场是什么（不是市场惯例——*他们的*立场）
- 以前接受过的备选方案
- 绝不接受的条款
- 谁审批什么
- 首先要检查的"交易破坏者"

如果合同存在"交易破坏者"，在备忘录顶部标记并停止详细审查。如果对方坚持无限责任或数据训练权，花 30 分钟审查违约金上限没有意义。

## 工作流

### 第一步：定位

快速阅读整份合同，回答：

| 问题 | 回答 |
|---|---|
| 这是什么类型的合同？ | 买卖合同 / 技术服务合同 / 租赁合同 / 合作协议 / 其他 |
| 我方是哪一方？ | 甲方 / 乙方（标记如果不是典型情况） |
| 对方当事人 | 名称，是大公司（不太会改条款）还是初创/中小企业（愿意协商）？ |
| 合同金额 | 总价/年费，如果有的话 |
| 合同期限 | 期限长度、续约机制 |
| 有无数据处理？ | 涉及个人信息/重要数据 / 不涉及 |
| 有无担保？ | 保证/抵押/质押/无 |

**合同金额处理。** 如果合同本身没有写明金额（框架协议由订单/结算单定价，这很常见），**停下来询问**后再应用升级数学或金额阈值：

> 本协议本身没有写明合同金额。价格由订单/结算单确定。你的升级阈值是 ¥[来自升级矩阵的金额]。在我路由之前，我需要知道合同预估金额。选项：
> 1. 提供订单/结算单的金额（首选——我会用于路由和备忘录）。
> 2. 告诉我这是高于还是低于 ¥[阈值]，我会相应路由；备忘录会标记路由假设了 [高于/低于阈值] 而没有具体金额。
> 3. 保守地路由到更高级别的审批人——对于你还没有定价的审查，这样更安全。

不要默默假设一个金额然后用于路由。这个假设会传播到审批调用，而审查不应该在这里猜测。

### 第二步：交易破坏者检查

首先检查实践配置文件中的"交易破坏者"（绝不可接受的条款）。如果存在：

```markdown
## ⛔ 存在交易破坏者

**第 [X.X] 条** 包含 [交易破坏者]。根据团队审查立场，这是硬性的"不"。建议：

- [ ] 坚持修改——建议 [具体替代表述]
- [ ] 放弃——如果对方不让步，我们不签

详细审查如下供参考，但此问题不解决则其他审查无意义。
```

### 第三步：逐项对比

对于实践配置文件中的每个审查类别，找到对应的合同条款并进行对比。

**对于每个偏离，生成：**

```markdown
### [第 X.X 条]：[问题名称]

**我们的立场：** [团队标准审查立场，引用自实践配置文件]

**合同写的是：**
> "[合同中原文引用]"

**偏离类型：** [缺失条款 | 弱于标准 | 弱于备选 | 非典型结构 | 不可接受]

**法律风险：** 🔴 关键 | 🟠 高 | 🟡 中等 | 🟢 低
**商业摩擦：** 🔴 阻碍交易 | 🟠 拖慢交易 | 🟡 造成困惑 | 🟢 无感

**为什么重要：** [一到两句白话解释——如果条款保持原样，业务会出什么问题]

**建议修改：**
> "[具体的替换语言——可以直接粘贴到修订版中]"

**如果对方不让步：** [实践配置文件中的备选方案，或"升级到 [人员]"
如果没有备选方案]
```

**风险等级校准：**

| 级别 | 含义 |
|---|---|
| 🔴 关键 | 不修复不签署。团队"绝不接受"清单上的条款，或交易破坏者。 |
| 🟠 高 | 强烈推动；不让步则升级。超出审查立场备选范围的条款。 |
| 🟡 中等 | 第一轮推动；如果是最后一个开放问题可以接受。在备选范围内但未达到标准立场。 |
| 🟢 低 | 记录，不花谈判资本。审查立场明确容忍的条款，或纯形式偏离。 |

风险等级始终对照实践配置文件。如果条款不能清楚映射到审查立场类别，询问用户它属于哪个类别，并提供将答案记录到实践配置文件中的选项。

#### 违约金/责任限制决策程序

**金额本身不是最重要的部分。** 在审查责任限制条款时，不要只生成一行"检查责任限制"。要完成以下四个维度并在发现中明确陈述每个维度：

1. **直接损失 vs 间接损失。** 限制是否适用于所有责任，还是仅直接损失？对直接损失的 12 个月限制与对间接损失无限制，和对所有损失的 12 个月总限制是完全不同的立场。明确陈述两种处理。

2. **基数定义——逐字引用。** "12 个月限制"可能意味着：(a) 索赔前 12 个月已支付的费用，(b) 当前 12 个月期间应付的费用，(c) 最后 12 个月使用期间的费用，(d) 当前订单项下的费用，(e) 已支付的总费用。这些可能相差一个数量级。引用确切语言。如果不明确，标记："基数定义不明确——`[引用的语言]`——可能意味着 [X] 或 [Y]。签署前确认。"

3. **例外交互。** ¥100K 的限制但对数据泄露、知识产权和保密有不设限的赔偿，在 SaaS 争议中实际上是不设限的。列举哪些在限制之上（例外），哪些在限制之下（实际被限制的），评估被限制的表面是否有意义："限制覆盖 [一般合同违约]。数据泄露、知识产权赔偿和保密例外且不受限。对于这个供应商的风险状况，被限制的表面是 [有意义的 / 名义上的]。"

4. **你每个维度的审查立场。** 实践配置文件应该有每个维度的立场：直接限制（费用的倍数）、间接损失（排除/限制/不设限）、例外清单（什么可以接受在限制之上）、基数定义（你接受哪个定义）。

#### 管辖法律差异检查

**审查立场适用一个管辖法律偏好。可执行性因法律而异。** 在按字面接受审查立场之前，对照主要差异检查合同实际适用的管辖法律：

- **竞业限制/竞业禁止：** 《劳动合同法》第 23-24 条有特定限制（期限不超过 2 年、需支付经济补偿）。各地司法实践有差异。`[管辖法律 — 需验证]`
- **自动续约：** 民法典合同编对格式条款有特别规定。消费者合同有额外保护。`[管辖法律 — 需验证]`
- **责任排除：** 民法典第 506 条规定造成对方人身损害、因故意或重大过失造成对方财产损失的免责条款无效。`[管辖法律 — 需验证]`
- **违约金调整：** 民法典第 585 条规定违约金过高或过低可请求法院调整。`[管辖法律 — 需验证]`
- **格式条款：** 民法典第 496-498 条对格式条款有特别规制。`[管辖法律 — 需验证]`

当审查立场与合同管辖法律的可执行性冲突时，标记："你的审查立场倾向于 [X]，但该合同受 [Y] 法律管辖，在那里 [X] 是 [不可执行 / 受限制 / 受法定override]。`[管辖法律 — 需验证]`"

### 第四步：有利条款和缺失

两个短列表：

**优于我方标准：** 对方给了我们比要求的更多的条款。注意这些——如果你需要在其他地方让步，它们是交易筹码。

**完全缺失：** 根本没有的标准条款。最常见的：转让限制、审计权（如果我们想要的话）、不可抗力、保险要求。

### 第五步：升级路由

检查实践配置文件中的升级矩阵，对照：
- 合同金额
- 是否存在任何 🔴 关键问题
- 任何自动升级触发条件（无限责任、知识产权转让等）

清楚说明谁需要审批：

```markdown
## 审批路由

基于 [金额 / 风险等级]，本协议需要：

- [ ] **[姓名/角色]** 审批——[原因]
- [ ] **业务负责人确认** [具体商业条款他们应该参与]

**建议下一步：** [向对方发送修订 | 在回复前升级到合伙人 | 在法务回复前获取业务方对商业条款 X 的意见]
```

## 修改粒度

**以尽可能小的粒度修改。** 修订是谈判工具，不是重写。 wholesale 条款替换发出"我们扔掉了你的草稿"的信号——这是侵略性的，它迫使对方重新阅读整个条款，并丢弃了他们没问题的部分。精细的修订——删除一个词、插入一个短语、重组一个子条款——发出"我们有具体要求"的信号，更容易阅读、理解和接受。

默认以最小的修改实现审查立场：
- 替换一个**词**优先于短语。
- 替换一个**短语**优先于句子。
- 重组一个**子条款**优先于替换句子。
- 替换一个**句子**优先于替换条款。
- 仅在对方的版本与你的立场相差太远以至于精细修改比新草稿更难阅读时才替换**整个条款**——当你这样做时，在传输中说明："我们替换了第 8.2 条而不是标记，因为修改范围很大。很高兴带你过一遍差异。"

拿不准时，选小的。收到精细修订的客户相信你是仔细读过的。收到 wholesale 替换的客户怀疑你是否读过。

### 第六步：组装备忘录

 prepend 实践配置文件中 `## 输出` 部分的工作产物标题（它根据用户角色而不同——见 `## 谁在使用`）。

本备忘录和基础协议可能是 privileged、confidential 或两者兼有。输出从源继承该状态。仅在律师-当事人特权圈内分发；在 privileged 材料存放处标记和存储；在任何外部交付前剥离工作产物标题。

应用的审查立场反映了实践配置文件中记录的管辖法律。如果此交易涉及不同的管辖法律或法律选择问题，在备忘录中标记——分析可能不会按原样转移。

> **无默默补充。** 如果对配置的法规检索工具的查询返回的规则或备忘录需要的结果很少（违约金的可执行性、赔偿范围、管辖法律选择），报告找到的内容并停止。不要用网络搜索或模型知识填补空白而不询问。说："搜索从 [工具] 返回了 [N] 个结果。对 [规则/管辖法律] 的覆盖似乎较薄。选项：(1) 扩大搜索范围，(2) 尝试不同的检索工具，(3) 搜索网络——结果会被标记 `[网络搜索 — 需验证]` 并在依赖前应对照原始来源检查，或 (4) 标记为未验证并停止。你想要哪个？" 律师决定是否接受较低置信度的来源。
>
> **来源归属。** 备忘录引用法规、规章或案例时，标记引用：`[法宝]`、`[法规/监管机构网站]` 或从法规检索连接器检索的 MCP 工具名称；`[网络搜索 — 需验证]` 用于网络搜索引用；`[模型知识 — 需验证]` 用于从训练数据回忆的引用；`[用户提供]` 用于来自对方草稿或内部文件的引用。标记为 `verify` 的引用具有更高的捏造风险，应首先检查。绝不剥离或折叠标签。
