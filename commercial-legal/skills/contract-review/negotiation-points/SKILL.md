# 谈判要点子技能

> **迁移说明**：本子技能从原 contract-review/SKILL.md 的「有利条款和缺失」「升级路由」中提炼谈判相关内容，并从审查立场推导谈判策略。
> 可被编排入口调用，也可单独触发：`/commercial-legal:negotiation-points [合同]`

---

## 目的

从实践配置文件中的审查立场出发，将风险识别结果转化为可操作的谈判路线图：

1. **硬底线**（交易破坏者）— 绝不让步的核心条款
2. **让步空间**（可接受备选）— 可在谈判中灵活应对的范围
3. **交易筹码**（有利条款）— 对方给了比标准更多的条款，可用于在其他地方换取让步
4. **谈判节奏建议**（对方类型适应）— 大公司 vs 中小企业的差异策略

---

## 前置：加载资源

在执行谈判策略前，读取以下资源：

1. **实践配置文件**：`commercial-legal/CLAUDE.md`（审查立场、交易破坏者、可接受备选、升级矩阵）
2. **风险识别结果**：来自 risk-identification 子技能（若单独触发，则先执行）
3. **修改建议**：来自 clause-revision 子技能（若已执行，可提取「对方不让步」部分）

**事项上下文**：检查实践配置文件 `## 事项工作区` 是否启用。

## 详细步骤

### 第一步：从审查立场推导谈判三层次

**层次 1 — 硬底线（交易破坏者）**

从实践配置文件「交易破坏者（The one thing）」条款提取：
- 这些是绝不接受的条款
- 谈判策略：坚持，不让步，若对方坚持则考虑放弃交易
- 谈判措辞建议：「这是我们法务合规方面的硬性要求，不在谈判范围内」

**层次 2 — 让步空间（可接受备选）**

从实践配置文件「可接受的备选」提取：
- 这是审查立场的底线范围，可在谈判中让步至此
- 谈判策略：先推标准立场，对方反对则退至备选
- 谈判措辞建议：「我们的首选是 [X]，如果这对你有困难，我们可以接受 [备选]」

**层次 3 — 交易筹码（有利条款）**

从风险识别中「优于我方标准」的条款提取：
- 对方已经给了比我方要求更多的条款
- 谈判策略：不主动提出，留作在其他条款上换取让步
- 记录：「第X.X条对方给了我们比标准更多的条款，这是谈判资本」

### 第二步：对方类型差异化策略

**大公司（标准合同，改动空间小）：**

| 策略维度 | 应对思路 |
|---------|---------|
| 整体姿态 | 接受大框架，精准推动核心风险条款修改 |
| 修改范围 | 聚焦 🔴🟠 风险条款，🟡🟢 非关键时可接受 |
| 传递方式 | 以「法律合规要求」包装硬底线（更难拒绝） |
| 升级信号 | 大公司联系人若无权改，请求升级决策层 |
| 预期管理 | 知会业务方：某些条款可能无法改，需提前了解风险 |

**中小企业（灵活，愿意协商）：**

| 策略维度 | 应对思路 |
|---------|---------|
| 整体姿态 | 全面推动，对方通常比大公司更灵活 |
| 修改范围 | 所有偏离条款均可推动修改 |
| 传递方式 | 解释商业逻辑（共赢），对方更容易理解接受 |
| 风险提示 | 注意中小企业可能存在主体资格、偿付能力风险 |
| 预期管理 | 可能因谈判时间较长影响交易节奏 |

### 第三步：谈判顺序规划

**建议的谈判顺序（优先级）：**

1. 先解决交易破坏者（若存在），否则继续审查意义不大
2. 解决 🔴 关键风险条款（法定无效或绝不接受清单）
3. 推动 🟠 高风险条款（超出备选范围）
4. 在其他条款上让步，换取 🟠 条款的让步
5. 🟡 中等风险在「最后一轮」时可接受
6. 🟢 低风险不花谈判资本

### 第四步：升级路由建议

检查实践配置文件中的升级矩阵，对照：
- 合同金额与审批权限阈值
- 是否存在任何 🔴 关键问题
- 自动升级触发条件（无限责任、知识产权转让、境外管辖等）

```markdown
## 审批路由建议

基于 [金额/风险等级]，本协议需要：

- [ ] **[姓名/角色]** 审批 — [原因]
- [ ] **业务负责人确认** — [具体商业条款需确认]

**建议下一步：** [发送修订给对方 | 升级到合伙人后再回复 | 先获取业务方对X条款的意见]
```

### 第五步：谈判路线图汇总

```markdown
## 谈判路线图

### 硬底线（不让步）

| 条款 | 底线要求 | 谈判措辞建议 |
|------|---------|------------|
| [第X.X条] | [我方坚持的最低要求] | [建议措辞] |

### 让步区间（可谈）

| 条款 | 标准立场 | 可接受底线 | 换取条件 |
|------|---------|---------|---------|
| [第X.X条] | [标准] | [备选] | [需要对方在X换取] |

### 交易筹码（对方已让步，留用）

| 条款 | 对方给的 | 用途 |
|------|---------|------|
| [第X.X条] | [有利条款] | [可换取X条款让步] |

### 建议升级

[升级路由建议]
```

## 检查清单

- [ ] 硬底线从实践配置文件「交易破坏者」提取（有据可查）
- [ ] 让步区间从「可接受备选」提取（不是自行判断）
- [ ] 有利条款（优于标准）已从风险识别结果中提取并标注
- [ ] 已识别对方类型（大公司/中小企业）并调整策略
- [ ] 谈判顺序按优先级排列（先破坏者，再关键，再高风险）
- [ ] 升级路由检查完成（合同金额 + 风险等级 + 自动触发条件）
- [ ] 谈判路线图表格列数一致（四列/三列等）

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 配置文件无交易破坏者定义 | 基于 🔴 风险条款推导硬底线，标注「临时底线，建议更新配置文件」 |
| 配置文件无备选方案 | 建议律师明确备选，标注「本次谈判需律师判断让步下限」 |
| 对方类型不明确 | 询问：「对方是大公司（标准合同不灵活）还是中小企业（可以谈）？这影响谈判策略」 |
| 合同金额超过所有审批阈值 | 在升级路由中说明需最高级审批 |
| 所有条款都是交易破坏者 | 标注「本合同大部分条款不符合立场，建议考虑是否要推进这笔交易」 |
| 谈判已进入最后阶段 | 标注「最后阶段谈判建议：保留 🔴🟠，放弃 🟡🟢，集中资本在核心条款」 |

## 错误处理

| 错误情形 | 处理方式 |
|---------|---------|
| 未先运行风险识别 | 「需要风险识别结果才能推导谈判策略。运行 risk-identification 子技能，或提供风险清单」 |
| 配置文件仍有占位符 | 提示：「配置文件中审查立场未填写，无法准确推导谈判底线。建议运行冷启动访谈或手动填写」 |
| 业务方意见冲突（业务要签，法务不同意） | 标注升级信号，提供「业务方确认风险接受」的选项（明确写明风险后签字确认） |
