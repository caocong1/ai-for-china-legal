# 输出格式化子技能

> **迁移说明**：本子技能包含原 contract-review/SKILL.md 中的「组装备忘录」「输出格式」（完整备忘录/微信群摘要/修订文档）「交付前质量检查」「下一步决策树」「目的地检查」「静默模式规则」。
> 可被编排入口调用，也可单独触发：`/commercial-legal:output-formatting [审查结果]`

---

## 目的

将其他四个子技能的结构化输出组装为可交付的审查文件：

1. **完整备忘录**（默认）— 律师内部使用的完整审查报告
2. **微信群摘要**（可选）— 两行摘要，用于微信群快速同步
3. **修订文档**（可选）— 带修订标记的 .docx，用于传递给对方

---

## 前置：加载资源

在执行输出格式化前，读取以下资源：

1. **实践配置文件**：`commercial-legal/CLAUDE.md`（`## 输出`：工作产物标题、审查者注释块、静默模式规则、下一步决策树模板）
2. **四个子技能输出**：risk-identification / clause-revision / negotiation-points / legal-basis 的结果
3. **事项上下文**：检查实践配置文件 `## 事项工作区` 是否启用；若启用，确认输出写入路径

---

## 详细步骤

### 第一步：目的地检查（优先）

**在生成任何输出之前，检查输出目的地（来自原 SKILL.md 「目的地检查」）：**

若用户指定了目的地，询问是否在律师-当事人特权圈内：

| 目的地类型 | 特权风险 | 处理方式 |
|---------|---------|---------|
| 内部法务文件夹 | 低风险 | 直接输出完整备忘录 |
| 律师-当事人工作群（仅法务） | 低风险 | 输出完整备忘录 |
| 公司全员群 | 特权失效风险 | 提供脱敏版本 |
| 对方当事人 | 特权失效风险 | 提供净化版本（剥离内部评估） |
| 业务部门群 | 中等风险 | 询问是否在特权圈内 |

当目的地看起来在特权圈之外：提供两个选项：
(a) 仅法务的 privileged 版本；(b) 面向更广渠道的脱敏版本；(c) 两者

### 第二步：读取工作产物标题

从实践配置文件 `## 输出` 的「工作产物标题」读取，根据角色不同：

- **律师角色**：`律师工作产物 — 受律师-当事人特权保护 / 保密`
- **非律师角色**：`研究笔记 — 非法律建议，供律师审查`

若配置文件为临时模式，标题附加：`[临时模式 — 配置你的档案以获得定制输出]`

### 第三步：组装完整备忘录

**备忘录结构（prepend 工作产物标题后）：**

```markdown
[工作产物标题 — 根据实践配置文件 ## 输出]

# 合同审查：[对方当事人] [合同类型]

**审查日期：** [日期]
**合同金额：** ¥[金额] / [期限]
**我方立场：** 甲方 / 乙方
**管辖法律：** [合同约定]
**应用审查立场：** [来自实践配置文件，或临时模式]

---

> **审查者注：** [来自实践配置文件 ## 输出 审查者注释块模板]
> 法规来源：[列出引用的法规]
> 阅读范围：[本合同]
> 标记项：[N]🔴 [N]🟠 [N]🟡 [N]🟢
> 时效性：[日期] — 法律变更可能影响分析

---

## 底线

[两句话。我们能签吗？什么必须先改？]

**问题（法律风险）：** [N]🔴 [N]🟠 [N]🟡 [N]🟢
**问题（商业摩擦）：** [N]🔴 [N]🟠 [N]🟡 [N]🟢

**需要审批：** [来自 negotiation-points 的升级路由]

---

## 交易破坏者检查

[✅ 未发现交易破坏者 | ⛔ 存在 — 第X.X条 [名称]]

---

## 按风险等级分类的问题

[来自 risk-identification 和 clause-revision 的合并结果，按 🔴 → 🟠 → 🟡 → 🟢 分组]

[每个问题块包含：风险等级、原文引用、偏离类型、修改建议（含替换语言）、对方不让步时的备选、法律依据]

---

## 有利条款

[来自 risk-identification 的「优于我方标准」列表]

## 缺失条款

[来自 risk-identification 的「完全缺失」列表]

---

## 法律依据

[来自 legal-basis 子技能的完整依据清单，按法条/案例/学说分类]

---

## 谈判要点

[来自 negotiation-points 的谈判路线图摘要]

---

## 审批路由

[来自 negotiation-points 的升级路由建议]

---

## 修订包

[若已完成 clause-revision：所有建议更改的整合，按条款顺序排列]
```

### 第四步：微信群摘要版

**触发条件**：用户请求微信群摘要，或目的地是工作群。

**格式（两行加链接）：**

```
[对方当事人] [类型] — 需要修改。[N]🔴（[最关键问题简述] 第X.X条），[N]🟠。完整审查：[链接/文件位置]。需要 [审批人姓名] 审批。
```

**若无风险**：
```
[对方当事人] [类型] — 可以签，有几个小的跟进事项。完整审查：[链接/文件位置]。
```

### 第五步：修订文档版

**触发条件**：用户要求输出修订标记文档（.docx）。

**格式要求：**
- 每个修改用修订标记（Track Changes）格式标出
- 每个修改的批注引用：审查立场 + 法条（如适用）
- 若无 docx 工具，输出可粘贴的 markdown 修订记录，说明「无法直接生成 .docx，以下为可手动应用的修订记录」

### 第六步：交付前质量检查

在输出前逐项确认（来自原 SKILL.md 「交付前的质量检查」）：

- [ ] 实践配置文件已加载并引用——不是通用市场立场
- [ ] 交易破坏者已优先检查并明确说明
- [ ] 每个问题都有具体替换语言（不是「建议修改」）
- [ ] 风险等级已校准（不全是关键，对照配置文件）
- [ ] 审批人有具体姓名，不是「升级到法务」
- [ ] 对方当事人背景已考虑（大公司 vs 中小企业影响谈判策略）
- [ ] 工作产物标题已 prepend（来自实践配置文件）
- [ ] 临时模式已在每个发现块和输出末尾标注（若适用）
- [ ] 目的地检查已完成，特权保护风险已处理
- [ ] 法律依据按来源分类并标注来源标签

### 第七步：下一步决策树

以下一步决策树结束（来自实践配置文件 `## 输出` 下一步决策树模板，根据本次审查结果定制）：

```markdown
---

## 下一步

基于本次审查结果，请选择：

1. **起草修订版** — 根据建议修改生成修订版合同（运行 `/commercial-legal:clause-revision`）
2. **升级** — 将关键问题升级给 [指定人员]（需要：[描述问题]）
3. **获取更多事实** — 需要补充信息：[列出缺少的信息]
4. **观望等待** — 标记风险但暂不行动，等待对方回应
5. **其他** — 律师自定义下一步

---
```

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 某子技能输出缺失 | 在对应章节标注「[本章节需先运行 X 子技能]」，不留空白章节 |
| 临时模式 | 在每个发现块标 `[临时模式]`，输出末尾附加冷启动引导 |
| 面向对方的交付物 | 应用静默模式：净化特权标题和内部风险评估，保留修订建议 |
| 面向董事会/业务方 | 简化法律术语，聚焦商业影响，使用 `## 输出` 中的静默模式规则 |
| 无法生成 .docx | 输出可手动应用的修订记录，说明「无法直接生成 .docx」 |
| 实践配置文件无下一步决策树模板 | 使用五个默认分支（起草/升级/获取事实/观望/其他），标注「请在配置文件中自定义决策树」 |

---

## 错误处理

| 错误情形 | 处理方式 |
|---------|---------|
| 四个子技能输出均缺失 | 「请先完成合同审查（risk-identification → legal-basis → clause-revision → negotiation-points），或运行完整流程 `/commercial-legal:contract-review`」 |
| 目的地明确是对方当事人 | 停下，提供两个版本选项，不直接输出内部备忘录 |
| 配置文件工作产物标题仍为占位符 | 使用通用律师标题，标注「[配置文件标题占位符，请运行冷启动访谈]」 |

## 法律依据

> 本子技能为输出格式化工具，不直接产出新的法律依据。法律依据由 legal-basis 子技能负责，并在备忘录中呈现。以下为输出格式化相关的规范依据。

### 法条（格式化合规参考）

|------|---------|---------|---------|

### 备注

> 输出格式化子技能的「法律依据」章节，直接引用 legal-basis 子技能的输出，不重新生成。确保来源标签和待验证标注从 legal-basis 原样保留，不剥离或折叠。
