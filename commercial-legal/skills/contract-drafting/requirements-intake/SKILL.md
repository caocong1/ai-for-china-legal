# 需求收集（requirements-intake）

> **前置**：读取 `commercial-legal/CLAUDE.md`（实践配置文件），确认配置已填充（无 `[PLACEHOLDER]`）。
> 若配置含占位符，按 `commercial-legal/skills/_shared/practice-profile-schema.md` §3.2 提示律师选择「配置模式」或「临时模式」。
> 本子技能可被 `contract-drafting/SKILL.md` 编排入口调用，也可通过 `/commercial-legal:requirements-intake` 单独触发。

---

## 目的

在起草工作正式开始前，完整、准确地收集所有关键信息，避免因信息缺失导致后续返工。
信息不全时主动追问，绝不静默假设——假设错误方向将导致整个草案偏离用户真实需求。

---

## 详细步骤

### 步骤 1：加载实践配置

读取 `commercial-legal/CLAUDE.md` 中以下字段：
- `## 我们是谁` → 公司/律所名称、实体类型、主要交易对手类型
- `## 谁在使用` → 执业角色（律师 / 非律师）
- `## 审查立场` → 对应立场的标准立场、可接受备选、绝不接受
- `## 升级矩阵` → 金额阈值与自动升级触发条件

### 步骤 2：核实合同基本类型

按以下合同类型导入对应的标准条款框架：

| 合同类型 | 适用标准框架 | 特别关注点 |
|---------|------------|---------|
| 买卖合同 | 标的/价格/交付/验收/质保/违约 | 所有权转移时点、质量检验期 |
| 技术服务合同 | 服务范围/里程碑/验收/IP/保密/违约 | 知识产权归属、工作成果定义 |
| 租赁合同 | 租金/押金/期限/维护/续租/违约 | 物业状态、转租限制 |
| 合作协议（框架） | 合作范围/分工/利益分配/退出 | 股权/利润分配、竞业 |
| 许可协议 | 许可范围/费用/独占性/质量控制/终止 | 次许可权、改进归属 |
| 其他 | 根据描述推断关键条款 | 询问用户确认 |

**动作**：确认用户所述合同类型后加载对应标准条款清单（供步骤 3 使用）。

### 步骤 3：系统性需求收集访谈

按以下问题矩阵逐一收集，未回答项目**不得跳过**，须追问：

| 问题类别 | 具体问题 | 优先级 | 不全时处理 |
|---------|---------|-------|---------|
| **合同类型** | 确认具体合同类型 | 🔴 必须 | 阻断，先确认类型 |
| **我方立场** | 甲方（买方/委托方）还是乙方（卖方/受托方）？ | 🔴 必须 | 询问：「[机构名称] 在这笔交易中是哪一方？」 |
| **对方当事人** | 对方名称、类型（大公司/中小企业/个人）、谈判地位 | 🔴 必须 | 提示：了解对方背景有助于设计谈判空间 |
| **合同金额** | 总价/年费/计价方式、付款安排 | 🔴 必须 | 标记升级阈值警告（对照 `## 升级矩阵`） |
| **合同期限** | 期限长度（首期/续约期）、续约机制 | 🔴 必须 | 若期限超过 2 年，提示情势变更风险 |
| **交付标准** | 核心交付物、验收标准、时间节点 | 🔴 必须 | 不能起草，追问直至明确 |
| **付款方式** | 预付/里程碑/月付/到付，付款条件 | 🔴 必须 | 影响风险分配 |
| **知识产权** | 是否涉及创作/开发？归属倾向？ | 🟠 重要 | 若有软件开发/设计，须明确 |
| **数据处理** | 是否涉及个人信息或重要数据？ | 🟠 重要 | 若是，加载数据合规条款模块 |
| **竞业限制** | 是否需要限制对方合作竞争对手？ | 🟡 视情况 | 行业特殊时追问 |
| **担保需求** | 是否需要保证、抵押或质押担保？ | 🟡 视情况 | 若金额大或对方信用未知，建议考虑 |
| **争议解决** | 诉讼偏好（哪个法院？）还是仲裁（哪个仲裁机构）？ | 🟠 重要 | 与实践配置文件 `## 审查立场` 对照 |
| **特殊要求** | 合规限制、母公司/关联方/集团要求、特殊行业规定 | 🟡 视情况 | 询问：「还有什么特殊要求？」 |

### 步骤 4：需求完整性验证

收集完毕后，运行以下检查：

- [ ] 合同类型已确认，标准框架已加载
- [ ] 我方立场已确认（甲方 or 乙方）
- [ ] 对方当事人信息充分
- [ ] 金额已知且已对照升级阈值
- [ ] 合同期限已知
- [ ] 交付/服务标准可量化
- [ ] 付款条件已明确
- [ ] 知识产权归属已明确（若涉及）
- [ ] 数据合规需求已评估
- [ ] 争议解决偏好已确认

**未勾选项**：须追问用户补充，不得在信息不全时进入下一步骤。

### 步骤 5：生成需求摘要

将收集结果结构化输出（见「输出模板」），供后续 `framework-design` 和 `clause-drafting` 使用。

---

## 检查清单

**信息完整性**：
- [ ] 🔴 必须项（合同类型、立场、对方、金额、期限、交付、付款）全部已收集
- [ ] 🟠 重要项（知识产权、数据处理、争议解决）已评估（即使决定不特别处理）
- [ ] 🟡 视情况项（竞业、担保、特殊要求）已询问，无遗漏

**立场对照**：
- [ ] 已确认我方立场（甲/乙方）
- [ ] 已加载 `commercial-legal/CLAUDE.md` 对应立场的「标准立场」「绝不接受」字段
- [ ] 合同金额已与升级矩阵阈值比对，须升级时已标记

**法律风险预警**：
- [ ] 若期限超 2 年，已提示情势变更条款需要
- [ ] 若涉及个人信息，已触发数据合规模块
- [ ] 若涉及担保，已记录后续需主体资格核查（见 `framework-design`）

## 边界条件

| 场景 | 处理方式 |
|------|---------|
| 用户提供的合同类型不明确（如「我需要一个合作协议」） | 询问：「这是框架合作协议（无直接交付物）还是特定项目合同（有具体交付物）？」 |
| 用户不确定我方立场 | 询问：「[机构名称] 在这笔交易中提供服务/商品，还是购买服务/商品？」 |
| 合同金额未定（如「根据实际工作量计算」） | 接受，但标记：「计价方式需明确计算公式和上限，以便责任限制条款设计」 |
| 对方是国有企业或上市公司 | 提示：「请确认对方主体授权（签约人是否有授权？）及合规审批流程」 |
| 涉及个人信息处理 | 必须触发：「涉及个人信息处理，需增加数据处理条款，对照《个人信息保护法》」 |
| 用户在信息不全时要求直接起草 | 明确说明：「以下信息缺失将导致草案有重大漏洞：[列出未收集项]。建议先补充，或确认使用假设值（须在草案中明确标注为假设）」 |
| 合同金额超过实践配置文件升级阈值 | 自动标记：「此合同金额超过升级阈值，需升级至 [指定人员]。继续起草前请确认」 |

## 错误处理

| 错误场景 | 处理方式 |
|---------|---------|
| 实践配置文件未填充（含 `[PLACEHOLDER]`） | 提示配置模式或临时模式选项，不静默假设 |
| 用户给出矛盾需求（如「要求完整责任保障」且「违约金不超过 1000 元」） | 明确指出矛盾：「你要求完整责任保障，但违约金上限 1000 元远低于潜在损失。请确认优先事项」 |
| 超出 commercial-legal 插件范围（如要求起草跨境合同适用境外法） | 说明：「跨境合同适用境外法律超出本插件范围，请咨询专门的跨境交易律师」 |
| 信息追问超过 3 轮对方未回答 | 汇总未解决项，建议：「以下信息暂时无法获取，我可以按通用模式起草并在草案中标记需填充的占位符」 |
