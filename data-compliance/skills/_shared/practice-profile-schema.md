# 实践配置文件契约（数据合规版 Practice Profile Schema）

> 本文件定义 data-compliance 插件的实践配置文件契约：字段定义、存储位置、技能读取规则，以及与 `data-compliance/CLAUDE.md` 实践配置模板的映射关系。
>
> **所有数据合规技能通过本契约读取个性化配置。** 冷启动访谈子技能（cold-start-interview）按本契约填充配置文件。

---

## 一、配置文件是什么

### 真相来源

实践配置文件的真相来源是 **`data-compliance/CLAUDE.md`**（散文形式）。该文件：

- 由 `/data-compliance:cold-start-interview` 冷启动访谈生成和更新
- 可由律师/DPO 直接编辑（无需重新运行访谈），编辑后即时生效
- 为所有数据合规技能提供个性化设置的读取来源
- 本阶段（05-01）**首次创建**，此前插件无此文件

### 架构决策（不引入冲突新格式）

**决策：不引入与现有体系冲突的新配置文件格式。**

`data-compliance/CLAUDE.md`（散文模板）是唯一真相来源。

**理由：**

1. `data-compliance/CLAUDE.md` 是律师/DPO 可直接阅读和编辑的人类友好格式，散文胜于 JSON/YAML 用于表达「合规立场」与「为什么这样处理」
2. Claude 的配置读取本质是文本理解，不需要机器解析的结构化格式
3. 引入独立 `profile.json` 会产生双重真相来源，维护成本高且容易产生不一致
4. 沿用 commercial-legal / employment-legal 已验证的散文单一真相来源模式

**纠正现有访谈表述：** 现有 `cold-start-interview/SKILL.md` 末尾写有「生成数据合规实践配置文件（YAML 格式）」。这一表述应理解为：冷启动访谈子技能收集信息后，**将结果填充到 `data-compliance/CLAUDE.md` 的对应占位符章节**，而非另外保存独立的 YAML 文件。05-03 深化 cold-start-interview 时须按此契约修订。

## 三、技能读取契约

### 3.1 技能读取时机

**在执行任何实质性工作之前**，每个技能（或其编排入口）必须：

1. 读取 `data-compliance/CLAUDE.md`
2. 检查关键字段是否已填充（不含 `[PLACEHOLDER]`）
3. 根据检查结果选择运行模式

### 3.2 配置缺失或含占位符时的处理

当技能检测到 `data-compliance/CLAUDE.md` 缺失或关键字段含 `[PLACEHOLDER]` 时，**不得静默使用猜测值**，须按以下逻辑处理：

**标准提示（配置未完成）：**

> 我注意到你还没有配置数据合规实践档案——这是让我根据你的数据处理者画像、合规立场（敏感数据政策/跨境政策）和风险校准来定制输出的关键。
>
> **两个选择：**
> - 运行 `/data-compliance:cold-start-interview`（约 5-8 分钟）来配置你的档案，然后我会根据你的真实合规立场进行工作。
> - 说 **「临时模式」**，我会使用通用默认值——并在每个输出上标记 `[临时模式 — 配置你的数据合规档案以获得定制输出]`。

### 3.3 临时模式默认值

| 字段 | 临时模式默认值 |
|-----|-------------|
| 整体合规偏好 | 平衡型 |
| 执业角色 | 律师（数据合规律师） |
| 数据处理者角色 | 个人信息处理者（非受托处理者） |
| 是否为 CIIO | 否（默认，若迹象显示可能为 CIIO 则提示） |
| 处理个人信息规模 | 不明（逐案判断出境门槛，不预设路径） |
| 敏感数据政策 | 按严格门槛标记常见风险（生物识别/医疗/金融账户/未成年人）；触发单独同意提示 |
| 跨境政策 | 按门槛逐案判定，不预设首选路径；涉及出境时提示启动路径分析 |
| 数据安全立场 | 按数据分类分级保守处理（不确定即标重要数据风险） |
| 风险等级 | 使用配置文件中预设的🔴🟠🟡🟢定义 |
| 文书风格 | 结构化清单 + 整改建议；含法规引用；不含案例引用 |

## 四、数据处理者/受托处理者/CIIO 立场选择

### 4.1 立场确认流程

技能在执行实质性工作前须从 `## 我们是谁` 读取并确认：

1. **处理者角色**：个人信息处理者 vs 受托处理者（委托处理场景）vs 两者兼有
2. **CIIO 状态**：是否为关键信息基础设施运营者

### 4.2 处理者立场对义务强度的影响

| 身份 | 义务重点 | 额外义务 |
|------|---------|---------|
| 个人信息处理者 | PIPL 全章义务；告知-同意；PIA；个人权利响应；个人信息保护负责人 | — |
| 受托处理者（委托处理） | 按处理者指示处理；不得超授权处理；配合处理者履行义务；配合 PIA | 需签订委托处理协议约定双方义务 |
| 两者兼有 | 对外作为处理者履行义务；对委托方作为受托处理者 | 须清晰区分两种角色下的数据流 |
| CIIO（关基运营者） | 在上述基础上：数据本地化；重要数据出境须安全评估（DSL第31条，已核实）；关基保护条例义务 `[待验证]` | 重要数据出境须安全评估，不可走标准合同路径 |

### 4.3 不明确时的处理

若 `## 我们是谁` 未填写或含占位符，技能须询问：

> 在开始分析前，请确认：
> 1. 贵方在本次数据处理活动中的角色是：（A）个人信息处理者；（B）受托处理者；（C）两者兼有
> 2. 贵方是否为关键信息基础设施运营者（CIIO）？（是/否/不确定）

## 五、重跑 / 编辑 / 版本控制

### 5.1 重跑访谈

律师/DPO 可随时重新运行完整访谈或任一子技能（05-03 深化后可用）：

- **完整重跑**：`/data-compliance:cold-start-interview` — 顺序填充全部字段
- **部分更新**：单独触发某一子技能，只更新配置文件的对应章节

### 5.2 直接编辑

律师/DPO 可直接编辑 `data-compliance/CLAUDE.md`，无需重跑访谈。编辑后即时生效，下次技能调用自动读取新值。推荐直接编辑的场景：

- 更新 DPO/负责人联系人
- 调整升级矩阵（如新增角色、调整权限）
- 更新处理个人信息规模（系统扩容后）
- 根据新版办法更新跨境政策中的路径偏好（须注意时效性）

### 5.3 版本控制

- `data-compliance/CLAUDE.md` 纳入 git 版本控制（与代码同仓库）
- 查看历史版本：`git log --oneline data-compliance/CLAUDE.md`
- 查看变更内容：`git diff HEAD~1 data-compliance/CLAUDE.md`
- 回滚方式：`git checkout <commit-hash> -- data-compliance/CLAUDE.md`

## 六、法律事实合理性检查

本节定义冷启动访谈子技能在写入配置文件前的合理性检查规则。

### 6.1 适用范围

对以下类型的用户口述内容进行合理性检查，写入前须核实：

- 具体法规条款号（如「PIPL 第52条要求处理超过X万条记录须指定负责人」）
- 出境门槛数值（如「标准合同适用于处理不超过X万条记录」）——属于网信办办法规定，须标 `[待验证 — 数值随办法修订变动]`
- 具体期限（如「评估有效期2年」「日志留存6个月」「PIA 记录存3年」）——PIPL第56条已核实3年；其他须核实
- 引用国家标准编号（须标 `[待验证]`）

### 6.2 数据合规特别规则（阈值待验证）

**重要规则**：凡用户口述涉及网信办办法的具体数值（出境数量门槛、备案时限、评估有效期、认证有效期等），均须标 `[待验证 — 数值随办法修订变动，须核实最新版]`，不得直接写入为「确认值」。

提示格式：

> 你提到标准合同路径适用于处理 [X] 万条记录以下；这是基于 [某版本] 的理解。由于网信办办法定期修订，具体门槛数值须核实最新版。建议写入为「参考值 [X] 万条 `[待验证 — 须核实最新版标准合同办法门槛]`」？`[前提 flagged — 需验证]`

### 6.3 合理性检查写入前确认流程

对于所有 `[前提 flagged — 需验证]` 标记的字段：

1. 显示检测到的潜在问题
2. 提供两个选项：(a) 按用户口述写入（保留 `[前提 flagged — 需验证]` 标注）；(b) 按建议值修改后写入（保留 `[待验证]` 标注）
3. 不得静默写入未经确认的存疑值
