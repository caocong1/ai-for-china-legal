# 安全评估申报

> **可单独触发**：本子技能可脱离 cross-border-assessment 编排入口独立运行。
> 斜杠命令：`/data-compliance:security-assessment-filing`
>
> **前置依赖**：
> - 读取 `data-compliance/CLAUDE.md` → 获取：CIIO 身份、数据处理者类型、跨境政策、重要数据出境立场、文书风格
> - 引用 `data-compliance/skills/_shared/data-protection-citations.md` → 出境评估办法（E.1 条目）
> - 引用 `data-compliance/skills/_shared/legal-basis-conventions.md` → 部门规章引用规范（不写裸条号，描述规则）
> - 建议已完成 `pathway-determination`（路径判定）确认须走路径①安全评估
>
> **配置缺失处理**：若 CLAUDE.md 中 CIIO 身份、跨境政策为 `[PLACEHOLDER]`，启用临时模式：
> - 临时模式：假设须强制申报；申报材料清单按最全集合准备；所有门槛数值按 `[待验证]` 处理
>
> **分工说明**：本子技能覆盖安全评估的申报环节；路径判定见 `pathway-determination`；标准合同路径见 `standard-contract`；最终合规清单与报告组装见 `cross-border-assembly`。

---

## 目的

依据《数据出境安全评估办法》（2022年，网信办），指导数据处理者完成数据出境安全评估申报，包括：判断是否须强制申报、开展出境风险自评估、准备申报材料包、了解申报流程与评估有效期，以及识别须重新评估的情形。

---

## 详细步骤

### 步骤一：申报情形判断

根据《数据出境安全评估办法》（2022年，网信办）规定，以下情形须申报数据出境安全评估（**具体情形与数量门槛 `[待验证 — 须核实最新版评估办法条文]`**）：

| 情形 | 判断要素 | 须申报结论 |
|------|---------|----------|
| 情形A：出境重要数据 | 出境数据中包含重要数据（依 DSL 第21条认定） | 🔴 须申报 |
| 情形B：CIIO 出境个人信息或重要数据 | 数据处理者为关键信息基础设施运营者 | 🔴 须申报 |
| 情形C：处理个人信息达到规定数量的处理者出境个人信息 | 处理个人信息累计超过规定数量门槛 `[待验证 — 具体数量门槛须核实最新版评估办法]` | 🔴 须申报 |
| 情形D：出境敏感个人信息达到规定数量 | 出境敏感个人信息的人数超过规定门槛 `[待验证 — 具体数量门槛须核实最新版评估办法]` | 🔴 须申报（待验证） |
| 情形E：其他网信部门认定须评估的情形 | 网信部门作出认定 `[待验证]` | 视情况 |

**申报情形判断工作表**：

| 判断项 | 是/否/不确定 | 备注 |
|-------|------------|------|
| 本次出境数据包含重要数据 | — | 建议先完成 `data-security-compliance/important-data-handling` |
| 本组织已认定为 CIIO | — | 来源：CLAUDE.md 或行业主管部门认定 |
| 本组织处理的个人信息达到规模门槛 | — | **门槛数值 `[待验证]`** |
| 本次出境的个人信息数量达到门槛 | — | **门槛数值 `[待验证]`** |

**判断结论**：
- 如上述任一项为「是」→ **须强制申报安全评估**，继续步骤二
- 如全部为「否」→ 可考虑标准合同路径（转 `standard-contract`）或保护认证路径（转 `cross-border-assembly`）
- 如存在「不确定」项 → 保守处理，按须申报对待；建议专业机构确认

| 保护措施 | 是否已落实 |
|---------|----------|
| 技术措施（加密传输、访问控制） | ☐ |

**自评估结论**（从以下选择）：

- **可以出境**：综合评估风险可控，接收方保护能力满足要求，已采取充分保护措施
- **须整改后出境**：存在可整改的合规缺口（列出整改项），整改完成后可继续申报
- **暂不具备出境条件**：存在重大合规风险，当前不建议推进出境活动

### 步骤三：申报材料准备

申报须向国家网信部门提交申报材料（**具体材料清单以最新版评估办法为准 `[待验证 — 材料清单随办法修订变动]`**），以下为参考性材料框架：

| 材料类型 | 内容要点 | 准备状态 |
|---------|---------|---------|
| **申报书** | 数据处理者基本信息、出境数据概况、路径选择理由、申报事由说明 | ☐ |
| **出境风险自评估报告** | 覆盖步骤二全部评估维度，含结论与整改计划（如有） | ☐ |
| **与境外接收方拟订立的法律文件** | 合同草案或已签订合同；须包含数据保护义务条款、个人权利保障机制、违约责任 | ☐ |
| **其他证明性材料** | 视评估办法规定而定（**具体要求 `[待验证]`**） | ☐ |

**申报材料质量要点**：
- 自评估报告须客观反映风险，不得刻意回避高风险项
- 法律文件须体现接收方承担与境内相当的保护义务
- 材料须与实际数据处理活动一致，不得伪造或误导

### 步骤四：申报流程与时间安排

**申报路径**：向省级以上网信部门提交申报材料，由国家网信部门负责组织安全评估（**具体受理流程以最新规定为准 `[待验证]`**）。

| 阶段 | 主要工作 | 时间预估 |
|------|---------|---------|
| 准备阶段 | 完成自评估、准备申报材料包 | 依材料完整性定 |
| 申报提交 | 向省级网信部门提交申报 | 申报当日 |
| 受理确认 | 网信部门确认材料是否完整（**受理期限 `[待验证]`**） | — |
| 安全评估 | 国家网信部门组织安全评估（**评估期限 `[待验证 — 随办法修订变动，须核实最新版]`**） | — |
| 结果出具 | 出具安全评估结果 | — |
| 整改补充（如需） | 如评估发现问题须整改后重新提交 | — |

> **重要提示**：数据出境安全评估须在数据出境**之前**完成；不得在未通过评估情况下启动数据出境。评估通过前，出境活动须暂停。

### 步骤五：评估有效期与重新评估情形

安全评估通过后，评估结果具有一定有效期（**具体有效期年限 `[待验证 — 数值随办法修订变动，须核实最新版]`**）。

**须重新申报安全评估的情形**（**以评估办法最新版本为准 `[待验证]`**）：

| 重新评估触发情形 | 说明 |
|--------------|------|
| 评估有效期届满 | **有效期 `[待验证]`**；届满前须提前申请续评 |
| 出境目的/方式/数据种类发生变化 | 原评估范围无法覆盖新变化 |
| 境外接收方数据保护政策/法律环境发生重大变化 | 需重新评估接收方保护能力 |
| 发生数据安全事件 | 事件影响可能改变原有风险判断 |
| 其他评估办法规定的重新评估情形 `[待验证]` | — |

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 评估申报未通过 | 禁止出境（🔴 强制要求）；针对评估意见整改；整改完成后重新申报 |
| 评估有效期届满但尚未续评 | 暂停出境活动；立即启动续评申请；标 🔴 |
| 出境活动发生变化须重新评估但企业未识别 | 属合规缺口（🔴）；识别后立即启动重新评估并暂停超出原评估范围的出境活动 |
| 申报材料与实际情况不符 | 属违规申报，可能影响评估通过及后续出境合规性（🔴）；须整改申报材料 |
| 紧急业务需求须立即出境 | 无例外——数据出境安全评估审批不可绕过；可临时暂缓业务，优先推进申报流程 |
| 申报过程中监管部门要求补充材料 | 积极配合，按要求时限补充；须视补充材料内容决定是否暂停出境活动 |

## 错误处理

| 错误类型 | 触发情形 | 处理方式 |
|---------|---------|---------|
| 申报情形判断不确定 | 数据量、CIIO 身份等关键信息缺失 | 暂停申报；列出待确认项；保守处理按须申报对待 |
| 自评估报告质量不足 | 遗漏重要评估维度或结论不支持实际风险水平 | 退回完善；不得以不完整报告申报 |
| 接收方信息不完整 | 无法完成接收方保护能力评估 | 暂停材料准备；向接收方索取安全信息（安全问卷/认证证书/合规报告等） |
| 网信办规章最新版本与当前指引不符 | 办法已修订 | 以最新版评估办法为准；本指引相应条目重新核实 |
| 申报系统/流程问题 | 网信部门申报系统变更或流程不明 | 联系当地省级网信部门咨询；或通过行业协会了解最新申报实践 |
