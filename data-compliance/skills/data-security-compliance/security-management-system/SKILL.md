> **可单独触发**：本子技能可脱离 data-security-compliance 编排入口独立运行。

## 目的

依据《数据安全法》（2021年）关于数据处理者须建立数据安全管理制度的要求（**具体条款 `[待验证 — 须核实 DSL 配套条文]`**），建立覆盖数据收集/存储/使用/加工/传输/提供/公开/删除全生命周期的数据安全管理制度，涵盖组织保障、安全教育培训与技术措施，满足数据处理者的基本数据安全合规义务。

---

## 详细步骤

### 步骤一：数据安全管理制度框架建立

数据安全管理制度须覆盖数据处理全生命周期的八个阶段（**《数据安全法》规定的全流程 `[待验证具体条号]`**）：

| 生命周期阶段 | 主要安全管理要求 | 关键措施 |
|---------|--------------|---------|
| **收集** | 最小化原则；合法来源；知情同意（个人信息） | 数据收集审批；收集渠道审查；告知同意机制 |
| **存储** | 安全存储介质；访问控制；数据隔离；备份与恢复 | 分级存储（按数据级别）；加密存储重要/敏感数据；备份机制 |
| **使用** | 最小权限原则；使用目的限制；操作日志 | 权限管理制度；使用审批流程（重要数据） |
| **加工** | 匿名化/去标识化要求（如适用）；处理结果安全 | 加工处理日志；数据质量控制 |
| **传输** | 安全传输渠道（加密）；传输完整性 | 传输加密（TLS/VPN 等）；传输记录 |
| **提供** | 数据共享审批；接收方评估；合同约束 | 数据共享协议；接收方安全评估；共享记录 |
| **公开** | 公开前审查（含重要数据不可公开）；公开范围控制 | 公开审批；去标识化处理 |
| **删除** | 到期删除；安全销毁；删除记录 | 数据保留期限制度；安全销毁证明 |

**制度文件体系**：

| 制度层级 | 文件类型 | 主要内容 |
|---------|---------|---------|
| 顶层制度 | 数据安全管理总则 | 分级保护原则、责任主体、制度适用范围 |
| 专项制度 | 数据分类分级管理制度 | 分级标准、评级流程、级别调整 |
| 专项制度 | 数据访问控制制度 | 权限申请审批、最小权限、特权账号管理 |
| 专项制度 | 数据共享与对外提供制度 | 审批流程、合同要求、出境合规 |
| 专项制度 | 重要数据管理制度 | 识别、台账、风险评估报送、出境限制 |
| 专项制度 | 数据安全事件应急处置制度 | 事件分级、处置流程、报告要求 |
| 操作规程 | 数据操作规程 | 各岗位数据处理操作规范 |
| 记录表单 | 数据安全记录台账 | 处理活动记录、权限申请记录、共享记录、事件记录等 |

### 步骤二：组织保障

#### 2.1 数据安全负责人（必须建立）

| 类型 | 适用对象 | 职责 |
|------|---------|------|
| 数据安全负责人（DSL 要求，**重要数据处理者强制** `[待验证具体要求]`） | 处理重要数据的数据处理者 | 统筹管理数据安全，对数据安全工作负总责 |
| 个人信息保护负责人（PIPL 第52条 `[待验证]`，规模达到要求时须设立） | 处理个人信息达到规定规模的处理者 | 个人信息保护合规监督、权利响应 |

**两类负责人可由同一人担任**，但须明确各自职责范围（数据安全 vs. 个人信息保护）。

**负责人核心职责清单**：
- [ ] 制定并实施数据安全管理制度
- [ ] 组织开展数据安全教育培训
- [ ] 协调处理数据安全事件
- [ ] 监督数据处理活动合规性
- [ ] 向高层管理层报告数据安全状态
- [ ] 对接监管部门（检查/问询/报送）

#### 2.2 数据安全管理机构

| 适用对象 | 机构形式 | 职责范围 |
|---------|---------|---------|
| 重要数据处理者（须设立专门机构 `[待验证]`） | 数据安全委员会/数据安全管理部门 | 制度制定、审批、监督、培训、事件处置 |
| 一般数据处理者 | 可由现有法务/IT/合规部门承担 | 数据安全相关合规职能 |
| CIIO | 须建立专门安全管理机构（关基保护条例要求 `[待验证]`） | 包含关基安全保护相关职能 |

#### 2.3 责任矩阵（RACI）

| 数据安全活动 | 数据安全负责人 | 各业务部门 | IT/安全团队 | 法务/合规 |
|---------|--------------|---------|-----------|---------|
| 数据分类分级 | 主责 | 配合（提供数据信息） | 配合（技术标签）| 审核 |
| 数据共享审批 | 审核 | 提出申请 | 技术实施 | 合规审查 |
| 重要数据台账维护 | 主责 | 配合（数据资产信息）| 配合（系统信息）| — |
| 风险评估（重要数据） | 主责 | 配合 | 配合（技术测评）| 报告撰写 |
| 安全事件处置 | 主责 | 配合 | 技术处置 | 法律应对 |
| 监管沟通/报送 | 主责 | — | 配合 | 主责（法律层面）|

### 步骤三：安全教育培训

数据安全法要求数据处理者须开展数据安全教育培训（**具体条款 `[待验证]`**）：

| 培训对象 | 培训重点 | 建议频率 |
|---------|---------|---------|
| 全员（含管理层） | 数据安全基本要求、数据分类分级常识、个人信息保护义务、违规后果 | 每年至少一次 |
| 数据处理相关岗位 | 本岗位数据操作规程、权限管理要求、事件识别与报告 | 每年 + 上岗前 |
| 技术/IT 人员 | 数据安全技术措施实施（加密/访问控制/安全配置）、漏洞管理 | 每年 + 技术变更时 |
| 管理层 | 数据安全法律义务、合规风险、重大事件处置决策 | 每年 |
| 数据安全负责人 | 最新法规动态、实践案例、跨部门协调 | 持续学习 |

**培训记录要求**：
- 保留培训记录（培训内容、参与人员、日期、测试结果）
- 建议至少保存 **三年**（与 PIA 报告保留期一致，PIPL 第56条）

### 步骤四：技术措施

| 技术措施类别 | 具体措施 | 适用对象 | 优先级 |
|---------|---------|---------|-------|
| **加密保护** | 数据存储加密（静态加密）；数据传输加密（TLS/VPN）；重要/敏感数据强制加密 | 全部数据，重要数据须强加密 | 🔴 高 |
| **访问控制** | 身份认证（多因素认证 MFA）；最小权限分配；特权账号管理（PAM）；权限定期审查 | 全部系统，重要数据系统强制 MFA | 🔴 高 |
| **安全审计** | 操作日志记录（用户、操作类型、时间、数据范围）；日志留存期符合监管要求（**具体期限 `[待验证]`**）；定期日志审查 | 全部数据处理系统 | 🔴 高 |
| **数据备份** | 定期备份（重要数据须有容灾备份）；备份加密；备份恢复测试 | 全部重要数据及关键业务数据 | 🟠 高 |
| **容灾与恢复** | 业务连续性计划（BCP）；灾难恢复计划（DRP）；RTO/RPO 目标 | 重要系统与重要数据 | 🟠 高 |
| **数据隔离** | 重要数据与一般数据分区存储；生产/测试环境数据隔离 | 重要数据 | 🟠 高 |
| **漏洞管理** | 定期漏洞扫描；补丁及时修复；等保测评（见 network-security-compliance，05-03） | 全部系统 | 🟡 中等 |
| **数据水印/溯源** | 重要数据传输/共享时加水印；数据泄露溯源能力 | 重要数据（视业务需要） | 🟡 中等 |
| **脱敏/去标识化** | 测试/开发环境数据脱敏；对外提供数据去标识化（视情况匿名化） | 非生产环境；对外提供场景 | 🟡 中等 |

**CIIO 额外技术要求**（`[CSL关基衔接 — 需核实关基条例技术要求]`，详见 network-security-compliance）：
- 等级保护不低于三级（部分关基须达到四级）
- 关基专项安全检测（**具体要求 `[待验证]`**）

### 步骤五：制度落地审查

定期审查制度落地情况：

| 审查维度 | 审查要点 | 建议频率 |
|---------|---------|---------|
| 制度完整性 | 八阶段管理制度是否完整、是否已更新（反映最新法规/业务变化） | 年度 |
| 制度执行性 | 实际操作是否遵循制度；操作记录是否完整 | 半年度 |
| 技术措施有效性 | 加密/访问控制/日志等技术措施是否正常运行 | 季度 |
| 权限合规性 | 最小权限是否落实；权限变更是否及时同步 | 季度 |
| 培训落实情况 | 培训是否按计划开展；培训记录是否完整 | 年度 |
| 事件处置有效性 | 上期安全事件处置总结与改进 | 事件后 + 年度 |

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 制度存在但内容严重滞后（未更新至 DSL/PIPL 时代） | 优先更新制度，标为 🔴 整改项；建议 [时限] 内完成 |
| 中小企业资源有限，无法配置专职安全团队 | 指导性：最低限度应做到（加密传输/访问控制/事件报告机制）；提供精简制度框架 |
| CIIO 的技术要求与一般制度不同 | 标注「CIIO 额外要求，交由 network-security-compliance 处理」，不在本子技能中重叠 |
| 已有现成制度体系（如 ISO 27001 认证） | 评估现有体系与 DSL/PIPL 要求的对应和差距；补充缺失的中国法要求部分 |
| 集团型企业（境内外多个主体） | 各境内主体须独立满足中国法要求；境外主体适用各自属地法；集团统一制度须在境内主体层面有效执行 |

## 错误处理

| 错误类型 | 触发情形 | 处理方式 |
|---------|---------|---------|
| 制度文件缺失关键章节 | 审查发现某生命周期阶段缺少制度 | 标为整改项；提供该阶段制度要点模板 |
| 负责人未指定 | 重要数据处理者无数据安全负责人 | 标为 🔴 强制整改；建议 30 天内完成指定 |
| 技术措施未落实 | 关键技术措施（加密/访问控制）未落地 | 标为 🔴/🟠 整改项；评估过渡期风险控制方案 |
| 制度与实际操作严重脱节 | 有制度但员工不执行 | 重大合规风险；建议开展专项合规审查；加强培训与监督执行 |
