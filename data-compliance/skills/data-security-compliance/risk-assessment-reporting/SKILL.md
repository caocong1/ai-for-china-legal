## 目的

依据《数据安全法》（2021年）的风险监测、安全事件应急处置与报告、数据交易合规等规定（**具体条款 `[待验证 — 须核实 DSL 配套条文号]`**），建立覆盖以下四个方面的合规框架：(1) 重要数据处理者的定期风险评估与主管部门报送；(2) 数据安全风险监测机制；(3) 数据安全事件应急处置与报告；(4) 数据交易合规审查；并就境外执法数据调取请求的合规应对提供框架。

## 详细步骤

### 步骤一：重要数据处理者定期风险评估

**适用对象**：处理重要数据的数据处理者（须履行定期风险评估与报送义务，**具体义务条款与频率 `[待验证 — 须核实 DSL 配套规定]`**）。

#### 1.1 风险评估范围

| 评估维度 | 评估内容 | 评估方法 |
|---------|---------|---------|
| **数据处理合法合规性** | 重要数据处理活动是否有充分法律依据；处理目的/方式/范围是否符合法定要求 | 文档审查 + 处理活动访谈 |
| **数据分类分级准确性** | 重要数据识别是否全面；新增数据是否及时纳入评估 | 数据清单核对 + 业务访谈 |
| **安全措施有效性** | 加密/访问控制/审计日志等技术措施是否正常运行且有效 | 技术测试 + 日志审查 |
| **人员管理合规性** | 权限管理、培训落实、离职数据处理 | 权限台账审查 + 培训记录 |
| **数据共享与出境合规** | 共享场景合规性；出境场景安全评估状态 | 共享/出境记录审查 |
| **威胁与漏洞识别** | 网络攻击威胁、系统漏洞、内部威胁 | 漏洞扫描 + 威胁情报 + 渗透测试（重要系统） |
| **安全事件回顾** | 上期发生的安全事件及处置情况、改进措施落实 | 事件记录审查 |

#### 1.2 风险评估周期与要点表

| 评估类型 | 频率 | 主要输出 |
|---------|------|---------|
| **定期全面风险评估**（重要数据处理者 `[待验证]`） | **频率 `[待验证 — 须核实 DSL 配套规定]`**，建议至少年度一次 | 风险评估报告 |
| **触发式风险评估** | 发生重大安全事件/出境安全评估前/重大业务变化时 | 专项评估报告 |
| **日常安全检查** | 月度/季度 | 安全检查记录 |

#### 1.3 向主管部门报送风险评估报告

重要数据处理者须向有关主管部门报送风险评估报告（**报送周期、对象、渠道 `[待验证 — 须核实行业主管部门要求]`**）：

| 报送要素 | 内容 |
|---------|------|
| 报送对象 | 行业主管部门（**具体部门 `[行业监管 flagged — 需核实主管部门]`**） |
| 报送周期 | **`[待验证 — 须核实行业主管部门规定]`** |
| 报告核心内容 | 重要数据识别范围、处理活动合规性、安全措施现状、风险识别与应对、历史事件回顾、改进计划 |
| 报送渠道 | **`[待验证 — 须核实主管部门指定渠道]`** |

**报送前审查**：
- 报告须经数据安全负责人审签
- 向主管部门提交的报告不受律师-当事人特权保护（来源：CLAUDE.md 目的地/特权检查护栏）

### 步骤二：数据安全风险监测机制

建立日常风险监测机制，实现对数据安全风险的持续感知：

| 监测维度 | 监测方式 | 告警机制 | 负责团队 |
|---------|---------|---------|---------|
| 网络入侵与异常访问 | 安全信息和事件管理系统（SIEM）/入侵检测系统（IDS）；日志分析 | 异常行为告警（设置基线与阈值） | IT/安全团队 |
| 数据异常访问 | 数据库活动监控（DAM）；异常查询、批量下载检测 | 超阈值告警；非授权访问告警 | IT/安全团队 |
| 漏洞跟踪 | 定期漏洞扫描；CVE 订阅；补丁管理 | 高危/严重漏洞即时告警 | IT/安全团队 |
| 重要数据异常流出 | 数据泄露防护（DLP）；传输监测 | 重要数据异常传输告警 | IT/安全团队 |
| 外部威胁情报 | 威胁情报订阅；行业信息共享 | 高相关性威胁告警 | IT/安全团队 |

**监测记录**：所有监测日志须按规定留存（**日志留存期限 `[待验证 — 须结合等保要求与 DSL 配套规定核实]`**）。

### 步骤三：数据安全事件应急处置与报告

#### 3.1 数据安全事件分级

| 事件等级 | 定义（参考标准，**具体分级以主管部门规定为准 `[待验证]`**） | 典型场景 |
|---------|--------------------------------------------------|---------|
| **特别重大**（Ⅰ级） | 可能对国家安全、社会稳定或重大公共利益造成严重危害；涉及国家核心数据或大规模重要数据 | 国家关键基础设施数据大规模泄露 |
| **重大**（Ⅱ级） | 涉及重要数据或大规模个人信息；可能影响社会秩序或重大公共利益 | 超过百万条个人信息泄露；重要行业数据被窃取 |
| **较大**（Ⅲ级） | 较大规模个人信息泄露；较为重要的业务数据泄露或破坏 | 数万条个人信息泄露；内部重要业务数据泄露 |
| **一般**（Ⅳ级） | 小规模个人信息泄露或一般数据安全事件 | 少量用户数据泄露；内部非重要数据泄露 |

#### 3.2 应急处置流程

```
事件发生/发现
    ↓
[1] 立即响应（发现后即刻）
    - 确认事件真实性
    - 启动应急响应小组
    - 临时隔离受影响系统/数据（止损）

    ↓
[2] 初步评估（[时限待验证]）
    - 确认事件类型与影响范围（数据类型/数量/受影响个人数）
    - 识别事件来源/原因
    - 确定事件等级（Ⅰ/Ⅱ/Ⅲ/Ⅳ级）

    ↓
[3] 报告决策
    - 特别重大/重大事件 → 立即启动向主管部门报告（报告时限[待验证]）
    - 涉及个人信息泄露 → 评估是否须通知受影响个人（[条件与时限待验证]）
    - 通知法务/合规/数据安全负责人

    ↓
[4] 深度处置（持续进行）
    - 遏制事件传播（封堵漏洞/恢复系统）
    - 证据保全（日志/现场）
    - 系统恢复与业务连续性

    ↓
[5] 事件报告（向主管部门）
    - 初报（事件基本情况）[待验证时限]
    - 进展报告（处置进展）[待验证]
    - 终报（事件全貌与整改）[待验证]

    ↓
[6] 事后复盘
    - 根本原因分析
    - 制度/技术改进措施
    - 复盘报告
```

#### 3.3 安全事件应急处置与报告表

| 报告内容 | 初报 | 进展报告 | 终报 |
|---------|------|---------|------|
| 事件基本情况 | ✓ | — | ✓（完整版） |
| 影响范围（数据类型/量级/受影响个人） | 初步估算 | 更新 | 最终确认 |
| 事件起因与经过 | 初步 | 更新 | 最终 |
| 已采取的应对措施 | ✓ | 更新 | ✓ |
| 后续整改计划 | — | — | ✓ |

#### 3.4 向主管部门报告

发生数据安全事件，须及时向有关主管部门报告（**报告时限、对象、渠道 `[待验证 — 须核实 DSL 配套规定及行业主管部门要求]`**）：

| 报告对象 | 报告时限 | 内容要求 | 验证状态 |
|---------|---------|---------|---------|
| 行业主管部门 | `[待验证 — 须核实行业规定]` | 事件基本情况、影响范围、已采取措施 | `[待验证]` |
| 网信部门（个人信息相关）| `[待验证 — 须核实 PIPL 配套规定]` | 个人信息泄露情况 | `[待验证]` |
| 公安部门（涉及刑事的）| `[待验证]` | 涉嫌犯罪情况 | `[待验证]` |

#### 3.5 受影响个人通知义务

发生个人信息泄露事件，可能须通知受影响的个人（**通知触发条件与时限 `[待验证 — 须核实 PIPL 及配套规定]`**）：

| 通知内容 | 说明 |
|---------|------|
| 泄露事件描述 | 何时/何种数据/可能后果 |
| 已采取措施 | 已采取的补救措施 |
| 个人可采取的保护措施 | 如更改密码、警惕诈骗等 |
| 投诉/咨询渠道 | 企业联系方式 |

### 步骤四：数据交易合规

数据处理者参与数据交易（出售、购买或提供数据中介服务）时须满足以下合规要求（**具体义务条款 `[待验证 — 须核实 DSL 相关规定]`**）：

#### 4.1 数据来源合法性

| 交易角色 | 义务 |
|---------|------|
| 数据卖方 | 须能证明数据来源合法（合法收集/处理授权）；不得出售违法收集/处理的数据 |
| 数据中介机构 | 须要求交易双方说明数据来源；审核双方身份；留存审核和交易记录（**留存期限 `[待验证]`**）|
| 数据买方 | 须核实数据合法来源；不得购买明知来源违法的数据 |

#### 4.2 数据交易合规清单

- [ ] 数据来源合法性已验证（收集/处理授权链清晰）
- [ ] 涉及个人信息的数据：原始处理合法性已确认（符合 PIPL 合法性基础）
- [ ] 涉及重要数据：重要数据不得随意交易，须确认是否符合重要数据共享规定 `[行业监管 flagged]`
- [ ] 数据交易合同已签订，含数据用途限制、安全措施要求、违约责任
- [ ] 数据中介机构（如适用）已履行审核义务并保存记录
- [ ] 数据交易记录已建立（数据内容、交易时间、交易方、用途）
- [ ] 国家核心数据：**绝对禁止** 进入数据交易市场（🔴 硬性要求）

#### 4.3 重要数据的交易限制

> 重要数据须谨慎评估是否可交易，须确认无禁止性规定且符合数据安全要求（`[行业监管 flagged — 需核实主管部门规定]`）。

### 步骤五：境外执法数据调取请求的合规应对

当境外司法或执法机构要求提供境内存储的数据时，须审慎处理（**DSL 相关规定 `[待验证 — 须核实 DSL 具体条款]`**）：

| 处置原则 | 说明 |
|---------|------|
| 🔴 **未经批准不得提供** | 境内数据未经主管机关批准，不得向外国司法或执法机构提供（**DSL 相关规定 `[待验证]`**） |
| **立即法律咨询** | 收到境外调取请求后，立即通知法务/外部律师；评估应对义务与时限 |
| **向主管部门报告** | 向有关主管部门报告收到的境外调取请求，听取处置意见（**具体报告要求 `[待验证]`**） |
| **评估合法途径** | 是否有司法协助条约（MLAT）等合法渠道；任何数据提供须经合法程序 |
| **不得单方面向境外提供** | 在获得主管机关批准前，不得单方面向境外执法机构提供任何数据（🔴 强制） |

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 安全事件等级无法立即确定 | 先按较高等级处理（🟠/🔴），同时加速评估；不要因等级不确定而延误报告 |
| 跨国集团内部数据泄露（境外主体泄露境内个人信息） | 境内个人信息保护义务仍须满足；须评估是否触发 PIPL 相关义务；境外主体数据事件也可能影响境内报告义务 |
| 数据交易对方无法证明数据合法来源 | 终止交易（🔴 风险不可接受）；合理尽调未发现，但事后发现违法时，须评估买方责任 |
| 境外执法机构以紧急为由要求立即提供数据 | 紧急情形不构成绕过中国法律程序的理由（🔴）；立即咨询律师并向主管部门报告；向境外机构说明须经中国合法程序 |
| 数据安全事件与个人信息泄露同时发生 | 两套报告义务并行：DSL 数据安全事件报告 + PIPL 个人信息泄露通知（须分别满足各自时限） |

## 错误处理

| 错误类型 | 触发情形 | 处理方式 |
|---------|---------|---------|
| 未建立应急响应机制即发生事件 | 企业无应急预案 | 临机处置，优先止损；同步建立临时应急小组；事件处置后立即建立正式应急机制 |
| 风险评估报送逾期 | 未按规定频率/时限向主管部门报送 | 评估是否须说明情况；标为合规缺口（🟠/🔴）；立即补报并建立提醒机制 |
| 数据交易后发现数据来源违法 | 购买/使用了违法来源数据 | 立即停止使用；向主管部门报告（如适用）；评估损害赔偿与责任 |
| 报告内容与实际事件不符 | 向主管部门提交了不实报告 | 极高合规风险（🔴）；立即纠正；可能须主动向主管部门说明 |
