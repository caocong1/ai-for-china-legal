# 重要数据识别与处理

> **可单独触发**：本子技能可脱离 data-security-compliance 编排入口独立运行。
> 斜杠命令：`/data-compliance:important-data-handling`
>
> **前置依赖**：
> - 读取 `data-compliance/CLAUDE.md` → 获取：所属行业、CIIO 身份、重要数据出境立场、重要数据识别立场
> - 引用 `data-compliance/skills/_shared/data-protection-citations.md` → DSL 第31条（已核实）、DSL 第21条（已核实）
> - 引用 `data-compliance/skills/_shared/legal-basis-conventions.md` → 行业监管标注规范
> - 建议已完成 `data-classification-grading`（数据分类分级）以获取初步重要数据清单
>
> **配置缺失处理**：若 CLAUDE.md 中 CIIO 身份、行业不确定，启用临时模式：
> - 临时模式：重要数据按保守标准从严推定；CIIO 不确定时按 CIIO 处理
>
> **分工说明**：本子技能负责重要数据识别与处理义务评估；数据分类分级三级框架见 `data-classification-grading`；数据安全管理制度全流程见 `security-management-system`；风险评估报告机制见 `risk-assessment-reporting`；重要数据出境的安全评估申报见 `cross-border-assessment/security-assessment-filing`。

---

## 目的

依据《数据安全法》（2021年）第31条（**已核实锚点，建议复核**）及第21条，明确重要数据的识别方法、重要数据处理者的强制性义务（明确负责人和管理机构、定期风险评估报送），以及重要数据出境须经安全评估的合规路径。本子技能是数据安全合规的核心节点——重要数据的错误识别或义务遗漏将导致重大合规风险。

---

## 详细步骤

### 步骤一：重要数据识别方法

重要数据须对照以下来源判断，识别所有可能属于重要数据的数据类型：

#### 1.1 对照行业主管部门重要数据目录

> **重要数据目录由各地区、各部门负责制定本行业本领域的重要数据具体目录。**
> 企业须主动向行业主管部门获取或查阅适用目录。 `[行业监管 flagged — 需核实主管部门规定]`

| 行业 | 主要目录制定机关 | 目录状态（参考） |
|------|--------------|----------------|
| 金融 | 人民银行/金融监管总局 | `[待验证 — 须核实最新目录]` |
| 医疗健康 | 国家卫健委 | `[待验证 — 须核实最新目录]` |
| 工业制造 | 工信部 | `[待验证 — 须核实工业数据分类分级规范]` |
| 汽车 | 工信部/网信办 | `[待验证 — 须核实汽车数据安全相关规定]` |
| 电信 | 工信部 | `[待验证 — 须核实电信数据安全规定]` |
| 能源/交通 | 国家发改委/交通运输部 | `[待验证 — 须核实主管部门规定]` |
| 互联网/平台 | 网信办 | `[待验证 — 须核实网络数据安全管理相关规定]` |

**目录对照工作表**：

| 数据资产名称 | 是否在行业目录中 | 目录依据 | 结论 |
|---------|--------------|---------|------|
| [数据名称] | 是/否/目录未发布 | [目录名称/条目] | 重要数据/一般数据/待确认 |

#### 1.2 保守推定标准（目录缺失时）

当适用行业目录尚未公布或无法获取时，按以下保守标准推定是否为重要数据（`[行业监管 flagged — 目录待公布，按保守标准推定]`）：

| 保守推定情形 | 推定结论 |
|---------|---------|
| 数据规模达到一定量级且数据类型关系行业主要功能（如能源调度数据、金融交易数据） | 推定为重要数据 |
| 数据被竞争对手或外国机构获取后可能对产业安全造成影响 | 推定为重要数据 |
| 数据反映了国家治理或公共服务的运营状况（如交通运行、医疗资源分布） | 推定为重要数据 |
| 数据汇聚后可反推出原本分散的敏感信息（汇聚效应） | 推定为重要数据（须评估汇聚后影响） |
| 数据仅涉及企业内部运营，无法通过汇聚产生更高层次影响 | 推定为一般数据（需记录理由） |

### 步骤二：重要数据处理者义务清单

一旦识别数据处理者处理的数据属于重要数据，须履行以下强制性义务（**具体义务条款以《数据安全法》最新版及配套规定为准 `[待验证]`**）：

#### 2.1 明确数据安全负责人和管理机构（🔴 强制）

| 义务项 | 具体要求 | 合规状态 |
|-------|---------|---------|
| 数据安全负责人 | 须明确专门的数据安全负责人（**是否须为高级管理人员须核实 `[待验证]`**） | ☐ 已落实 / ☐ 待落实 |
| 数据安全管理机构 | 须设立或指定专门的数据安全管理机构（**规模要求 `[待验证]`**） | ☐ 已落实 / ☐ 待落实 |
| 职责界定 | 负责人和管理机构的职责须清晰界定，包含数据分类分级、安全评估、事件处置等 | ☐ 已落实 / ☐ 待落实 |

> **与 PIPL 的衔接**：若处理个人信息达到规定规模，还须指定个人信息保护负责人（PIPL 第52条 `[待验证]`）。两者职能可整合但须明确区分职责范围。

#### 2.2 定期开展风险评估并向主管部门报送风险评估报告（🔴 强制）

| 义务项 | 具体要求 | 合规状态 |
|-------|---------|---------|
| 风险评估开展频率 | 须**定期**开展风险评估（**具体频率/周期 `[待验证 — 须核实 DSL 配套规定]`**） | ☐ 已安排 / ☐ 待安排 |
| 风险评估报告内容 | 须覆盖：数据处理活动合规性、安全措施有效性、重要数据安全状态、风险识别与应对 | ☐ 模板已建立 / ☐ 待建立 |
| 向主管部门报送 | 风险评估报告须按规定向有关主管部门报送（**报送频率/对象/渠道 `[待验证 — 须核实主管部门要求]`**） | ☐ 报送计划已建立 / ☐ 待建立 |

> **风险评估详细方法**：见 `risk-assessment-reporting` 子技能。

#### 2.3 重要数据出境须经安全评估

重要数据处理者向境外提供重要数据，须按照 DSL 第31条（**已核实**）经安全评估后方可出境：

| 重要数据出境主体 | 出境路径要求 | 依据 |
|--------------|------------|------|
| **关键信息基础设施运营者（CIIO）**在境内运营中收集和产生的重要数据出境 | 参照《网络安全法》及关基条例，须向主管部门申报安全评估（**具体规定 `[待验证 — 须核实 CSL 关基规定及配套办法]`**） | DSL 第31条前段（已核实）；CSL 关基保护制度 |
| **其他数据处理者**（非 CIIO）重要数据出境 | 按国家网信部门会同有关部门制定的安全管理办法执行安全评估（**具体办法 `[待验证 — 须核实网信办数据出境安全评估办法是否已覆盖]`**） | DSL 第31条后段（已核实） |

**出境路径说明**：
- 重要数据出境**不适用**标准合同路径（路径③）和保护认证路径（路径②）
- 重要数据出境**一律须经安全评估**（路径①），须触发 `cross-border-assessment/security-assessment-filing`
- 出境前须完成跨境个人信息保护影响评估（PIPL 第55条第(四)项+第56条：向境外提供个人信息属须进行 PIA 的法定情形，已核实锚点）

### 步骤三：重要数据出境对照表

| 评估维度 | 评估要点 | 结论 |
|---------|---------|------|
| 出境数据是否包含重要数据 | 对照步骤一识别结果 | 是/否 |
| 处理者是否为 CIIO | CLAUDE.md 配置 + 行业认定 | 是/否/不确定 |
| 出境业务需求合法正当必要性 | 是否有实质业务需求；能否通过境内处理替代出境 | — |
| 数据最小化 | 出境数据范围是否可进一步缩小（如去除重要数据部分） | — |
| 安全评估是否已申报 | 见 `security-assessment-filing` | 是/否/进行中 |

**出境结论**：
- 重要数据确需出境 → 🔴 强制安全评估；触发 `cross-border-assessment/security-assessment-filing`
- 重要数据不建议出境 → 采用境内处理方案（数据不出境）；记录决策理由
- 重要数据出境范围不明确 → 先缩小数据范围，去除不确定部分，再评估

### 步骤四：重要数据台账建立

重要数据须建立专项台账（独立于一般数据管理），作为重要数据处理者义务的管理基础：

| 台账项 | 内容 | 更新频率 |
|-------|------|---------|
| 重要数据名称及类型 | 名称、格式、数量级 | 变化时更新 |
| 重要数据来源 | 内部产生/外部获取 | 变化时更新 |
| 识别依据 | 行业目录/保守推定 | 年度复核 |
| 存储位置 | 境内/境外（🔴 重要数据须境内存储，出境须评估） | 变化时更新 |
| 访问权限 | 可访问人员/系统，权限分级 | 每季度复核 |
| 出境情况 | 是否出境/安全评估状态 | 变化时更新 |
| 风险评估状态 | 上次评估日期/下次评估计划 | 每次评估后更新 |
| 负责人 | 数据安全负责人 | 变化时更新 |

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 行业重要数据目录尚未公布 | 按保守标准推定；标 `[行业监管 flagged — 目录待公布]`；建议向主管部门主动咨询 |
| 数据处理者同时处理多个行业的重要数据 | 各行业目录均须对照；最高等级要求适用于全部重要数据管理 |
| 数据处理者不确定是否为重要数据处理者 | 按保守推定，按重要数据处理者义务建立合规体系；待主管部门确认后调整 |
| CIIO 身份确认中，尚未收到认定通知 | 若业务性质接近关基认定范围，建议主动向行业主管部门咨询；同时按 CIIO 义务建立合规框架 |
| 重要数据出境有紧迫业务需求 | 安全评估不可绕过（🔴）；可探索能否通过境内数据处理+仅提供处理结果的方式替代数据出境 |
| 重要数据范围超出预期，影响业务开展 | 评估数据最小化方案（剔除重要数据后出境）；整体评估业务模式调整可行性 |

## 错误处理

| 错误类型 | 触发情形 | 处理方式 |
|---------|---------|---------|
| 重要数据识别范围争议 | 业务团队与合规团队对某类数据是否属于重要数据存在分歧 | 记录分歧；按较高等级处理；建议寻求主管部门或外部专家意见 |
| 风险评估报送主管部门不明确 | 跨行业数据，主管部门多头监管 | 识别主要监管主体（按行业/数据类型），同时向多个主管部门咨询；标 `[行业监管 flagged — 多头监管需核实主报部门]` |
| 重要数据已出境但未经安全评估 | 已发现合规缺口 | 立即暂停该数据出境活动（🔴）；评估历史出境的影响；启动安全评估申报；必要时主动向监管部门报告 |
| 数据安全负责人尚未设立 | 重要数据处理者义务尚未落实 | 标为 🟠/🔴 整改项；建议在 [时限] 内完成；提供职责清单模板 |
