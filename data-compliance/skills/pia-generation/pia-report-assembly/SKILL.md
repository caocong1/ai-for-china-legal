# PIA 报告组装（pia-report-assembly）

> **前置**：
> 1. 读取 `data-compliance/CLAUDE.md`（实践配置文件），加载：文书风格（`## 文书风格`）、PIA 报告风格与存放位置、工作产物标题（`## 输出`）、升级矩阵（`## 升级矩阵` → 监管报告触发）。
> 2. 读取 `data-compliance/skills/_shared/legal-basis-conventions.md`（数据合规引用规范）。
> 3. 读取 `data-compliance/skills/_shared/data-protection-citations.md`（数据保护引用库，重点：PIPL第55-56条PIA情形与记录存3年）。
> 4. 读取前序三个子技能产出：trigger-consent-basis（PIA触发+合法性基础+同意机制评估）、data-mapping-necessity（数据映射+最小必要+委托处理评估）、risk-mitigation-assessment（风险矩阵+保护措施+个人权利响应评估）。若单独触发，须询问三个子技能评估结论或以简化方式收集关键发现。
> 5. **分工说明**：本子技能负责「汇总并组装完整 PIA 报告」、「确认报告保存义务（第55-56条）」、「结论与整改建议优先级排序」、「与监管检查/合规审计衔接」、「交付前质量检查清单」。前三层评估不在本子技能重复，仅汇总引用。
>
> **本子技能可被 pia-generation 编排入口调用，也可由律师/DPO 通过 `/data-compliance:pia-report-assembly` 单独触发（单独触发时须提供前三层评估输出，或以简化模式工作）。**

---

## 目的

本子技能覆盖 PIA 生成流程的第四层（最终环节）：

1. **汇总三层评估发现**：整合 trigger-consent-basis/data-mapping-necessity/risk-mitigation-assessment 的核心发现
2. **组装完整 PIA 报告**：按 PIPL 第55条评估内容要求，生成结构完整的个人信息保护影响评估报告
3. **确认保存义务**：依第56条，确保报告与处理情况记录至少保存3年
4. **结论与整改建议**：给出可操作的优先级整改清单，高风险须整改后方可处理
5. **与监管检查/合规审计衔接**：说明 PIA 报告在监管检查和合规审计中的使用方式

### 步骤 4：报告保存义务确认（PIPL第55-56条，已核实锚点）

**保存义务合规提示**：

> **法律依据（已核实）**：《个人信息保护法》（2021年）**第56条**规定，个人信息保护影响评估报告和处理情况记录**至少保存三年**。

| 保存项目 | 保存期限 | 存放方式 | 合规提示 |
|---------|---------|---------|---------|
| 个人信息保护影响评估报告（本报告）| 至少3年（第56条，已核实）| [文件管理系统/纸质/加密存储] | 须控制传阅；须纳入合规档案管理 |
| 处理情况记录（处理活动的日常记录）| 至少3年（第56条，已核实）| [数据库/日志系统] | 须可审计；须可供监管检查 |
| 整改措施跟踪记录 | 建议与本报告同期保存 | [整改跟踪表] | 记录整改完成情况 |

**版本管理建议**：
- 报告须注明评估日期、版本号、评估人员
- 每次重新评估须保留历史版本（不覆盖）
- 建议在 git 版本控制中管理报告版本（`data-compliance/CLAUDE.md` 同仓库）

### 步骤 5：与监管检查/合规审计的衔接

**监管检查准备**（如网信办或行业主管部门检查）：

| 监管检查环节 | 本 PIA 报告的作用 | 准备要点 |
|-----------|---------------|---------|
| 提交 PIA 报告 | 第56条要求保存报告，监管检查时须能提供 | 报告须完整、结构清晰、整改措施有记录 |
| 说明合规状态 | PIA 报告是合规证明文件之一 | 报告结论须与实际处理活动一致 |
| 整改记录 | 发现的风险点须有整改记录 | 附件3整改跟踪表须有实际整改记录 |

**合规审计衔接**（PIPL第54条 `[待验证]` 定期合规审计）：

- 本 PIA 报告作为合规审计的基础材料
- 审计时需核查：报告是否按计划复核、整改是否完成、保存期限是否满足
- 委托第三方审计机构时，须注意报告的保密等级处理

## 边界条件

| 场景 | 处理方式 |
|------|---------|
| 评估发现须重新设计处理活动 | 报告结论写「建议暂停处理，须重新设计」；提示回到 trigger-consent-basis 和 data-mapping-necessity 重评；不强行出具「可继续处理」结论 |
| 涉及跨境的 PIA 报告 | 在第三部分（数据映射）中说明出境事实，在第四部分（风险评估）中标注「跨境风险须 cross-border-assessment 深度评估」；跨境路径结论不在本报告中呈现，留待 cross-border-assessment 专项报告 |
| 报告须提交给监管机构 | 提醒：对外提交版本须经律师审核；须区分「工作产物（特权保护）」与「对外提交版本」；对外版本须注意措辞（避免过度承认违规）|
| 前三个子技能产出不完整（单独触发 pia-report-assembly）| 告知：「PIA 报告组装需要完整的三层评估结论。请提供：(1) PIA 触发情形结论；(2) 处理活动数据映射；(3) 风险矩阵。或运行 /data-compliance:pia-generation 执行完整流程」|
| 配置文件缺失 | 临时模式：工作产物使用律师标题；报告风格使用正式格式；存放位置标为待填写；输出标 `[临时模式]` |

## 错误处理

| 错误场景 | 处理方式 |
|---------|---------|
| 整改建议后续未落实（再次触发 PIA 时发现历史整改未完成）| 在新 PIA 报告中注明：「本次评估发现 [年份] PIA 报告中整改建议项 [N] 尚未完成（参见附件历史 PIA 文件编号）」；历史未整改项自动升一级风险等级 |
| 报告须同时对内（含风险分析）和对外（监管提交）用途 | 输出两个版本：内部版（含完整风险分析）和监管提交版（措辞审慎，建议律师单独审核监管版本）；两版均须注明版本区别 |
| 处理活动发生重大变更（需要更新 PIA）| 提醒：「重大变更须重新开展 PIA（不能仅更新本报告）；建议重新运行 /data-compliance:pia-generation 完整流程，产出新版 PIA 报告并保留历史版本」|
