# 评估情形与告知-同意基础（trigger-consent-basis）

> **前置**：
> 1. 读取 `data-compliance/CLAUDE.md`（实践配置文件），加载：数据处理者画像（`## 我们是谁`）、是否为 CIIO、是否处理敏感个人信息（含类型）、是否有数据出境场景、敏感数据政策（`## 合规立场` → `### 敏感数据政策`）、合规偏好（整体）、风险等级定义（`## 风险校准`）。
> 2. 读取 `data-compliance/skills/_shared/legal-basis-conventions.md`（数据合规引用规范）。
> 3. 读取 `data-compliance/skills/_shared/data-protection-citations.md`（数据保护引用库，重点：PIPL第13-14条/第28-29条/第24条/第55条）。
> 4. 读取 `data-compliance/skills/_shared/practice-profile-schema.md`（配置读取契约）。
> 5. **分工说明**：本子技能负责「何时须做 PIA」（第55条触发情形判定）与「处理合法性基础」（第13-14条）及「敏感个人信息单独同意」（第28-29条）及「自动化决策」（第24条）。数据映射与最小必要性见 `data-mapping-necessity`；风险评估与保护措施见 `risk-mitigation-assessment`；PIA 报告组装见 `pia-report-assembly`。
>
> **本子技能可被 pia-generation 编排入口调用，也可由律师/DPO 通过 `/data-compliance:trigger-consent-basis` 单独触发。**

---

## 目的

本子技能覆盖 PIA 生成流程的第一层评估：

1. **PIA 触发情形判定**：依《个人信息保护法》第55条，识别当前处理活动是否须开展个人信息保护影响评估
2. **处理合法性基础审查**：依第13-14条，确认每类处理活动是否有合法性依据，同意机制是否合规
3. **敏感个人信息单独同意评估**：依第28-29条，识别敏感类别并审查单独同意机制
4. **自动化决策合规分析**：依第24条，评估自动化决策透明度与个人权利保障

---

## 详细步骤

### 步骤 1：加载合规立场

从 `data-compliance/CLAUDE.md` 读取：
- `## 我们是谁` → 数据处理者角色（处理者/受托处理者/兼有）、是否处理敏感个人信息（及类型）
- `## 合规立场` → `### 敏感数据政策` → 内部门槛、单独同意获取方式、是否处理未成年人信息
- 如配置缺失/含占位符 → 临时模式：合规偏好「平衡型」；敏感数据按严格门槛标记；单独同意未配置时提示须审查

**临时模式提示**：

> 我注意到实践配置文件尚未填写。我将以临时模式运行——平衡型合规偏好、律师角色，对所有可能的敏感类别按严格门槛处理，输出标记 `[临时模式]`。运行 `/data-compliance:cold-start-interview` 可配置完整档案。

### 步骤 2：PIA 触发情形判定（PIPL第55条，已核实锚点）

依《个人信息保护法》第55条，逐一评估以下触发情形：

| 触发情形 | 判断问句 | 认定说明 | 是否触发 |
|---------|---------|---------|---------|
| 处理敏感个人信息 | 处理的信息是否属于第28条定义的敏感个人信息（生物识别/宗教信仰/特定身份/医疗健康/金融账户/行踪轨迹及未成年人信息）？ | 只要处理任一类敏感信息即触发 | [是/否] |
| 自动化决策 | 是否利用个人信息进行自动化决策（用户画像/推荐算法/信用评分/招聘筛选/自动审批等）？ | 决策系统有自动化判断逻辑即触发 | [是/否] |
| 委托处理/对外提供/公开 | 是否委托他人处理个人信息？是否向第三方提供个人信息？是否公开个人信息？ | 三情形之一满足即触发 | [是/否] |
| 向境外提供个人信息 | 是否将个人信息提供给境外的接收方（含境外服务器存储、境外公司访问）？ | 数据实际流转到境外即触发 | [是/否] |
| 其他重大影响情形 | 处理活动是否对个人权益有其他重大影响（如大规模监控/定向广告/个人画像用于重大决策）？ | 须结合实际场景判断 | [是/否] |

**PIA 触发结论表**：

| 情形 | 是否触发 | 触发依据 | 备注 |
|------|---------|---------|------|
| 处理敏感个人信息 | [是/否] | PIPL第55条第(一)项（已核实） | 敏感信息类型：[填入] |
| 自动化决策 | [是/否] | PIPL第55条第(二)项（已核实） | 决策系统描述：[填入] |
| 委托/提供/公开 | [是/否] | PIPL第55条第(三)项（已核实） | 情形描述：[填入] |
| 向境外提供 | [是/否] | PIPL第55条第(四)项（已核实） | 境外接收方：[填入] |
| 其他重大影响 | [是/否] | PIPL第55条第(五)项（已核实） | 影响描述：[填入] |

**结论**：满足一项即须开展 PIA → 进入步骤 3。若一项未满足 → 无须开展正式 PIA，但建议记录评估过程。

### 步骤 3：处理合法性基础审查（PIPL第13-14条，已核实锚点）

**合法性基础清单**（PIPL第13条，已核实）：

| 合法性基础 | 描述 | 适用场景 | 告知要求 | 需另取同意？ |
|---------|------|---------|---------|-----------|
| 取得个人同意（第13条第(一)项）| 取得个人自愿、明确、知情的同意 | 营销推广、非必需数据处理 | 须 | 是（同意本身即告知） |
| 订立或履行合同所必需（第13条第(二)项）| 个人为一方当事人，为履行合同须处理 | 配送地址、支付信息 | 须 | 否（但仍须告知） |
| 法定职责或义务所必需（第13条第(三)项）| 依法律/行政法规须处理 | 税务申报、实名制义务 | 须 | 否 |
| 突发公共卫生事件（第13条第(四)项）| 应对突发公共卫生事件 | 传染病防控（特殊适用） | 须 | 否 |

## 边界条件

| 场景 | 处理方式 |
|------|---------|
| 合同所必需基础被宽泛使用（如将营销也纳入「合同所必需」） | 标 🟠 高风险 — 「合同所必需」须与合同目的直接相关，不可拓展至非核心处理活动；营销须另取同意 |
| 「已公开信息」基础被宽泛适用（如将公开的姓名用于画像） | 标 🟠 — 第六项「依法公开」须为合理使用，不可超出公开目的范围；大规模画像须结合其他基础 |
| 同意与合法性基础竞合（如合同场景再取同意） | 说明可叠加使用但须告知所有合法性基础；不建议以「合法基础竞合」为由混淆告知义务 |
| 受托处理者（非数据处理者本身） | 本分析针对委托方（处理者）义务；受托处理者不独立承担告知-同意义务，但须在受托协议范围内处理 |
| 数据出境触发 | 第55条第(四)项触发 PIA 的跨境评估维度，但跨境路径判定（三路径）须交由 cross-border-assessment 深度评估 |
| 配置文件缺失/含占位符 | 临时模式：按严格门槛评估敏感类；同意机制按「无任何配置」状态标记全部审查项；输出标 `[临时模式]` |

| 错误场景 | 处理方式 |
|---------|---------|
| 处理活动描述不完整（仅系统名称，无数据类型说明） | 标记：「须补充以下信息才能评估触发情形：[数据类型/处理目的/接收方]」，不猜测 |
| 用户声称不处理敏感个人信息，但系统包含人脸识别功能 | 标记矛盾：「描述中提到人脸识别，属于PIPL第28条生物识别信息，须按敏感个人信息处理」 |
| 告知文件（隐私政策）未提供 | 以「未见告知文件」标记告知事项全项为 🟠 高风险，建议提供后重新评估 |
| 多系统共享个人信息但同意仅在入口取得 | 分析「入口同意是否覆盖后续系统使用」，通常须补充说明或重新取得同意 |
