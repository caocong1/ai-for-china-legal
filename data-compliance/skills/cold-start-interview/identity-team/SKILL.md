# 身份与团队访谈（identity-team）

> **所属编排**：`data-compliance/skills/cold-start-interview/SKILL.md`（冷启动访谈）
> **遵循契约**：`data-compliance/skills/_shared/practice-profile-schema.md` 字段映射（§2.1 数据处理者画像字段组）
> **写入目标**：`data-compliance/CLAUDE.md` — `## 我们是谁` / `## 谁在使用` / `## 可用集成`
> **分工说明**：本子技能收集数据处理者画像、执业角色与集成情况。合规立场（敏感数据/跨境/数据安全）由 `compliance-stance` 收集；风险与升级规则由 `risk-escalation` 收集；文书风格与行业监管偏好由 `style-cross-border-policy` 收集。
> **可单独触发**：`/data-compliance:identity-team`（只更新身份/团队部分，不影响其他章节）

---

## 目的

通过结构化访谈收集数据处理者/律所/DPO 团队的身份与画像信息，将答案写入 `data-compliance/CLAUDE.md` 对应章节的 `[PLACEHOLDER]` 位置，使所有数据合规技能能够：

1. 正确识别数据处理者角色（个人信息处理者/受托处理者/CIIO）；
2. 根据行业触发行业主管部门合规要求识别（`[行业监管 flagged]`）；
3. 根据个人信息处理规模判断出境路径门槛；
4. 根据执业角色选择工作产物标题（律师版 vs 非律师版）；
5. 根据合规团队规模调整整改建议的可行性。

---

## 前置

1. **读取 `data-compliance/CLAUDE.md`** — 检查 `## 我们是谁`、`## 谁在使用`、`## 可用集成` 是否已有非 `[PLACEHOLDER]` 内容（若已有，提示：「这些字段已有配置，是否覆盖更新？」）
2. **读取 `data-compliance/skills/_shared/practice-profile-schema.md`** — 字段映射表（§2.1 数据处理者画像字段组），了解每个答案写入哪个章节的哪个字段
3. 本子技能产出是更新后的 `data-compliance/CLAUDE.md` 章节内容，**不生成独立文件、不另存 YAML**

## 访谈问题清单

### 第一部分：基本身份（写入 `## 我们是谁`）

**Q1. 贵方的组织/律所名称和实体类型是什么？**

> 写入：`## 我们是谁` → 组织/律所名称、实体类型

可选类型：有限责任公司 / 股份有限公司 / 国有企业 / 律所 / 其他

---

**Q2. 贵方所属行业是什么？**

> 写入：`## 我们是谁` → 所属行业
> 影响：行业主管部门识别（监管主体：互联网→网信办/工信部；金融→人行/金融监管总局；医疗→卫健委；教育→教育部）

可选范围：互联网/科技 / 金融（银行/保险/证券）/ 医疗健康 / 教育 / 交通 / 制造 / 能源 / 电信 / 其他（请说明）

*追问*：是否有跨行业业务？（如同时经营金融和科技）

---

**Q3. 贵方在数据处理活动中的角色是什么？**

> 写入：`## 我们是谁` → 数据处理者角色
> 影响：受托处理者的义务框架不同（须签订数据处理协议，按委托方指示处理）

可选：
- **个人信息处理者**：自主决定处理目的和处理方式
- **受托处理者**：接受其他处理者委托处理（不自主决定目的）
- **两者兼有**：对外作为处理者，对委托方作为受托处理者

---

**Q4. 贵方是否为关键信息基础设施运营者（CIIO）？**

> 写入：`## 我们是谁` → 是否为 CIIO
> 影响：CIIO 须额外履行：数据本地化（CSL 第37条 `[待验证]`）、出境须安全评估（DSL 第31条，已核实）、关基保护条例义务

可选：是 / 否 / 不确定（若不确定，说明所属行业后由系统提供初步判断）

*追问（若不确定）*：业务系统若宕机，是否会严重影响金融运行/公共服务/能源供应/电信通信等？如是，建议向行业主管部门确认关基认定情况。`[行业监管 flagged — 关基认定须行业主管部门确认]`

---

**Q5. 贵方处理个人信息的大致规模是多少？**

> 写入：`## 我们是谁` → 处理个人信息的大致规模
> 影响：出境路径门槛判断（门槛由网信办办法规定 `[时效性 flagged — 须核实最新版办法]`）

参考量级：1万条以下 / 1万-10万条 / 10万-100万条 / 100万条以上

---

**Q6. 贵方是否处理敏感个人信息？**

> 写入：`## 我们是谁` → 是否处理敏感个人信息（及类型）

可选：否 / 是，类型包括（可多选）：
- 生物识别（人脸/指纹/声纹等）
- 医疗健康信息
- 金融账户信息
- 行踪轨迹信息
- 宗教信仰/特定身份
- 不满十四周岁未成年人个人信息
- 其他（请说明）

---

**Q7. 贵方是否有数据出境场景？**

> 写入：`## 我们是谁` → 是否有数据出境场景
> 影响：cross-border-assessment 触发与出境路径分析

可选：是（描述场景） / 否 / 不确定（有境外系统或境外合作方）

---

**Q8. 贵方合规团队规模大约是多少？**

> 写入：`## 我们是谁` → 合规团队规模

参考量级：1-2人（仅兼职）/ 3-5人（小型专职团队）/ 5人以上（中大型团队）/ 外部顾问（无内部团队）

---

**Q9. 贵方个人信息保护负责人（DPO 类似职能）是谁？**

> 写入：`## 我们是谁` → 个人信息保护负责人
> 影响：升级矩阵中的联系人路由

可选：已设置（姓名+职位） / 尚未设置 / 正在评估是否须设置（PIPL 第52条 `[待验证]`）

---

**Q10. 贵方目前最大的数据合规痛点是什么？**

> 写入：`## 我们是谁` → 最大痛点
> 帮助定制输出重点

可选（多选）：出境路径不确定 / PIA 未系统开展 / 数据分类分级未完成 / 等保定级未落实 / 供应商/受托处理者管理 / 隐私政策不完善 / 其他（请说明）

---

### 第二部分：执业角色（写入 `## 谁在使用`）

**Q11. 您的执业角色是什么？**

> 写入：`## 谁在使用` → 角色
> 影响：工作产物标题选择（律师版 vs 非律师版）；非律师须暂停提示

可选：
- 数据合规律师
- 企业 DPO（个人信息保护负责人）
- 企业法务
- 非律师但有律师可咨询
- 非律师且无律师可咨询

*若非律师*：所有输出将作为研究和初步分析供律师审查，而非法律结论。在数据处理活动合规结论、出境路径决定或监管申报建议前，我会暂停并建议升级给律师确认。

---

**Q12. 如需升级，联系哪位律师/法务总监？**

> 写入：`## 谁在使用` → 律师联系人

---

### 第三部分：可用集成（写入 `## 可用集成`）

**Q13. 贵方使用哪些数据合规相关系统工具？**

> 写入：`## 可用集成`

各项可选：已连接 / 未连接 / 不适用：
- 数据资产/数据地图工具
- 合规管理平台（如 OneTrust 类工具）
- 隐私政策/同意管理平台（CMP）
- 企业微信/钉钉
- 文档存储位置（如：企业内网/共享目录）

## 法律事实合理性检查

当用户口述以下内容时，在写入配置前进行合理性检查：

| 口述内容 | 合理性检查 |
|---------|----------|
| 声称不是 CIIO 但属于金融/电信/能源等高风险行业 | 提示关基认定风险，建议向行业主管部门确认 `[行业监管 flagged]` |
| 声称处理个人信息规模很小但实际描述业务体量较大 | 询问是否包含用户注册数据、交易记录等，帮助合理估算规模 |
| 声称无数据出境但有境外合作方或境外系统 | 询问境外系统是否处理境内用户数据（API 调用/数据同步等），识别隐性出境场景 |
| 口述 PIPL DPO 数量阈值（如「处理超过X万条须设置DPO」）| 标注 `[待验证 — 须核实 PIPL 第52条 具体规定]` `[前提 flagged — 需验证]` |

## 写入说明（填充 data-compliance/CLAUDE.md）

访谈完成后，将用户答案填充到 `data-compliance/CLAUDE.md` 对应章节的 `[PLACEHOLDER]` 位置：

1. **不另存独立 YAML 文件** — 只填充 `data-compliance/CLAUDE.md`，这是唯一真相来源
2. **保留待验证标注** — 如用户提供的值涉及网信办办法阈值，写入时保留 `[待验证]`
3. **确认前展示预览** — 将填充结果以 markdown 代码块形式展示，确认后写入

写入格式示例（`## 我们是谁` 章节）：

```markdown
## 我们是谁

- **组织/律所名称：** [用户答案]
- **实体类型：** [有限责任公司 / 律所 / ...]
- **所属行业：** [用户答案] `[行业监管 flagged — 需核实主管部门规定]`（如行业为金融/医疗/能源/交通等）
- **数据处理者角色：** [个人信息处理者 / 受托处理者 / 两者兼有]
- **是否为关键信息基础设施运营者（CIIO）：** [是 / 否 / 不确定 — 建议向行业主管部门确认]
- **处理个人信息的大致规模：** [约X万条记录]（影响出境路径门槛判断，具体门槛须核实最新版办法 `[时效性 flagged]`）
- **是否处理敏感个人信息：** [是（类型：[清单]）/ 否]
- **是否有数据出境场景：** [是 / 否]
- **合规团队规模：** [用户答案]
- **个人信息保护负责人（DPO 类似职能）：** [姓名/职位 / 尚未设置]
- **最大痛点：** [用户答案]
```
