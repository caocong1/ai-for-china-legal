# 关键信息基础设施保护（cii-protection）

> **可单独触发**：本子技能可脱离 network-security-compliance 编排入口独立运行。
> 斜杠命令：`/data-compliance:cii-protection`
>
> **前置依赖**：
> - 读取 `data-compliance/CLAUDE.md` → 获取：是否为 CIIO、所属行业、数据本地化立场、重要数据出境立场、合规团队规模
> - 引用 `data-compliance/skills/_shared/data-protection-citations.md` → CSL 第31条起（已核实）、第37条（待验证）、关基条例相关条目
> - 引用 `data-compliance/skills/_shared/legal-basis-conventions.md` → 来源标签规范、行业监管识别标注（公安部/网信办/行业主管部门）
> - 引用 `data-compliance/skills/_shared/practice-profile-schema.md` → CIIO 字段读取契约、出境路径字段
>
> **配置缺失处理**：若 `data-compliance/CLAUDE.md` 中「是否为 CIIO」为 `[PLACEHOLDER]` 或「不确定」，启用临时模式：
> - 临时模式：按 CIIO 处理，从严推定，标注 `[临时模式 — 按 CIIO 从严推定]`
> - 建议向行业主管部门确认关基认定情况
>
> **分工说明**：本子技能专注关基识别、关基运营者特别义务、数据本地化与出境；等保基础框架见 `mlps-grading`（关基系统通常在等保基础上叠加关基保护义务）；网络运营者一般义务见 `operator-obligations`；出境评估申报见 `cross-border-assessment`（衔接）。

---

## 目的

依据《网络安全法》（2017年）**第31条起**（**已核实锚点，建议复核**）及《关键信息基础设施安全保护条例》（2021年9月1日施行，国务院），对关键信息基础设施运营者（CIIO）或可能被认定为关基的组织开展合规审查：评估关基识别风险、落实关基运营者特别安全保护义务、审查数据本地化与出境安全评估合规，形成关基保护合规建议。

关基保护是中国网络安全最高优先级领域，在等保制度基础上实施重点保护，适用最严格的安全保护要求。

---

## 详细步骤

### 步骤一：理解关键信息基础设施制度框架

依据《网络安全法》（2017年）**第31条起**（**已核实锚点，建议复核**），国家对以下行业和领域的关键信息基础设施（CII）实施重点保护：

| 关基重点行业与领域（CSL第31条，已核实）| 说明 |
|--------------------------------------|------|
| **公共通信和信息服务** | 基础电信运营商、互联网基础设施 |
| **能源** | 电力、石油、天然气等能源基础设施 |
| **交通** | 铁路、公路、水运、航空、城市交通 |
| **水利** | 水利工程信息化系统 |
| **金融** | 银行、证券、保险核心系统 |
| **公共服务** | 社保、医疗、教育、社区服务等公共服务信息系统 |
| **电子政务** | 政府信息系统与电子政务平台 |
| **其他** | 其他一旦遭到破坏或数据泄露可能严重危害国家安全/国计民生/公共利益的重要信息基础设施 |

**关基的核心标准（综合理解）**：网络设施或信息系统一旦遭到破坏、丧失功能或数据泄露，可能严重危害国家安全、国计民生或公共利益。

### 步骤二：关基识别与认定风险评估

**关基认定流程**（具体程序 `[待验证 — 须核实《关键信息基础设施安全保护条例》具体条款]`）：

| 阶段 | 主要工作 | 责任主体 | 注意事项 |
|-----|---------|---------|---------|
| **1. 保护工作部门认定** | 各行业主管部门（保护工作部门）负责认定本行业关键信息基础设施 `[行业监管 flagged]` | 行业主管部门 | 认定程序由各保护工作部门制定 `[待验证]` |
| **2. 通知运营者** | 保护工作部门完成认定后，通知运营者其被认定为关基运营者 `[待验证]` | 保护工作部门 | 运营者须配合认定工作 `[待验证]` |
| **3. 运营者配合** | 配合保护工作部门的认定工作，提供所需材料 `[待验证]` | 运营者 | 不配合认定属违法 `[待验证]` |
| **4. 通报网信、公安** | 保护工作部门将关基认定结果通报国家网信部门和公安部门 `[待验证]` | 保护工作部门 | 实现跨部门监管协同 |

**关基识别风险判断表（自查）：**

| 自查项目 | 高风险信号 | 处理建议 |
|---------|---------|---------|
| 是否属于 CSL 第31条列举的行业领域 | 属于金融/电信/能源/交通/医疗公共服务/电子政务等 | `[行业监管 flagged]` 建议主动向行业主管部门了解关基认定情况 |
| 系统中断后果的严重性 | 系统中断会影响大量用户（如数百万以上）或关键公共服务 | 较高关基认定风险 |
| 是否已收到关基认定通知 | 已收到行业主管部门的认定通知 | 立即按关基运营者要求履行义务 `[待验证]` |
| 是否被列入重要数据目录处理者 | 处理被行业主管部门列为重要数据或国家核心数据的数据 | 与关基认定高度相关 `[行业监管 flagged]` |
| 是否为关基供应链重要组成 | 是其他已认定关基系统的关键技术供应商 | 须关注关基保护条例中关于供应链安全的规定 `[待验证]` |

### 步骤三：关基运营者特别安全保护义务

依据《关键信息基础设施安全保护条例》（2021年，国务院），关基运营者须履行以下特别义务（具体条号 `[待验证 — 须核实条例具体条款]`）：

| 义务类别 | 主要内容 | 验证状态 |
|---------|---------|---------|
| **设置专门安全管理机构** | 设立专门负责关基安全保护工作的机构，配备专职安全管理人员 `[待验证]` | `[待验证]` |
| **安全负责人** | 设置网络安全负责人（类似 CSO/CISO 职能），明确岗位职责 `[待验证]` | `[待验证]` |
| **人员审查** | 对重要岗位人员进行安全背景审查（`[待验证 — 具体审查要求]`） | `[待验证]` |
| **定期安全检测与评估** | 每年至少开展一次安全检测与风险评估（`[待验证 — 具体频率]`），邀请保护工作部门或其指定机构 | `[待验证]` |
| **应急预案与演练** | 制定关基安全保护应急预案，定期开展应急演练（`[待验证 — 演练频率]`） | `[待验证]` |
| **事件报告** | 发生重大网络安全事件时，及时向保护工作部门和公安机关报告（`[待验证 — 报告时限]`） | `[待验证]` |
| **网络产品与服务采购安全审查** | 采购可能影响国家安全的网络产品和服务，须申报网络安全审查（`[待验证 — 具体适用标准]`） | `[待验证]` |
| **等保加码** | 在等保第三级及以上基础上，按关基保护要求实施更高安全措施 `[待验证]` | `[待验证]` |
| **与保护工作部门协作** | 配合保护工作部门组织的安全检查、指导、监督 `[待验证]` | `[待验证]` |

**义务对比：关基运营者 vs. 一般网络运营者：**

| 义务维度 | 一般网络运营者（等保基础）| 关基运营者（等保+关基）| 风险等级 |
|---------|----------------------|---------------------|---------|
| 等级保护 | 按定级履行等保义务 | 须≥三级等保 `[待验证]` | 🟠 |
| 专门安全机构 | 无强制要求 | 须设立专门安全管理机构 `[待验证]` | 🔴 |
| 定期安全检测评估 | 等保测评（认可机构）| 须每年额外开展安全检测评估 `[待验证]` | 🔴 |
| 应急演练 | 一般应急预案 | 须定期演练 `[待验证]` | 🟠 |
| 采购安全审查 | 无强制申报要求 | 特定采购须申报网络安全审查 `[待验证]` | 🔴 |
| 数据本地化 | 无强制本地化 | 须本地存储（CSL 第37条 `[待验证]`） | 🔴 |
| 出境限制 | 满足 PIPL 跨境路径之一即可 | 出境须安全评估（DSL 第31条，已核实） | 🔴 |

### 步骤四：数据本地化义务审查

**数据本地化义务**（CSL 第37条 `[待验证具体条号]`；结合 DSL 第31条（已核实））：

依据目前理解（具体条号须验证），关键信息基础设施运营者须在中国境内存储其在境内运营中收集和产生的个人信息和重要数据；确需向境外提供的，须经安全评估：

| 数据类型 | 本地化要求 | 出境路径 | 验证状态 |
|---------|---------|---------|---------|
| **个人信息（CIIO 收集）** | 须在境内存储 `[待验证 — CSL第37条]` | 确需出境须经安全评估 `[待验证]` | `[待验证]` |
| **重要数据（CIIO 产生或处理）** | 须在境内存储 `[待验证 — CSL第37条]` | 确需出境须经安全评估（DSL第31条，已核实） | 已核实（DSL第31条） |
| **一般业务数据** | 等保制度下的数据安全措施，无强制本地化 | PIPL 三路径之一（非重要数据、非 CIIO 的个人信息） | 须判断具体数据类型 |

**本地化合规审查要点：**

- [ ] 已识别 CIIO 在境内运营中收集或产生的个人信息的完整范围
- [ ] 已识别 CIIO 处理的重要数据范围（参照行业目录 `[行业监管 flagged]`）
- [ ] 个人信息和重要数据的主要存储位置：确认在境内（包括云存储） `[待验证]`
- [ ] 如有境外系统备份/同步，是否满足本地化与出境安全评估要求 `[待验证]`
- [ ] 境外第三方处理/SaaS 服务是否涉及 CIIO 数据出境，是否已评估 `[待验证]`

### 步骤五：出境安全评估合规（衔接 cross-border-assessment）

CIIO 向境外提供数据须经安全评估（DSL 第31条，**已核实**）：

| 出境情形 | 合规路径 | 衔接技能 |
|---------|---------|---------|
| CIIO 向境外提供重要数据 | 🔴 必须经安全评估（DSL第31条，已核实）| `/data-compliance:security-assessment-filing` |
| CIIO 向境外提供大规模个人信息 | 🔴 须经安全评估（安全评估办法适用门槛 `[待验证]`）| `/data-compliance:pathway-determination` + `/data-compliance:security-assessment-filing` |
| CIIO 向境外提供一般规模个人信息 | 按 PIPL 第38条三路径判断（但 CIIO 不适用标准合同的具体规则 `[待验证]`）| `/data-compliance:pathway-determination` |

> **关基出境合规的特别注意**：CIIO 不得以「合同所必需」「标准合同」等一般路径替代强制安全评估要求（具体规则 `[待验证]`）。所有 CIIO 出境场景均须首先评估是否触发安全评估申报义务。

## 检查清单

**关基保护合规关键检查项：**

- [ ] 已自查业务是否属于 CSL 第31条列举的关键领域，评估关基认定风险
- [ ] 如已被认定为 CIIO，已收到并归档行业主管部门的认定通知
- [ ] 已按关基保护要求设立专门安全管理机构和安全负责人 `[待验证]`
- [ ] 关键岗位人员已完成安全背景审查 `[待验证]`
- [ ] 每年安全检测与风险评估计划已制定并按期执行 `[待验证]`
- [ ] 应急预案已制定，并按计划开展演练 `[待验证]`
- [ ] 境内运营数据（个人信息 + 重要数据）的本地化存储已落实 `[待验证 — CSL第37条]`
- [ ] 有境外出境场景的，已识别是否属于须安全评估的范围（DSL第31条，已核实）
- [ ] 关基供应链核心产品和服务采购已评估网络安全审查申报要求 `[待验证]`
- [ ] 已建立与保护工作部门和公安机关的事件报告机制 `[待验证]`

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 未被认定为 CIIO，但属于 CSL 第31条列举行业 | 仍须评估关基认定风险，标 `[行业监管 flagged]`；在正式被认定前，按等保基础义务履行 |
| 关基认定不确定（组织不清楚） | 临时模式按 CIIO 从严推定；建议向行业主管部门或专业律师确认 |
| CIIO 系统同时承载一般业务数据 | 按最严格标准（关基保护）统一处理；分类存储可降低部分合规风险 |
| 外资企业在华关基运营 | 须遵守中国关基保护规定；外资身份不影响关基义务 `[待验证]` |
| 关基与非关基系统共用基础设施 | 须评估网络隔离与数据隔离措施，防止关基数据泄露 `[待验证]` |
| 供应链安全（关基供应商） | 若为已认定关基运营者的关键供应商，须评估网络安全审查申报需求 `[待验证]` |

## 错误处理

| 错误情形 | 处理动作 |
|---------|---------|
| 用户不确定是否为 CIIO | 按关基自查问题清单逐一评估，高风险信号出现时升级处理，标 `[行业监管 flagged]` |
| 用户口述关基认定细节（如认定条件、义务清单） | 合理性检查，核心义务框架正确，具体条款标 `[待验证 — 须核实关基条例具体条款]` `[前提 flagged — 需验证]` |
| 发现有境外数据备份但未评估出境 | 🔴 关键风险 — 提示须触发 `/data-compliance:security-assessment-filing` 开展安全评估 |
