# 网络安全等级保护（mlps-grading）

> **可单独触发**：本子技能可脱离 network-security-compliance 编排入口独立运行。

> - 引用 `data-compliance/skills/_shared/legal-basis-conventions.md` → 来源标签规范、公安部/网信办行业监管定位

> - 临时模式：按保守定级策略，遇不确定等级时建议按上一级定级（宁高勿低）

> **分工说明**：本子技能负责等级保护的定级、备案、测评与整改；关键信息基础设施特别保护见 `cii-protection`；网络运营者一般义务（日志留存/实名制/应急预案）见 `operator-obligations`；个人信息收集合规见 `personal-info-collection`。等保制度是关基保护的基础层，CIIO 在等保之上还须满足 `cii-protection` 的额外义务。

## 目的

依据《网络安全法》（2017年）**第21条**（**已核实锚点，建议复核**）规定的网络安全等级保护制度，结合等保 2.0（国家标准 GB/T 22239 `[待验证年份与版本]`），对网络运营者（企业/机构运营的网络和信息系统）开展等级保护定级、备案、等级测评与安全建设整改的全流程合规审查，评估系统是否达到对应等保级别的安全保护要求，并形成整改建议与合规清单。

等保制度是中国网络安全基础合规框架，适用于所有等级的网络运营者；关键信息基础设施运营者（CIIO）在等保之上叠加关基保护特别义务（见 `cii-protection`）。

## 详细步骤

### 步骤一：理解等保五级制度框架

依据《网络安全法》（2017年）**第21条**（**已核实锚点，建议复核**），网络运营者须按等级保护制度要求履行安全保护义务。等保 2.0 将系统划分为五个安全保护等级（具体定级标准和 GB/T 国家标准编号 `[待验证]`）：

| 等级 | 受侵害客体 | 危害程度 | 典型系统举例 | 监管要求 |
|------|---------|---------|------------|---------|
| **第一级** | 公民、法人和其他组织的合法权益 | 一般损害 | 小型内部办公系统、个人网站 | 自主保护，无需备案与测评 |
| **第二级** | 公民、法人和其他组织的合法权益 | 严重损害；或对社会秩序和公共利益造成危害 | 中型企业内部信息系统、小型电商平台 | 应向公安机关备案（`[待验证 — 具体备案规定条号]`）；建议开展等级测评 |
| **第三级** | 社会秩序和公共利益 | 严重损害；或对国家安全造成危害 | 大型互联网平台、金融机构核心系统、政府信息系统 | **须**向公安机关备案（`[待验证]`）；**须**每年开展等级测评（`[待验证具体频率]`） |
| **第四级** | 社会秩序和公共利益 | 特别严重损害；或对国家安全造成严重危害 | 金融核心基础设施、重要政府信息系统 | **须**备案；**须**每半年开展等级测评（`[待验证具体频率]`）；须主管部门审核 |
| **第五级** | 国家安全 | 特别严重损害 | 国家安全级别系统 | 最严格管控，国家指定专门部门负责（绝大多数企业不涉及） |

> **定级基本原则**（`[待验证 — 具体定级标准条款]`）：定级须综合考虑：（1）受侵害的客体（公民权益/社会秩序/公共利益/国家安全）；（2）侵害发生后的危害程度。两个维度交叉确定等级。

### 步骤二：系统信息采集与初步定级

**采集以下系统信息以判断等级：**

| 采集项 | 说明 | 判断影响 |
|-------|------|---------|
| 系统名称与功能描述 | 系统做什么、处理什么业务 | 业务性质影响受侵害客体判断 |
| 系统用户规模 | 注册用户数、活跃用户数 | 规模影响危害程度判断（大规模更易影响社会秩序） |
| 处理数据类型 | 个人信息/重要数据/敏感数据/一般数据 | 数据敏感度影响等级判断 |
| 系统行业归属 | 金融/医疗/能源/交通/互联网等 | 行业关键性影响受侵害客体等级 |
| 是否已完成等保定级 | 历史定级情况 | 判断是否需要重新定级或复核 |
| 是否为 CIIO 或承载关基 | 关基运营者的系统定级通常≥三级 | 触发 cii-protection 子技能 |
| 境外访问/跨境数据流 | 是否有境外用户访问或数据出境 | 影响安全边界与出境合规 |

**初步定级判断表：**

| 系统类型 | 建议最低等级 | 说明 |
|---------|------------|------|
| 金融机构核心业务系统 | 第三级起 `[待验证]` | 银行/证券/保险核心系统；影响社会秩序 |
| 大型互联网平台（超过一定用户规模） | 第三级起 `[待验证]` | 大规模用户数据；平台服务中断影响广 |
| 政府/公共服务信息系统 | 第三级起 `[待验证]` | 公共服务中断影响社会秩序 |
| 医疗卫生信息系统 | 第三级起 `[待验证]` | 医疗健康数据；系统中断影响医疗服务 |
| 能源/交通/水利等基础设施控制系统 | 第三级以上 `[待验证]` | 关基系统，危害公共利益与国家安全 |
| 中型企业内部系统 | 第二级起 `[待验证]` | 员工/客户数据；对公民权益有影响 |
| 小型内部管理系统 | 第一级 `[待验证]` | 仅影响内部运营，危害程度有限 |

> **注意**：`[待验证 — 具体行业定级标准与主管部门监管要求须结合行业规定核实]` `[行业监管 flagged]`

### 步骤三：等保定级申请与备案

第二级及以上系统须完成备案（具体要求 `[待验证 — 须核实公安部等保备案具体规定]`）：

**等保定级流程：**

| 阶段 | 主要工作 | 责任主体 | 注意事项 |
|-----|---------|---------|---------|
| **1. 定级申报** | 按等保定级标准（`[待验证]`）确定系统等级，填写定级报告 | 网络运营者 | 建议邀请专业测评机构辅助定级 |
| **2. 专家评审（第三级及以上）** | 组织专家对定级结果进行评审（`[待验证是否强制]`） | 网络运营者 | 争取主管部门/行业协会专家参与 |
| **3. 公安机关备案** | 向所在地公安机关（或上级公安机关）提交备案材料（`[待验证具体备案部门与材料清单]`） | 网络运营者 | 备案完成后取得「备案证明」 |
| **4. 备案审核** | 公安机关对备案材料进行审核 | 公安机关 | 第三级及以上须重点审核 `[待验证]` |

**备案所需材料（参考，具体清单 `[待验证 — 须核实公安部规定]`）：**
- 网络安全等级保护定级报告
- 网络安全技术措施说明（安全建设方案摘要）
- 主要负责人签字的责任声明（`[待验证]`）

### 步骤四：安全建设与整改（符合等保 2.0 要求）

根据定级结果，按等保 2.0 安全要求（具体要求类别 `[待验证 — GB/T 22239 等保2.0国标编号待验证]`）开展安全建设与整改：

**等保 2.0 安全要求维度（参考框架，具体标准条款 `[待验证]`）：**

| 要求类别 | 主要内容 | 适用等级（参考，须验证） |
|---------|---------|---------------------|
| **安全物理环境** | 机房选址、访问控制、防盗防破坏、防火防水、电力保障 | 所有等级（第一级基本要求起） `[待验证]` |
| **安全通信网络** | 网络架构、通信传输加密、可信验证 | 第二级起 `[待验证]` |
| **安全区域边界** | 边界防护（防火墙/IDS）、访问控制、入侵检测、恶意代码防范 | 第二级起 `[待验证]` |
| **安全计算环境** | 身份鉴别（双因子认证等）、访问控制、安全审计、入侵防范、恶意代码防范、数据完整性与保密性 | 第二级起 `[待验证]` |
| **安全管理中心** | 系统管理、安全管理、审计管理 | 第三级起 `[待验证]` |
| **安全管理制度** | 策略/制度/规程、文档管理 | 所有等级 `[待验证]` |
| **安全管理机构** | 岗位设置、人员配备、授权审批、沟通与合作、监督检查 | 第二级起 `[待验证]` |
| **安全管理人员** | 人员录用、离岗、考核、培训教育、外部人员管理 | 第二级起 `[待验证]` |
| **安全建设管理** | 安全方案设计、产品采购使用、软件开发、工程实施、测试验收 | 第三级起须更严格 `[待验证]` |
| **安全运维管理** | 环境/资产/介质/设备/网络/系统/漏洞/密码/变更/备份恢复/应急 | 所有等级（层次不同） `[待验证]` |

**等保 2.0 扩展要求（特殊场景，具体标准 `[待验证]`）：**

| 扩展类型 | 适用场景 | 关注重点 |
|---------|---------|---------|
| 云计算安全扩展要求 | 系统部署在云环境（公有云/私有云/混合云） | 租户隔离、虚拟化安全、云服务提供商等保合规 |
| 移动互联安全扩展要求 | 系统支持移动端接入 | 移动终端管理、无线接入安全 |
| 物联网安全扩展要求 | 系统包含物联网设备 | 感知层安全、传输安全 |
| 工业控制系统安全扩展要求 | OT/ICS 工控系统 | 控制设备安全、工控协议安全 |

### 步骤五：等级测评

第三级及以上系统须定期开展等级测评（具体频率与测评要求 `[待验证 — 须核实公安部等保测评规定]`）：

| 测评类型 | 要求 | 说明 |
|---------|------|------|
| **年度测评（第三级）** | 每年开展一次 `[待验证]` | 须委托经公安部认可的等保测评机构（`[待验证机构认可规定]`） |
| **半年测评（第四级）** | 每半年开展一次 `[待验证]` | 测评结论：优/良/中/差 `[待验证具体结论标准]` |
| **首次测评** | 系统初次定级备案后开展 `[待验证]` | 发现差距，形成整改项清单 |
| **测评机构选择** | 须选择公安部认可的等保测评机构（`[待验证机构资质要求]`） | 不得自行测评，须委托第三方 |

**等级测评关注维度（参考）：**
- 测评机构出具测评报告，评定测评结论（整体安全保护能力等级）
- 测评报告揭示不符合项，形成整改建议清单
- 整改完成后，重要差距项须整改验证

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 系统属于关键信息基础设施 | 触发 `/data-compliance:cii-protection`，等保与关基保护叠加运用；关基系统通常定级≥三级 `[待验证]` |
| 系统部署在公有云 | 须参照等保2.0云计算安全扩展要求；云服务提供商的等保合规资质需核实 `[待验证]` |
| 工业控制系统（OT/ICS） | 须参照等保2.0工控安全扩展要求；安全措施设计需考虑工控可用性优先原则 `[待验证]` |
| 多个信息系统合并定级 | 等保制度要求每个系统独立定级（一般不得合并），特殊情形须核实 `[待验证]` |
| 已定级系统发生重大变更 | 功能变更/用户规模大幅增加/新增重要数据处理，须评估是否触发重新定级 |
| 小微企业/低等级系统 | 一级系统自主保护，无强制备案和测评要求（`[待验证]`）；但基本安全措施（防病毒/备份/访问控制）仍应具备 |
| 系统已被认定为第三级但未开展测评 | 🔴 关键风险 — 公安机关检查中发现未测评可能面临行政处罚 |

## 错误处理

| 错误情形 | 处理动作 |
|---------|---------|
| 用户提供的系统信息不足以判断等级 | 列出缺失信息（业务功能/用户规模/数据类型），要求补充后再判断 |
