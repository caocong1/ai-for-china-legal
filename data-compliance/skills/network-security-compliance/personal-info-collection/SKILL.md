## 目的

依据《网络安全法》（2017年）关于网络运营者收集使用个人信息的规定（具体条号 `[待验证]`），结合《个人信息保护法》（PIPL，2021年11月1日施行）的核心要求（已核实），审查网络运营者个人信息收集使用的合法性——包括合法正当必要原则、公开规则与告知同意机制、最小必要原则、用户权利保障（查阅/更正/删除），以及 CSL 与 PIPL 的衔接适用关系，形成个人信息收集合规评估与整改建议。

---

## 详细步骤

### 步骤一：理解 CSL 与 PIPL 的关系与适用

**CSL/PIPL 衔接框架**：

| 维度 | CSL（2017年）| PIPL（2021年）| 适用原则 |
|-----|-------------|-------------|---------|
| 立法层级 | 全国人大常委会制定（同级） | 全国人大常委会制定（同级） | 同级，后法（PIPL）优先；特别法优先 |
| 适用范围 | 网络运营者收集个人信息（网络服务场景） | 个人信息处理者处理个人信息（所有场景，更广泛） | PIPL 范围更广；涵盖 CSL 网络运营者场景 |
| 个人信息保护主要规则 | 规定收集使用的合法正当必要原则、公开规则、同意（具体条号 `[待验证]`） | 全面规定合法性基础（第13条，已核实）、单独同意（第28-29条，已核实）、数据主体权利（第44-47条，已核实）等 | **PIPL 优先适用个人信息保护事项**；PIPL 未规定时适用 CSL |
| 主要执法机关 | 网信办/公安 | 网信办 | 同一机关，协同执法 |

**实践适用建议**：对于网络运营者处理个人信息的场景，**以 PIPL 为基本分析框架**（覆盖范围更广、规则更完整）；CSL 关于网络运营者的特定规定在 PIPL 未覆盖的边界情形下补充适用。

### 步骤二：合法正当必要原则审查

**合法正当必要原则**（CSL `[待验证条号]`；PIPL 第13条，**已核实**）：

| 原则 | CSL 要求（`[待验证]`）| PIPL 要求（已核实）| 审查要点 |
|-----|-------------------|--------------------|---------|
| **合法** | 收集须有合法依据（用户同意/合同所必需等） | 第13条：处理个人信息须满足七类合法性基础之一 | 每类处理活动是否有明确合法性基础 |
| **正当** | 须公开收集使用规则，不得欺骗误导 | 须履行告知义务（第17条 `[待验证]`）；不得采取误导方式 | 隐私政策是否清晰、准确、可获取 |
| **必要** | 收集须与服务有关联（不收集无关信息） | 最小必要原则：不得收集与处理目的无关的个人信息 | 逐项审查收集字段的必要性 |

**合法正当必要审查步骤：**

1. **梳理收集的个人信息清单**（与 `data-mapping-necessity` 分工：本技能做合规框架审查；`data-mapping-necessity` 做深度映射）
2. **判断每项信息的合法性基础**（PIPL 第13条，已核实）
3. **验证必要性**：是否与提供服务直接相关？是否超出服务需要？
4. **检查正当性**：隐私政策是否真实、清晰？收集行为是否欺骗用户？

### 步骤三：告知同意机制审查

**告知义务**（PIPL 第17条 `[待验证]`；CSL `[待验证条号]`）：

须告知用户：个人信息处理者名称/联系方式、处理目的与方式、个人信息类型与保存期限、行使权利的方式途径，以及其他须告知的事项。

**同意规则**（PIPL 第14条，**已核实**）：
- 同意须**自愿**（不捆绑，可与服务分离）
- 同意须**明确**（不默许，不预先勾选）
- 同意须**知情**（告知充分）
- 须提供便捷的**撤回同意**机制（撤回后不得遭受不合理后果）

| 告知同意审查项 | 合规要点 | 风险信号 |
|-------------|---------|---------|
| 隐私政策存在性 | 平台/应用须有对外公示的隐私政策 | 无隐私政策或难以访问 = 🔴 |
| 隐私政策内容完整性 | 须覆盖所有处理目的、信息类型、保存期限、接收方等 | 内容笼统缺乏实质 = 🟠 |
| 同意机制的自愿性 | 注册同意与服务提供须可分离（合同所必需的不须另取同意） | 捆绑同意（强制同意注册/营销）= 🔴 |
| 同意机制的明确性 | 不得预先勾选/默许同意 | 预选勾选同意框 = 🔴 |
| 撤回同意便捷性 | 须提供易于操作的撤回同意入口 | 撤回途径繁琐/需申请审批 = 🟠 |
| App 收集权限 | 须告知申请权限的目的，不得申请与功能无关的权限 | 超范围权限（如记事本申请通讯录）= 🔴 |

**敏感个人信息单独同意**（PIPL 第28-29条，**已核实**；须另行触发 `trigger-consent-basis`）：

| 敏感个人信息类别 | 单独同意要求 |
|--------------|------------|
| 生物识别信息（人脸/指纹/声纹等） | 须单独同意；不得与一般个人信息捆绑 |
| 医疗健康信息 | 须单独同意 |
| 金融账户信息 | 须单独同意 |
| 行踪轨迹信息 | 须单独同意 |
| 宗教信仰、特定身份 | 须单独同意 |
| 不满十四周岁未成年人信息 | 须取得监护人同意；须制定专门处理规则 |

> 深度单独同意机制审查，触发：`/data-compliance:trigger-consent-basis`

### 步骤四：最小必要原则审查

**最小必要原则**（PIPL `[待验证条号]`；App 最小必要要求 `[行业监管 flagged — 须核实网信办App最小必要规定]`）：

**审查方法：**

| 审查项 | 判断问题 | 合规标准 |
|-------|---------|---------|
| 字段必要性 | 该字段是否必须收集才能实现声称的服务目的？ | 非必要字段须删除或改为可选 |
| 频率必要性 | 收集该信息的频率是否与服务需求匹配？ | 不得超频采集（如持续获取位置）|
| 范围必要性 | 精度/范围是否过宽？ | 导航只需当前位置，不需历史轨迹 |
| 保存期限必要性 | 保存期限是否超出服务需要？ | 须设置并实施合理保存期限 |
| 功能外收集 | 是否收集了与当前功能无关的信息？ | 申请无关权限 = 最小必要违反 |

**最小必要合规清单（App/产品）：**

- [ ] 申请的系统权限列表与功能需求逐一对应（位置/摄像头/通讯录/麦克风等）
- [ ] 非核心功能使用敏感权限时，有明确的用户选择机制（可拒绝且不影响核心服务）
- [ ] 收集字段逐一验证必要性，非必要字段已标注说明（最小必要）
- [ ] 个人信息保存期限已在隐私政策中明确
- [ ] 保存期限届满的信息有自动删除机制（或人工定期清理）

### 步骤五：用户权利保障审查

**用户权利保障**（PIPL 第44-47条，**已核实**）：

| 权利 | 法律依据 | 合规要求 | 验证状态 |
|-----|---------|---------|---------|
| 知情权与决定权 | PIPL 第44条（已核实）| 须向用户提供充分知情；须尊重用户对处理活动的决定权 | 已核实 |
| 查阅权与复制权 | PIPL 第45条（已核实）| 须提供便捷查阅渠道；须及时提供复制 | 已核实 |
| 更正补充权 | PIPL 第46条（已核实）| 核实后须及时更正不准确信息 | 已核实 |
| 删除权 | PIPL 第47条（已核实）| 满足法定情形须删除（含主动删除义务） | 已核实 |
| 个人信息可携带权 | PIPL `[待验证条号]` | 达到一定规模的处理者须支持数据可携带 `[待验证]` | `[待验证]` |

**权利行使机制审查：**

- [ ] 用户有明确入口查阅自己的个人信息（账户设置/隐私中心）
- [ ] 更正请求响应机制（审核后及时更正，不得无故拒绝）
- [ ] 删除请求响应机制（满足第47条法定情形时须删除）
- [ ] 撤回同意的响应：撤回后停止相关处理活动
- [ ] 账户注销流程：注销后须删除/匿名化个人信息（`[待验证注销时限]`）
- [ ] 权利行使的响应时限：`[待验证 — 须核实 PIPL 对响应时限的具体规定]`

### 步骤六：CSL 与 PIPL 的衔接对照

**CSL/PIPL 衔接对照表（网络运营者收集场景）：**

| 合规事项 | CSL 规则（`[待验证条号]`）| PIPL 规则（条号已核实或待验证）| 实践适用 |
|---------|-------------------|-----------------------------|---------|
| 合法性基础 | 合法依据（具体条号 `[待验证]`）| 第13条（已核实）：七类合法性基础 | PIPL 优先，以第13条为分析框架 |
| 告知义务 | 公开收集使用规则（`[待验证]`）| 第17条 `[待验证]`：告知内容要求 | PIPL 第17条更完整，优先适用 |
| 同意 | 须取得被收集者同意（`[待验证]`）| 第14条（已核实）：自愿、明确、知情 | PIPL 优先 |
| 最小必要 | 不得收集无关信息（`[待验证]`）| 最小必要 `[待验证]`；App 最小必要规定 `[行业监管 flagged]` | PIPL + 行业规定 |
| 用户权利 | 须保护用户信息保密（`[待验证]`）| 第44-47条（已核实）：知情/查阅/更正/删除 | PIPL 优先，更完整 |
| 泄露补救 | 泄露时须通知用户 `[待验证]` | PIPL 第57条 `[待验证]`：泄露通知义务 | PIPL 优先 |
| 执法机关 | 网信办/公安 | 网信办 | 同一机关，协同执法 |

## 检查清单

**个人信息收集合规关键检查项：**

- [ ] 每类个人信息收集均有明确合法性基础（PIPL 第13条，已核实）
- [ ] 隐私政策存在且对外公示，内容完整（处理目的/方式/类型/保存期限/接收方等）
- [ ] 注册/使用同意机制：自愿、明确、知情；不捆绑非必要处理
- [ ] 撤回同意机制便捷，撤回后停止相关处理
- [ ] 敏感个人信息（生物识别/医疗/金融账户/行踪/宗教/未成年人）取得单独同意（PIPL 第28-29条，已核实）
- [ ] App 权限申请与功能需求逐一对应，不超范围申请
- [ ] 个人信息保存期限已设置并实施（到期自动/定期删除）
- [ ] 用户查阅/更正/删除权利行使机制已建立
- [ ] 账户注销流程：注销后删除/匿名化个人信息
- [ ] 行业特殊规定已核查（金融/医疗/教育/互联网等） `[行业监管 flagged]`

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 仅收集匿名化数据 | 匿名化（而非去标识化/假名化）后不属于个人信息，须核实匿名化是否真正不可逆 `[待验证标准]` |
| 受托处理者场景（委托处理）| 受托处理者按处理者指示处理，合规义务主要在委托方；受托方须签订数据处理协议 |
| 员工个人信息收集（HR 场景）| PIPL 同样适用于员工个人信息处理；劳动合同场景可援引合同所必需合法性基础 `[待验证]` |
| 开展直接营销 | 单独同意或退出权（opt-out）须核实适用规则 `[待验证 — 须核实 PIPL 营销场景具体规定]` |
| 算法推荐场景 | 涉及 PIPL 第24条自动化决策，须提供非针对个人的选项和拒绝权（已核实） |
| 泄露事件发生 | 触发 PIPL 泄露通知义务 `[待验证 — 须核实 PIPL 第57条具体规定]`；升级至 risk-escalation |
| 用户信息存储期满 | 触发主动删除义务（PIPL 第47条，已核实）；确认自动化删除机制 |

## 错误处理

| 错误情形 | 处理动作 |
|---------|---------|
| 用户声称「已有隐私政策无需审查」| 合理性检查隐私政策内容完整性，逐项核对告知义务要点 `[前提 flagged — 需验证]` |
| 用户口述同意满足PIPL要求但同意机制有捆绑 | 指出捆绑同意与 PIPL 第14条（已核实）不符，建议分离设计 |
| 用户提及「平均用时不到一秒的同意弹窗符合要求」| 指出告知充分性和自愿性问题（知情同意须有合理时间阅读）|
| 发现收集与服务无明显关联的个人信息 | 🔴 最小必要违反 — 建议删除该字段或明确必要性说明 |
| 用户询问是否需要 DPO | 解释 PIPL 第52条（待验证）DPO 要求（达到规定规模须指定负责人），建议确认适用 |
| 涉及 App 权限超范围申请 | 标记 🔴 关键风险，建议立即整改（网信办 App 专项整治执法较活跃） |
