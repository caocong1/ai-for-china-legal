## 目的

依据《网络安全法》（2017年）（具体网络运营者义务条文 `[待验证 — 须核实 CSL 各条对应义务条号]`）及等保制度（CSL 第21条，**已核实**），对网络运营者开展一般安全义务的全面审查，涵盖：用户实名制管理、网络日志留存、安全技术措施、安全事件应急预案与处置、配合监督检查，以及违法信息处置义务。形成运营者义务履行评估报告与整改清单。

本子技能适用于所有「网络运营者」——即网络的所有者、管理者和网络服务提供者（CSL 对「网络运营者」的定义 `[待验证具体条号]`）。

---

## 详细步骤

### 步骤一：确认网络运营者身份

**「网络运营者」定义**（CSL `[待验证具体条号]`）：网络的所有者、管理者和网络服务提供者。广义上，运营网络或提供网络服务的组织均属「网络运营者」范畴，须履行 CSL 规定的各项义务。

| 运营者类型 | 典型主体 | 义务重点 |
|---------|---------|---------|
| 互联网平台/应用服务商 | 电商平台、社交媒体、SaaS 服务 | 实名制/违法信息处置/日志留存/用户信息保护 |
| 电信基础运营商 | 基础电信企业 | 实名制（最严格）/网络安全/技术措施 `[行业监管 flagged]` |
| 企业内网运营者 | 企业 IT 部门、政府机关网络管理 | 等保义务/日志留存/访问控制/安全管理 |
| IDC/云服务提供商 | 互联网数据中心/云平台 | 用户实名/安全措施/数据保护/监管配合 |

### 步骤二：网络实名制义务审查

**网络实名制**（CSL `[待验证具体条号]`；电信运营商有更严格的电信用户实名制要求 `[行业监管 flagged]`）：

**核心规则**（具体条号 `[待验证 — 须核实 CSL 相关条款]`）：
- 网络运营者在与用户签订服务协议或确认提供服务时，须要求用户提供**真实身份信息**
- 用户未提供真实身份信息的，网络运营者**不得为其提供相关服务**
- 同时须尊重并保护用户依法享有的匿名表达权（具体边界 `[待验证]`）

| 实名制审查项 | 合规要点 | 风险等级 | 验证状态 |
|------------|---------|---------|---------|
| 注册/登录实名验证机制 | 须验证用户真实身份（手机号/身份证/人脸识别等方式）`[待验证具体要求]` | 🔴 | `[待验证]` |
| 匿名帐号限制措施 | 未完成实名的帐号须限制服务功能 `[待验证]` | 🔴 | `[待验证]` |
| 实名信息保存与保护 | 收集的实名信息须严格保护，防止泄露（同时适用 PIPL 个人信息保护义务） | 🔴 | `[待验证]` |
| 电信/互联网实名特别规定 | 电信运营商须遵守工信部电话实名制规定 `[行业监管 flagged — 须核实工信部最新规定]` | 🔴 | `[行业监管 flagged]` |

### 步骤三：网络日志留存义务审查

**日志留存义务**（CSL 第21条，**已核实**：监测记录网络运行状态，留存不少于六个月——具体留存期限须结合等保要求核实 `[待验证具体留存期限规定]`）：

> **注意**：CSL 第21条已核实（等保制度一般义务）规定须留存监测记录；「不少于六个月」是通常理解，**具体留存期限须核实公安部等保主管机关与网信办最新规定** `[待验证]`。行业特殊规定（如金融/电信）可能有更长留存期要求 `[行业监管 flagged]`。

| 日志类型 | 留存要求（参考，须验证） | 说明 |
|---------|---------------------|------|
| 网络访问日志 | 不少于六个月 `[待验证具体期限]` | 用户访问行为记录 |
| 系统运行状态日志 | 不少于六个月 `[待验证]` | 服务器/系统运行状态 |
| 用户操作行为日志 | 不少于六个月 `[待验证]` | 用户在平台的关键操作记录 |
| 安全事件日志 | 须保留，期限须核实 `[待验证]` | 安全告警、异常访问等 |
| 管理员操作审计日志 | 须保留，期限须核实 `[待验证]` | 特权账号操作记录 |
| 金融/电信特殊日志 | 行业规定留存期可能更长 `[行业监管 flagged — 须核实主管部门规定]` | 金融/电信行业须另行核实 |

**日志留存合规审查要点：**

- [ ] 网络访问日志：记录是否完整（IP/时间/请求内容摘要等），留存是否满足最低期限 `[待验证]`
- [ ] 日志的完整性与不可篡改性：是否有防篡改措施（如日志签名/集中审计平台）
- [ ] 日志存储的安全性：日志文件是否加密/访问受控
- [ ] 日志是否支持监管查阅：是否可向公安机关/网信办提供 `[待验证]`
- [ ] 行业特殊留存要求：金融/医疗/电信行业须额外核查 `[行业监管 flagged]`

### 步骤四：安全技术措施审查

**安全技术措施**（CSL 第21条，**已核实**；CSL 关于技术措施的具体条款 `[待验证]`）：

网络运营者须采取以下技术措施（依据等保义务与 CSL 第21条，具体措施标准按等保级别有所不同）：

| 技术措施类别 | 主要内容 | 合规检查要点 |
|------------|---------|------------|
| **防入侵措施** | 防火墙、入侵检测/防御系统（IDS/IPS）、漏洞扫描 `[待验证具体标准]` | 是否部署；是否定期更新规则 |
| **防病毒与恶意代码** | 防病毒软件、恶意代码检测 `[待验证]` | 是否覆盖所有终端与服务器；更新频率 |
| **数据分类保护** | 依数据分级采取差异化保护措施（CSL 第21条涉及数据分类）| 是否已完成数据分类；分类后是否采取对应措施 |
| **重要数据备份与加密** | 关键数据定期备份；敏感/重要数据加密存储 | 备份频率与恢复测试；加密算法是否合规 `[待验证]` |
| **身份鉴别与访问控制** | 多因子认证、最小权限原则、账户生命周期管理 | 特权账户管理；离职人员账户及时注销 |
| **安全监测** | 安全信息与事件管理（SIEM）、异常流量监测、安全基线检查 | 是否有持续监测机制；告警响应流程 |
| **传输加密** | 重要数据传输须加密（HTTPS、加密通道等） | API 接口是否使用加密通信；内外网传输 |

### 步骤五：安全事件应急预案与处置

**安全事件应急管理义务**（CSL `[待验证具体条号]`）：

**应急预案要素：**

| 预案要素 | 说明 | 合规要点 |
|---------|------|---------|
| **事件分级标准** | 按安全事件的危害程度分级（参考国家应急预案框架，具体分级标准 `[待验证]`） | 是否有明确的事件级别定义 |
| **响应流程** | 发现→报告→启动预案→遏制→调查→恢复→总结 | 响应流程是否文档化并定期演练 |
| **通知机制** | 须向主管部门报告（报告时限与机制 `[待验证]`）；通知受影响用户（PIPL 第57条 `[待验证]`） | 是否有预设的监管报告流程 |
| **取证保留** | 安全事件发生后须保留日志等证据，配合调查 | 日志是否可完整导出供调查使用 |
| **演练频率** | 定期开展应急预案演练（`[待验证 — 须核实 CSL 或等保规定的演练频率]`） | 年度演练计划是否执行 |

**安全事件处置合规要点表：**

| 事件类型 | 处置义务 | 报告义务 | 验证状态 |
|---------|---------|---------|---------|
| 数据泄露（涉及个人信息） | 立即采取止损措施；保留证据；通知受影响用户（PIPL `[待验证条号]`） | 须向网信/公安报告（时限 `[待验证]`） | `[待验证]` |
| 重大安全事件（网络瘫痪/系统被入侵） | 启动应急预案；遏制影响；向主管部门报告 | 须报告（时限 `[待验证]`） | `[待验证]` |
| DDoS 攻击 | 启用防御措施；向运营商/安全服务商求助 | 视影响程度决定是否报告 `[待验证]` | `[待验证]` |
| 勒索软件 | 立即隔离；不建议支付赎金；向公安机关报案 | 涉及重要数据须向主管部门报告 `[待验证]` | `[待验证]` |

### 步骤六：配合监督检查与违法信息处置

**配合监督检查**（CSL `[待验证具体条号]`）：
- 网络运营者须配合网信办、公安机关等监管部门的合法检查
- 包括：提供数据、日志、文件等；配合技术检测；提供场地与设施
- 在法定范围内提供协助，不得拒绝阻挠监管检查 `[待验证]`

**违法信息处置义务**（CSL `[待验证具体条号]`）：
- 发现法律、行政法规禁止发布或者传输的信息（如淫秽物品、危害国家安全信息等），须立即停止传输，保存相关记录，并向主管部门报告 `[待验证]`
- 须建立违法信息监测与处置机制（依据内容安全审核要求，具体标准 `[待验证]`）

## 检查清单

**网络运营者安全义务关键检查项：**

- [ ] 用户注册/登录环节实施实名验证，未实名用户无法获取相关服务 `[待验证]`
- [ ] 实名信息严格保护，访问受限，防止泄露
- [ ] 网络访问/操作/安全日志留存，期限满足法规最低要求（≥六个月，须验证） `[待验证]`
- [ ] 日志完整性保护（防篡改），支持监管查阅
- [ ] 防病毒/防入侵/漏洞管理措施已部署并定期更新
- [ ] 重要数据备份定期执行，恢复流程已测试
- [ ] 身份鉴别与访问控制：最小权限原则，特权账户管理，离职人员账户注销
- [ ] 传输加密：内外部数据传输使用加密通道
- [ ] 安全事件应急预案已制定并文档化，定期演练 `[待验证]`
- [ ] 安全事件报告流程：内部升级与监管报告时限明确 `[待验证]`
- [ ] 违法信息监测与处置机制已建立 `[待验证]`
- [ ] 行业特殊义务已核查（电信/金融/医疗等） `[行业监管 flagged]`

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 小微企业/低流量网站 | 基本义务仍须履行（实名制/基本安全措施）；但合规成本可与业务规模相适配 `[待验证]` |
| 纯 B2B 企业（无个人用户） | 实名制义务的适用范围须核实（CSL 主要针对「用户」，B2B 场景是否适用须验证） `[待验证]` |
| 开源社区/非营利性质网络 | 义务是否相同须核实（CSL 无明确豁免） `[待验证]` |
| 仅提供 API 服务 | 实名制在 API 场景的适用（API 客户是否「用户」）须核实 `[待验证]` |
| 跨境业务（境外用户访问） | 日志中的境外用户数据的保护须结合 PIPL 跨境规则处理 |
| 遭遇监管调查 | 须配合调查；保留相关日志和文件；升级至法务总监/外部律师 |

## 错误处理

| 错误情形 | 处理动作 |
|---------|---------|
| 用户声称已完成实名制但无书面证明 | 询问实名制覆盖范围、验证机制及记录保存情况，合理性检查 `[前提 flagged — 需验证]` |
| 用户口述日志留存期限（如「我们留存3个月」） | 提示六个月最低要求（须验证），标 `[待验证 — 须核实法规最新规定]`；建议整改 |
| 用户提及行业特殊日志要求 | 确认行业主管部门规定，标 `[行业监管 flagged — 须核实主管部门最新规定]` `[前提 flagged — 需验证]` |
| 发现严重安全事件隐患（如日志完全缺失）| 🔴 关键风险 — 立即建议整改，并评估是否须向监管部门主动沟通 |
| 用户询问监管检查配合义务范围 | 说明原则（按法定要求配合，不超范围）；具体要求须结合实际检查通知核实 |
