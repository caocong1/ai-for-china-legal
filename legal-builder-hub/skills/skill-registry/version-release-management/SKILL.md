# 版本发布管理（version-release-management）

> **所属编排**：`legal-builder-hub/skills/skill-registry/SKILL.md`（技能注册表）
> **分工说明**：本子技能负责版本号和发布流程管理。清单校验由 `skill-manifest-management` 处理；依赖解析由 `dependency-resolution` 处理；注册表操作由 `registry-search-publish` 处理。
> **可单独触发**：`/legal-builder-hub:version-release-management`

---

## 目的

管理技能版本的完整生命周期：语义化版本号管理（SemVer）、变更日志生成、发布前检查清单执行、版本发布操作、发布后验证。确保版本可追溯、变更可回溯。

---

## 前置

1. **获取当前版本信息** — 从 plugin.json 读取当前版本号
2. **获取变更内容** — Git 提交历史或手动输入的变更说明
3. **读取 `legal-builder-hub/CLAUDE.md`** — 获取：发布审核流程
4. 本子技能可修改版本号和生成发布文件

## 语义化版本（SemVer）规范

### 版本号格式

```
MAJOR.MINOR.PATCH[-pre-release][+build-metadata]
```

| 组件 | 变更规则 | 示例 |
|------|---------|------|
| MAJOR | 不兼容的 API 变更 | 1.0.0 → 2.0.0 |
| MINOR | 向后兼容的功能新增 | 1.0.0 → 1.1.0 |
| PATCH | 向后兼容的问题修复 | 1.0.0 → 1.0.1 |
| pre-release | 预发布标识 | 1.0.0-alpha.1, 1.0.0-beta.2 |

### 版本升降级决策树

```
本次变更是否包含不兼容的 API 变更？
├─ 是 → MAJOR 升级（MAJOR+1, MINOR=0, PATCH=0）
└─ 否
    ├─ 是否包含新功能？
    │   ├─ 是 → MINOR 升级（MINOR+1, PATCH=0）
    │   └─ 否 → PATCH 升级（PATCH+1）
    └─ 是否为预发布？
        ├─ 是 → 添加/更新 pre-release 标识
        └─ 否 → 移除 pre-release 标识
```

## 变更日志生成

### 变更分类

| 分类 | 标签 | 说明 |
|------|------|------|
| 新增功能 | `新增` | 新增的技能/子技能/功能 |
| 改进优化 | `改进` | 现有功能的改进和优化 |
| 问题修复 | `修复` | 缺陷和问题的修复 |
| 安全修复 | `安全` | 安全漏洞的修复 |
| 废弃移除 | `废弃` | 废弃或移除的功能 |
| 依赖变更 | `依赖` | 依赖项的升级或变更 |

### 变更日志格式

```markdown
## [版本号] - [发布日期]

### 新增
- [变更说明]（[#issue]）

### 改进
- [变更说明]

### 修复
- [变更说明]（修复 [#issue]）

### 安全
- [变更说明]（CVE-[编号]）
```

## 发布流程

### 步骤一：版本升级

1. 确认变更类型（MAJOR/MINOR/PATCH）
2. 更新 plugin.json 中的 `version` 字段
3. 生成变更日志条目
4. 提交版本变更

### 步骤二：发布审核

根据用户配置的发布审核流程：

| 审核流程 | 行为 |
|---------|------|
| 自动审核 | 通过发布前检查清单后自动发布 |
| 人工审核 | 提交审核请求，等待审核员批准 |
| 社区审核 | 发布到社区，等待社区投票 |

### 步骤三：执行发布

1. 创建版本标签（Git tag）
2. 打包发布产物
3. 推送到注册表（→ `registry-search-publish`）
4. 记录发布日志

### 步骤四：发布后验证

```markdown
## 发布验证

**技能**：[名称] v[版本号]
**发布时间**：[时间戳]
**注册表**：[注册表名称/地址]

| 验证项 | 结果 |
|-------|------|
| 注册表记录 | ✅ / ❌ |
| 下载测试 | ✅ / ❌ |
| 安装测试 | ✅ / ❌ |
| 基础功能 | ✅ / ❌ |
```

## 版本撤销

当已发布版本存在严重问题时：

1. 在注册表中标注版本为「已撤销」
2. 记录撤销原因
3. 通知已安装用户
4. 建议升级到修复版本

**注意**：已发布版本不可覆盖（只能撤销后发布新版本号）。

---

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 版本号已被使用 | 阻止发布；建议使用更高版本号 |
| 安全审查未通过 | 阻止发布；建议先完成安全审查 |
| 变更日志为空 | 警告但不阻止；建议补充变更说明 |
| 发布到远程注册表失败 | 重试；保留本地发布记录 |
