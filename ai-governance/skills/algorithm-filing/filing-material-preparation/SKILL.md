# 算法备案材料准备与提交（filing-material-preparation）

> **前置**：
> 1. 读取 `ai-governance/CLAUDE.md`，加载服务立场与合规严格度。
> 2. 读取 `ai-governance/skills/_shared/legal-basis-conventions.md`。
> 3. 读取 `ai-governance/skills/_shared/ai-law-citations.md`（§D.2 算法推荐管理规定）。
> 4. 已完成备案要求分析（本子技能的前序子技能或单独触发时由用户提供备案要求信息）。

---

## 目的

按算法备案要求系统准备全部备案材料，包括主体信息、算法信息、服务信息、安全管理制度、算法透明度方案、用户权益保护方案、算法自评估报告等，通过互联网信息服务算法备案系统完成提交。

具体目标：

1. **材料清单确认**：根据算法类型与备案要求确认全部所需材料
2. **算法信息材料准备**：算法机制说明·训练数据说明·算法基本信息
3. **管理制度材料准备**：安全管理制度·算法透明度方案·用户权益保护方案
4. **安全评估报告撰写**：算法自评估报告的各章节撰写
5. **材料质量检查**：确保材料完整·准确·一致
6. **备案系统提交**：通过备案系统完成信息填写·材料上传·提交审核

---

## 详细步骤

### 步骤 1：备案材料清单确认

根据算法推荐管理规定与备案系统要求，梳理备案所需材料清单：

| 材料类别 | 材料名称 | 说明 | 质量要求 |
|---------|---------|------|---------|
| **主体信息** | 企业基本信息 | 名称·统一社会信用代码·联系方式·地址·法定代表人 | 须与营业执照一致 |
| **主体信息** | 算法安全负责人信息 | 负责人姓名·职务·联系方式 | 须为实际负责人 |
| **算法信息** | 算法基本信息 | 算法名称·算法类型·应用领域·上线时间 | 须与前序子技能识别结果一致 |
| **算法信息** | 算法机制说明（算法原理说明） | 算法基本原理·模型架构·核心机制·运行机制·主要参数 | 须准确描述，不涉及商业秘密的核心参数可概括描述 |
| **算法信息** | 算法训练数据说明 | 数据来源·数据类型·数据规模·数据合规性说明 | 须说明数据来源合法性 |
| **算法信息** | 算法拟公示内容 | 向用户公示的算法推荐服务基本信息·基本原理·目的意图·主要运行机制 | 须符合算法透明度要求 `[待验证]` |
| **服务信息** | 服务说明 | 服务类型·应用场景·用户群体·服务覆盖范围 | 须与实际服务一致 |
| **管理制度** | 算法安全管理制度 | 算法安全审查制度·安全监测制度·应急响应制度 | 须为企业正式发布的制度文件 |
| **管理制度** | 内容安全管理制度 | 内容审核制度·违规内容处置制度·关键词过滤机制 | 须有具体操作流程 |
| **管理制度** | 数据安全管理制度 | 数据分类分级·存储保护·传输加密·访问控制 | 须与数据安全法要求衔接 |
| **管理制度** | 算法透明度方案 | 算法公示方案·用户告知方案 | 须符合算法推荐管理规定的透明度要求 `[待验证]` |
| **管理制度** | 用户权益保护方案 | 用户选择权保障方案·投诉举报机制·用户申诉处理 | 须有具体操作流程与响应时限 |
| **技术措施** | 算法安全技术措施 | 算法安全监测·偏见检测·异常行为识别·应急处置 | 须有技术实现说明 |
| **评估材料** | 算法自评估报告 | 算法安全评估结果与结论 | 须覆盖全部评估维度 |

### 步骤 2：算法信息材料准备

**算法基本信息**：
- 确认算法类型（生成合成/个性化推送/排序精选/检索过滤/调度决策）
- 描述算法名称与应用领域
- 说明算法上线时间或服务计划上线时间

**算法机制说明（算法原理说明）**：

算法机制说明是备案材料的核心技术文件，须包含以下内容：

| 章节 | 内容要求 | 撰写要点 |
|------|---------|---------|
| 算法基本原理 | 描述算法的基本技术原理与模型架构 | 用通俗语言描述，避免过度技术化；不涉及商业秘密的核心参数可概括描述 |
| 核心机制 | 描述算法的核心处理机制 | 说明输入→处理→输出的完整流程 |
| 运行机制 | 描述算法在实际服务中的运行方式 | 说明算法在用户请求→数据处理→结果返回中的具体作用 |
| 主要参数 | 描述影响算法输出的关键参数 | 关键参数须说明，商业秘密参数可概括描述 |
| 调优方式 | 描述算法模型的训练与调优方式 | 说明训练数据来源·训练方法·调优策略 |

**算法训练数据说明**：

| 说明项 | 内容要求 | 注意事项 |
|--------|---------|---------|
| 数据来源 | 公开数据集·自有数据·授权数据·网络爬取数据等 | 须说明各类来源的占比概况 |
| 数据类型 | 文本·图像·音频·视频·结构化数据等 | 须与算法类型匹配 |
| 数据规模 | 数据量级概况 | 可描述量级范围，不要求精确数字 |
| 数据合规性 | 数据来源合法性·知识产权合规·个人信息合规 | 须说明数据合规保障措施 |
| 数据标注 | 数据标注方式·标注质量控制 | 如涉及人工标注须说明标注规范 |

### 步骤 3：管理制度材料准备

**安全管理制度**：
- 算法安全管理制度：算法安全审查制度·安全监测制度·应急响应制度·安全事件报告制度
- 内容安全管理制度：内容审核制度（自动审核 + 人工审核）·违规内容处置制度·关键词过滤机制·内容分类分级
- 数据安全管理制度：数据分类分级制度·数据存储保护制度·数据传输加密制度·数据访问控制制度·数据删除制度

**算法透明度方案**：
- 向用户公示算法推荐服务基本情况（服务类型·功能描述）
- 说明算法推荐服务的基本原理·目的意图和主要运行机制（以用户可理解的方式）
- 提供关闭算法推荐服务的选项与操作指引
- 提供不针对个人特征的推荐选项

**用户权益保护方案**：
- 提供不针对个人特征的推荐选项（具体实现方式）
- 提供便捷的关闭算法推荐服务选项（操作路径·生效时间）
- 建立健全投诉举报机制（投诉渠道·响应时限·处理流程）
- 用户申诉处理机制（申诉受理·调查·反馈·救济）

### 步骤 4：算法自评估报告撰写

算法自评估报告是备案材料的核心评估文件，须覆盖以下章节：

| 评估章节 | 内容要求 | 撰写要点 |
|---------|---------|---------|
| **评估概述** | 评估目的·评估范围·评估方法·评估时间·评估人员 | 须明确评估范围覆盖全部算法推荐服务 |
| **算法安全评估** | 算法偏见检测·歧视风险评估·滥用风险评估·算法鲁棒性评估 | 须有具体检测方法与结果 |
| **内容安全评估** | 算法推荐内容合规性评估·违规内容识别能力评估·内容审核机制有效性评估 | 须有测试数据与结果 |
| **数据安全评估** | 训练数据来源合法性评估·用户数据保护措施评估·数据泄露风险评估 | 须与数据安全管理制度衔接 |
| **用户保护评估** | 用户知情权保障评估·用户选择权保障评估·投诉举报机制有效性评估 | 须有用户测试或验证结果 |
| **可解释性评估** | 算法可解释性评估·透明度评估·公示内容适当性评估 | 须评估公示内容是否为用户可理解 |
| **评估结论** | 总体评估结论·发现的风险与问题·改进建议·整改计划 | 结论须明确，风险须分级 |

### 步骤 5：材料质量检查

**材料完整性检查**：

| 检查项 | 检查内容 | 常见问题 |
|--------|---------|---------|
| 主体信息完整性 | 企业名称·信用代码·联系方式等是否完整 | 信息遗漏·与营业执照不一致 |
| 算法信息完整性 | 算法机制说明·训练数据说明是否完整 | 技术描述过于简略·训练数据来源未说明 |
| 管理制度完整性 | 各项制度是否齐全·是否有正式文件 | 制度文件未正式发布·缺少具体操作流程 |
| 评估报告完整性 | 评估报告是否覆盖全部维度 | 缺少某些评估维度·评估结论不明确 |
| 信息一致性 | 各材料之间的信息是否一致 | 算法类型描述不一致·服务描述前后矛盾 |

**材料准确性检查**：

| 检查项 | 检查内容 |
|--------|---------|
| 技术描述准确性 | 算法原理·机制·参数描述是否与实际一致 |
| 法律引用准确性 | 法律依据引用是否符合引用规范 |
| 数据一致性 | 各材料中引用的数据是否一致 |
| 格式规范性 | 材料格式是否符合备案系统要求 |

### 步骤 6：备案系统提交

按备案系统要求完成以下操作：

1. 注册/登录互联网信息服务算法备案系统
2. 填写企业基本信息与算法安全负责人信息
3. 填写算法基本信息（类型·名称·应用领域）
4. 上传算法机制说明·训练数据说明
5. 上传安全管理制度文件
6. 上传算法透明度方案与用户权益保护方案
7. 上传算法自评估报告
8. 上传拟公示内容
9. 确认信息无误后提交审核
10. 跟踪审核进度，及时响应补充材料要求

## 常见材料缺陷与补救

| 常见缺陷 | 缺陷描述 | 补救措施 |
|---------|---------|---------|
| 算法机制说明过于简略 | 仅描述算法名称，未说明原理·机制·参数 | 补充详细的技术说明，可请技术团队协助撰写 |
| 训练数据来源未说明 | 未说明训练数据的来源·类型·规模 | 补充数据来源说明，确保来源合法性 |
| 安全管理制度未正式发布 | 制度仅为草案，未经企业正式审批发布 | 完成制度审批流程，取得正式发布的制度文件 |
| 自评估报告维度不全 | 缺少某些评估维度（如可解释性评估） | 补充缺失的评估维度，确保覆盖全部要求 |
| 拟公示内容不符合要求 | 公示内容过于技术化，用户难以理解 | 以用户可理解的方式重写公示内容 |
| 各材料信息不一致 | 算法类型·服务描述在不同材料中不一致 | 统一各材料中的信息描述，确保前后一致 |
| 用户权益保护方案缺少操作细节 | 仅有原则性描述，缺少具体操作流程与响应时限 | 补充具体操作路径·响应时限·处理流程 |
| 涉及商业秘密的技术信息 | 算法核心参数涉及商业秘密 | 在不影响备案审查的前提下做概括描述，保留必要的技术说明 |

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 涉及商业秘密 | 在不影响备案审查的前提下做概括描述，保留必要的技术说明；标注 `[商业秘密 — 已做脱敏处理]` |
| 材料无法准备 | 标注缺失原因，给出补充建议与时间表 |
| 服务尚未上线 | 可先行准备备案材料，在服务上线前完成备案 |
| 多算法同时备案 | 为每个算法分别准备材料，主体信息可共用 |
| 备案系统要求变更 | 跟踪备案系统最新要求，及时调整材料格式与内容 |
| 算法涉及多个备案主体 | 明确各主体的备案义务与材料分工，避免重复或遗漏 |

## 错误处理

| 错误/敏感情形 | 处理方式 |
|------------|---------|
| 虚假备案需求 | 拒绝，标 `[合规边界 — 拒绝]` |
| 行刑衔接 | 标 `[强制升级 — 行刑衔接]`，联系执业律师 |
| 信息不足 | 列出需补充的信息清单，标注 `[待确认]` |
| 临时模式 | 标 `[待确认 — 临时模式]`，在输出末尾列出待确认事项 |
