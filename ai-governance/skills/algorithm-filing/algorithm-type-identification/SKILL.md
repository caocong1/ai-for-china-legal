# 算法类型识别与分类（algorithm-type-identification）

> **前置**：
> 1. 读取 `ai-governance/CLAUDE.md`，加载「所属行业」「是否提供AI相关服务」「AI服务类型描述」。
> 2. 读取 `ai-governance/skills/_shared/legal-basis-conventions.md`。
> 3. 读取 `ai-governance/skills/_shared/ai-law-citations.md`（§D.2 算法推荐管理规定）。
> 4. 本子技能负责「算法推荐服务类型识别·算法分类·备案义务初步判断」。

---

## 目的

识别企业当前或计划提供的算法推荐服务类型，按算法推荐管理规定分类确定适用的备案要求，输出「AI 服务—算法类型—备案义务」映射表。

具体目标：

1. **算法推荐服务识别**：根据企业 AI 服务描述，判断是否构成算法推荐服务
2. **算法类型分类**：按算法推荐管理规定五大类型逐一对照分类
3. **备案义务判断**：根据算法类型与服务性质确定是否需要完成算法备案
4. **交叉备案义务识别**：判断是否同时涉及深度合成备案或生成式 AI 备案义务
5. **特殊合规义务识别**：识别算法歧视·舆论属性·大数据杀熟等特殊合规风险

---

## 详细步骤

### 步骤 1：算法推荐服务定义对照

根据互联网信息服务算法推荐管理规定，算法推荐服务指利用生成合成类、个性化推送类、排序精选类、检索过滤类等算法技术向用户提供信息服务。

**算法推荐服务识别要素**：

| 要素 | 说明 | 判断标准 |
|------|------|---------|
| 技术基础 | 是否利用算法技术处理信息 | 是/否 |
| 算法类型 | 属于何种算法推荐类型 | 生成合成类/个性化推送类/排序精选类/检索过滤类/调度决策类/混合 |
| 服务性质 | 是否向用户提供互联网信息服务 | 是/否 |
| 用户规模 | 用户数量是否达到备案门槛 | 须核实具体门槛 `[待验证]` |
| 舆论属性 | 是否具有舆论属性或社会动员能力 | 是/否/待评估 |

### 步骤 2：算法类型详细分类

**算法推荐类型详细分类表**（含监管要点）：

| 算法类型 | 定义描述 | 典型场景 | 典型行业 | 监管要点 |
|---------|---------|---------|---------|---------|
| **生成合成类** | 利用算法生成或合成文本·图像·音频·视频等内容 | 文本生成·图像生成·语音合成·视频生成·虚拟数字人 | 大语言模型·AIGC·虚拟主播 | 可能同时触发深度合成备案与生成式AI备案 `[待验证]`；须关注内容标识义务 |
| **个性化推送类** | 利用算法根据用户特征进行个性化信息推送 | 信息流推荐·商品推荐·内容推荐·广告定向推送·短视频推荐 | 新闻·电商·短视频·社交·音乐 | 须保障用户选择权（关闭推荐选项）`[待验证]`；须防范大数据杀熟；须防范算法推荐歧视 |
| **排序精选类** | 利用算法对信息进行排序·精选·推荐 | 搜索结果排序·内容排序·热搜榜单·推荐列表·排行榜 | 搜索引擎·电商平台·应用商店·内容平台 | 须防范搜索结果操纵·热搜人为干预 `[待验证]`；须保障排序透明度 |
| **检索过滤类** | 利用算法进行信息检索·内容过滤·关键词审核 | 信息检索·内容过滤·关键词审核·敏感词过滤·信息分类 | 搜索引擎·内容平台·应用商店 | 须保障检索结果的客观性·公正性 `[待验证]`；须关注内容过滤标准的合理性 |
| **调度决策类** | 利用算法进行任务调度·定价决策·资源分配 | 网约车派单·外卖配送调度·动态定价·资源调度·信用评估 | 网约车·外卖·物流·共享经济·金融科技 | 须防范算法价格歧视（大数据杀熟）`[待验证]`；须保障劳动者权益；须关注动态定价公平性 |

### 步骤 3：算法类型分类决策树

```
开始：企业是否利用算法技术向用户提供互联网信息服务？
├── 否 → 不适用算法推荐管理规定，退出
└── 是 → 继续判断
    │
    ├─ 算法是否主要用于生成或合成内容（文本/图像/音频/视频）？
    │   └── 是 → 归类为【生成合成类】
    │       └── 附加判断：是否涉及深度合成？→ 是 → 交叉触发深度合成备案
    │       └── 附加判断：是否向公众提供生成式AI服务？→ 是 → 交叉触发生成式AI备案
    │
    ├─ 算法是否主要用于根据用户特征推送个性化信息？
    │   └── 是 → 归类为【个性化推送类】
    │       └── 附加判断：是否涉及商品/服务定价？→ 是 → 须排查大数据杀熟风险
    │
    ├─ 算法是否主要用于对信息进行排序·精选·推荐？
    │   └── 是 → 归类为【排序精选类】
    │       └── 附加判断：是否涉及热搜·榜单？→ 是 → 须关注舆论属性评估
    │
    ├─ 算法是否主要用于信息检索·内容过滤？
    │   └── 是 → 归类为【检索过滤类】
    │
    ├─ 算法是否主要用于任务调度·定价·资源分配？
    │   └── 是 → 归类为【调度决策类】
    │       └── 附加判断：是否涉及动态定价？→ 是 → 须排查价格歧视风险
    │
    └── 以上均不完全匹配 → 标注【新型算法 — 监管规则尚不明确】
        └── 建议：关注监管动态，必要时咨询监管机关
```

**多算法系统处理规则**：当企业算法系统同时涉及多种类型时，须逐一列示每种类型，分别评估其备案义务与合规要求。例如，一个电商平台可能同时使用个性化推送类（商品推荐）、排序精选类（搜索排序）、调度决策类（配送调度）三种算法。

**算法类型分类常见误区**：

| 误区 | 正确理解 |
|------|---------|
| 「我们的算法只是搜索，不需要备案」 | 搜索排序属于排序精选类，信息检索属于检索过滤类，均可能触发备案义务 `[待验证]` |
| 「我们的算法只做内部推荐，不涉及公众」 | 须评估是否构成「向用户提供互联网信息服务」，内部员工使用与企业向用户提供服务有区别 `[待验证]` |
| 「我们的算法是第三方提供的，不需要我们备案」 | 算法推荐服务提供者（即实际提供服务的企业）承担备案义务，算法技术提供方与使用方的义务分配须具体分析 `[待验证]` |
| 「我们只是电商平台，算法是商家自己设置的」 | 平台对平台内算法推荐服务可能承担管理责任或备案义务 `[待验证]` |

### 步骤 4：备案义务判断

| 条件 | 备案义务 | 备注 |
|------|---------|------|
| 提供算法推荐服务 | 须完成算法备案 `[待验证]` | 通过互联网信息服务算法备案系统 |
| 具有舆论属性或社会动员能力 | 须在备案前完成安全评估 `[待验证]` | 门槛须核实 |
| 仅内部使用的算法 | 可能不适用备案 `[待验证]` | 须评估是否构成互联网信息服务 |
| 算法涉及深度合成技术 | 可能同时触发深度合成备案 `[待验证]` | 须分别评估 |
| 向公众提供生成式AI服务 | 可能同时触发生成式AI备案 `[待验证]` | 须分别评估 |

### 步骤 5：交叉义务识别

| 交叉义务 | 触发条件 | 衔接技能 |
|---------|---------|---------|
| 深度合成备案 | 算法涉及深度合成技术（人脸生成/替换·语音合成/克隆·图像生成/编辑·视频生成/编辑） | `/ai-governance:deep-synthesis-filing` |
| 生成式AI备案 | 同时向公众提供生成式AI服务（文本·图像·音频·视频生成） | `/ai-governance:generative-ai` |
| 数据安全评估 | 处理重要数据或个人信息 | 与 data-compliance 插件衔接 |
| 个人信息保护影响评估 | 利用个人信息进行自动化决策 | 与 data-compliance 插件衔接 |

### 步骤 6：特殊合规风险识别

| 风险类型 | 触发场景 | 合规要求 |
|---------|---------|---------|
| **算法歧视** | 算法基于用户特征实施不合理差别待遇 | 须确保算法决策公平公正 `[待验证]` |
| **大数据杀熟** | 个性化推送类或调度决策类算法对相同交易条件实施差别定价 | 须防范算法价格歧视 `[待验证]`；与消费者权益保护法衔接 |
| **舆论属性风险** | 算法推荐服务具有舆论属性或社会动员能力 | 须在备案前完成安全评估 `[待验证]` |
| **未成年人保护** | 算法推荐服务面向未成年人 | 须满足未成年人保护特殊要求 `[待验证]` |
| **劳动者权益** | 调度决策类算法涉及平台劳动者（外卖骑手·网约车司机等） | 须保障劳动者知情权·休息权 `[待验证]` |
| **算法透明度缺失** | 未以显著方式告知用户算法推荐服务情况·未公示算法基本原理 | 须满足算法透明度义务 `[待验证]`；须在产品界面提供算法说明 |
| **用户选择权受限** | 未提供关闭算法推荐服务选项·关闭操作过于复杂 | 须保障用户选择权 `[待验证]`；关闭操作须便捷有效 |

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 算法类型不明确（新型 AI 应用） | 标注 `[新型算法 — 监管规则尚不明确]`，建议关注监管动态 |
| 同时涉及多种算法类型 | 逐一列示每种算法类型，分别评估备案义务与合规要求 |
| 算法更新导致类型变化 | 须重新进行算法类型识别，评估是否影响备案义务 |
| 跨平台使用同一算法 | 评估各平台是否分别构成独立的算法推荐服务，可能须分别备案 `[待验证]` |
| 算法仅用于内部决策 | 评估是否构成「向用户提供互联网信息服务」，可能不适用但须持续关注 `[待验证]` |
| 算法由第三方提供 | 评估企业是否为算法推荐服务提供者（算法使用方 vs 算法开发方的备案义务分配）`[待验证]` |
| 企业为算法推荐服务提供技术支持 | 评估是否构成算法推荐服务提供者或技术服务提供者，备案义务可能不同 `[待验证]` |
| 算法推荐服务面向境外用户 | 评估是否属于「在中国境内」提供算法推荐服务 `[待验证]` |
| 算法推荐服务通过 API 间接提供 | 评估 API 提供方与 API 调用方各自的备案义务 `[待验证]` |
| 算法推荐服务处于测试阶段 | 评估是否已构成「提供服务」，测试阶段可能已有备案义务 `[待验证]` |
