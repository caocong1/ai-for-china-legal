# 深度合成服务识别与分类（deep-synthesis-scope-identification）

> **前置**：
> 1. 读取 `ai-governance/CLAUDE.md`，加载「所属行业」「是否提供AI相关服务」「AI服务类型描述」「合规严格度」「监管关系」。
> 2. 读取 `ai-governance/skills/_shared/legal-basis-conventions.md`。
> 3. 读取 `ai-governance/skills/_shared/ai-law-citations.md`（§D.1 深度合成管理规定 / §D.2 算法推荐管理规定 / §D.3 生成式AI暂行办法）。
> 4. 本子技能负责「是否属于深度合成服务·适用何种备案要求」。

---

## 目的

识别企业当前或计划提供的 AI 服务是否属于深度合成服务范畴，按深度合成管理规定分类确定适用的备案要求，输出「AI 服务—深度合成分类—备案义务」映射表。

具体目标：

1. **深度合成服务识别**：根据企业 AI 服务描述，判断是否属于深度合成服务
2. **服务细分类型分类**：按深度合成管理规定及技术特征进行细分分类（人脸替换/语音克隆/虚拟数字人/图像生成/视频合成 等）
3. **备案义务判断**：根据服务分类确定是否需要进行深度合成备案，以及是否同时涉及其他备案义务
4. **风险场景初筛**：识别服务是否涉及高风险场景（换脸欺诈·深度伪造·未成年人保护等）
5. **与其他备案义务的交叉识别**：判断是否同时涉及算法备案或生成式 AI 备案义务
6. **备案路径建议**：根据识别结果给出备案路径与时间建议

---

## 信息收集要点

在执行识别与分类前，须收集以下关键信息。如用户未提供，须主动询问：

| 信息类别 | 须收集的信息 | 用途 |
|---------|------------|------|
| **服务描述** | AI服务的具体功能·技术实现方式·用户交互流程 | 判断是否属于深度合成服务 |
| **技术细节** | 使用的AI技术类型（GAN/扩散模型/Transformer等）·模型架构·输入输出类型 | 确定深度合成细分类型 |
| **业务模式** | ToC/ToB/特定群体·服务覆盖范围·预计用户规模 | 判断是否构成「向公众提供」 |
| **数据流向** | 用户输入数据类型·是否处理人脸/声纹等生物识别信息·数据存储方式 | 评估敏感个人信息处理风险 |
| **服务状态** | 已上线/计划上线/研发中·计划上线日期 | 确定备案时间要求 |
| **现有备案** | 是否已完成算法备案/生成式AI备案/其他备案 | 识别交叉义务与可复用材料 |
| **第三方依赖** | 是否使用第三方AI服务/SDK/API·第三方是否已备案 | 识别备案义务主体 |

## 详细步骤

### 步骤 1：深度合成服务定义对照

根据互联网信息服务深度合成管理规定，深度合成服务指利用深度学习、虚拟现实等生成或编辑文本、图像、音频、视频等内容的技术。

**深度合成服务识别要素**：

| 要素 | 说明 | 判断标准 |
|------|------|---------|
| 技术基础 | 是否利用深度学习、虚拟现实等 AI 技术 | 是/否/待确认 |
| 内容类型 | 是否涉及文本/图像/音频/视频的生成或编辑 | 文本/图像/音频/视频/多类型/否 |
| 功能性质 | 是否属于内容生成或内容编辑 | 生成/编辑/两者皆有/否 |
| 服务性质 | 是否向公众提供互联网信息服务 | 是/否（内部使用可能不适用） |
| 交互方式 | 用户是否可输入素材并获取合成结果 | 用户驱动/系统自动生成/混合 |
| 输出形态 | 合成结果是否为可传播的内容形态 | 图片/音频/视频/文本/多模态 |

### 步骤 2：深度合成服务细分类型对照

根据技术特征与应用场景，将深度合成服务细分为以下类型：

| 细分类型 | 技术特征 | 典型应用场景 | 是否属深度合成 | 备案义务 |
|---------|---------|------------|-------------|---------|
| **人脸生成** | 基于GAN/扩散模型生成不存在的人脸图像 | 虚拟头像生成、AI写真 | 是 `[待验证]` | 须备案 `[待验证]` |
| **人脸替换（换脸）** | 将一张人脸替换到另一图像/视频中 | AI换脸应用、影视后期、虚拟试妆 | 是 `[待验证]` | 须备案 `[待验证]`；高风险场景 |
| **人脸编辑** | 对人脸属性进行编辑（美颜/老化/表情迁移） | 美颜相机、人脸特效 | 是 `[待验证]` | 须备案 `[待验证]` |
| **语音合成（TTS）** | 文本转语音、语音风格迁移 | 有声读物、智能客服语音 | 是 `[待验证]` | 须备案 `[待验证]` |
| **语音克隆** | 基于少量样本克隆特定人声音 | 声音复刻、个性化语音助手 | 是 `[待验证]` | 须备案 `[待验证]`；高风险场景 |
| **图像生成** | 文本/图像驱动的图像生成 | AI绘图、图像风格转换、文生图 | 是 `[待验证]` | 须备案 `[待验证]` |
| **图像编辑** | 图像修复/扩展/风格迁移/目标移除 | 图片后期处理、电商图片生成 | 是 `[待验证]` | 须备案 `[待验证]` |
| **视频生成** | 文本/图像驱动的视频生成 | AI视频创作、文生视频 | 是 `[待验证]` | 须备案 `[待验证]` |
| **视频编辑** | 视频风格迁移/特效/剪辑增强 | 视频后期制作工具 | 是 `[待验证]` | 须备案 `[待验证]` |
| **虚拟数字人** | 综合人脸/语音/动作驱动的虚拟形象 | 虚拟主播、数字人客服、虚拟偶像 | 是 `[待验证]` | 须备案 `[待验证]` |
| **虚拟主播** | 实时驱动的虚拟形象直播 | 电商直播、新闻播报 | 是 `[待验证]` | 须备案 `[待验证]`；可能同时触发算法备案 |
| **数字人交互** | 具备对话能力的交互式数字人 | 数字人导游、数字人教师 | 是 `[待验证]` | 须备案 `[待验证]`；可能同时触发生成式AI备案 |
| **文本生成** | 大语言模型驱动的文本内容生成 | 智能写作、对话机器人 | 需进一步判断是否归入生成式AI `[待验证]` | 可能适用生成式AI备案 `[待验证]` |
| **多模态合成** | 跨模态（文-图-音-视频）联合生成 | 多模态AI助手 | 是 `[待验证]` | 须备案 `[待验证]`；可能同时触发多项备案 |

### 步骤 3：服务分类与备案义务判断

根据服务类型确定适用的备案要求：

| 服务分类 | 适用规章 | 备案义务 | 备注 |
|---------|---------|---------|------|
| 深度合成服务（图像/音频/视频生成编辑） | 深度合成管理规定 | 须完成深度合成备案 `[待验证]` | 通过算法备案系统备案 |
| 深度合成 + 生成式AI | 深度合成管理规定 + 生成式AI暂行办法 | 可能同时适用两项备案 `[待验证]` | 须分别评估 |
| 仅文本生成（大语言模型） | 生成式AI暂行办法 | 可能适用生成式AI备案 `[待验证]` | 须进一步评估 |
| 内部使用（不向公众提供） | 视具体情况 | 可能不适用备案义务 `[待验证]` | 须评估是否构成「向公众提供」 |
| 深度合成 + 算法推荐 | 深度合成管理规定 + 算法推荐管理规定 | 可能同时适用两项备案 `[待验证]` | 虚拟主播含个性化推送时触发 |

### 步骤 4：风险场景初筛

识别深度合成服务是否涉及以下高风险场景：

| 风险场景 | 识别标志 | 风险等级 | 处置建议 |
|---------|---------|---------|---------|
| 换脸服务可能被用于欺诈 | 提供人脸替换功能且缺乏身份核验 | 🔴 关键 | 标 `[高风险 — 行刑衔接审查]`，评估帮助信息网络犯罪活动罪风险 |
| 语音克隆可能被用于电信诈骗 | 提供声音克隆功能且缺乏声音所有者授权机制 | 🔴 关键 | 标 `[高风险 — 行刑衔接审查]`，评估声音权保护与诈骗风险 |
| 服务面向未成年人 | 主要用户群体为未成年人或内容面向未成年人 | 🟠 高 | 评估未成年人保护特别义务 |
| 涉及敏感个人信息处理 | 处理人脸信息·声纹信息等生物识别信息 | 🟠 高 | 评估个人信息保护法敏感个人信息规则（单独同意·PIA） |
| 跨境提供服务 | 向境外用户提供深度合成服务或数据出境 | 🟠 高 | 评估数据出境安全评估义务 |
| 生成政治敏感内容风险 | 服务可能被用于生成虚假新闻·政治谣言 | 🟠 高 | 评估内容安全管理制度充分性 |
| 涉及医疗/法律/金融等专业领域 | 深度合成内容涉及专业领域建议 | 🟡 中等 | 评估专业领域合规要求 |

### 步骤 5：交叉义务识别

判断企业是否同时涉及其他 AI 备案义务：

| 交叉义务 | 触发条件 | 衔接技能 |
|---------|---------|---------|
| 算法备案 | 使用算法推荐技术提供信息服务（如虚拟主播含个性化推送） | `/ai-governance:algorithm-filing` |
| 生成式AI备案 | 向公众提供生成式AI服务（如数字人交互含对话生成） | `/ai-governance:generative-ai` |
| 数据安全评估 | 处理重要数据或大规模个人信息 | 与 data-compliance 插件衔接 |
| 个人信息保护影响评估 | 处理人脸·声纹等敏感个人信息 | 与 data-compliance 插件衔接 |
| 网络安全等级保护 | 运营互联网信息服务 | 评估等保定级义务 |

### 步骤 6：常见识别误区与纠偏

在识别过程中，以下误区较为常见，须特别注意纠偏：

| 误区 | 正确理解 | 纠偏建议 |
|------|---------|---------|
| 「我们的AI只是简单的图片滤镜，不算深度合成」 | 深度合成的定义不取决于技术复杂度，而取决于是否利用深度学习等技术生成或编辑内容；即使看似简单的滤镜，如果底层使用深度学习模型，仍可能属于深度合成 | 审查技术实现底层，不只看用户界面表现 |
| 「我们只做ToB，不需要备案」 | 备案义务的触发条件是「提供深度合成服务」，而非仅面向公众（ToC）；向企业客户提供深度合成API/SDK也可能触发备案义务 | 评估服务对象是否构成「向公众提供互联网信息服务」 |
| 「我们用的是开源模型，不算我们提供的服务」 | 基于开源模型部署并向用户提供深度合成服务，部署方可能构成「深度合成服务提供者」 | 识别服务链条中各方法律角色，部署方通常须备案 |
| 「文本生成不算深度合成」 | 深度合成管理规定覆盖文本·图像·音频·视频等内容类型；纯文本生成可能更直接适用生成式AI暂行办法，但须具体评估 | 区分纯文本生成（可能适用生成式AI备案）与多模态生成（可能同时适用深度合成备案） |
| 「我们已经做了算法备案，不需要再做深度合成备案」 | 算法备案与深度合成备案是不同的备案义务，可能同时适用；须分别评估 | 分别评估算法备案义务与深度合成备案义务，不可互相替代 |
| 「我们的服务还在测试阶段，不需要备案」 | 如测试阶段已向公众开放（如公测·内测版向外部用户开放），可能已触发备案义务 | 评估测试阶段是否构成向公众提供服务 |

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 服务类型不明确（新型 AI 应用） | 标注 `[新型服务 — 监管规则尚不明确]`，建议关注监管动态，从严评估备案义务 |
| 同时涉及深度合成与生成式AI | 分别评估，可能须同时完成两项备案；建议先完成深度合成备案评估再评估生成式AI备案 |
| 仅内部使用不向公众提供 | 评估是否构成「向公众提供」——如通过API向第三方提供、嵌入小程序/公众号等可能构成向公众提供；纯内部研发工具可能不适用但须持续关注 |
| 企业使用第三方深度合成服务（非自研） | 评估企业是否属于「深度合成服务提供者」——如基于第三方SDK/API向用户提供深度合成功能，企业本身可能构成服务提供者 |
| 深度合成技术用于学术研究 | 学术研究场景是否适用备案义务须具体评估——如研究成果通过互联网向公众提供服务，可能仍须备案 `[待验证]` |
| 开源模型发布 | 开源模型发布是否构成「提供深度合成服务」须评估——模型发布本身与服务提供可能存在区别 `[待验证]` |
| 服务涉及多主体合作 | 多方合作提供深度合成服务时，须识别各方法律角色（服务提供者·技术支持方·平台方），确定备案义务主体 |
| 深度合成服务已上线但未备案 | 标 `[合规缺口 — 未备案]`，评估风险等级，建议尽快启动备案程序；如已被监管机关通知整改，升级为被动应对模式 |

## 错误处理

| 错误/敏感情形 | 处理方式 |
