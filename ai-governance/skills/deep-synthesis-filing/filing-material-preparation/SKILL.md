# 备案材料准备（filing-material-preparation）

> **前置**：
> 1. 读取 `ai-governance/CLAUDE.md`，加载服务立场与合规严格度。
> 2. 读取 `ai-governance/skills/_shared/ai-law-citations.md`（§D.1 深度合成管理规定）。
> 3. 已完成深度合成服务识别与分类（本子技能的前序子技能或单独触发时由用户提供服务类型信息）。
> 4. 读取 `ai-governance/skills/_shared/legal-basis-conventions.md`。

---

## 目的

按深度合成备案要求，系统准备备案所需材料，包括服务信息说明、安全管理制度、技术保障措施、用户信息保护方案等，确保材料完整且符合备案系统要求。

---

## 详细步骤

### 步骤 1：备案材料完整清单确认

根据深度合成管理规定及备案系统要求，梳理备案所需完整材料清单：

| 序号 | 材料类别 | 材料名称 | 说明 | 备案系统必填 |
|-----|---------|---------|------|------------|
| 1 | **主体信息** | 服务提供者基本信息 | 企业名称·统一社会信用代码·法定代表人·联系方式·服务地址·注册资本 | 是 `[待验证]` |
| 2 | **主体信息** | 运营主体资质证明 | 营业执照副本·增值电信业务经营许可证（如适用）| 是 `[待验证]` |
| 3 | **服务信息** | 深度合成服务说明 | 服务类型·应用场景·服务对象·服务内容描述·服务覆盖范围 | 是 `[待验证]` |
| 4 | **服务信息** | 深度合成算法/技术说明 | 技术基本原理·模型架构·运行机制·主要参数·训练数据概况 | 是 `[待验证]` |
| 5 | **服务信息** | 深度合成服务流程图 | 从用户输入到合成输出的完整流程说明 | 建议提供 `[待验证]` |
| 6 | **管理制度** | 安全管理制度总则 | 安全管理体系架构·责任分工·制度体系概述 | 是 `[待验证]` |
| 7 | **管理制度** | 内容安全管理制度 | 内容审核流程·审核标准·关键词过滤·人工审核机制·审核记录保存 | 是 `[待验证]` |
| 8 | **管理制度** | 数据安全管理制度 | 数据分类分级·存储保护·传输加密·数据生命周期管理·数据销毁 | 是 `[待验证]` |
| 9 | **管理制度** | 用户信息保护制度 | 个人信息收集使用规则·用户权利保障·投诉举报机制 | 是 `[待验证]` |
| 10 | **管理制度** | 深度合成内容标识方案 | 标识方式·标识位置·标识内容·不同场景的标识策略 | 是 `[待验证]` |
| 11 | **技术措施** | 技术保障措施说明 | 内容审核系统·风险监测系统·应急处置系统的技术能力说明 | 是 `[待验证]` |
| 12 | **技术措施** | 安全评估报告 | 安全风险评估范围·方法·结果·控制措施·结论 | 是 `[待验证]` |

### 步骤 2：主体信息材料准备

**服务提供者基本信息**：
- 收集企业营业执照信息（企业名称·统一社会信用代码·法定代表人·注册地址·经营范围）
- 确认服务运营主体与实际运营主体是否一致
- 收集联系方式（备案联系人·联系电话·联系邮箱）
- 如涉及增值电信业务经营许可证（ICP许可证），收集许可证信息

**注意事项**：
- 备案主体须与实际提供深度合成服务的主体一致
- 如集团内多家公司均提供深度合成服务，须分别备案或明确各自主体
- 联系方式须确保备案审核期间可及时联系

### 步骤 3：服务信息材料准备

**深度合成服务说明**：
- 描述深度合成技术类型（人脸生成/人脸替换/语音合成/语音克隆/图像生成/视频生成/虚拟数字人 等）
- 描述具体应用场景与功能（如AI换脸应用·虚拟主播·AI绘图工具 等）
- 说明服务对象（ToC/ToB/特定群体）与预计用户规模
- 说明服务覆盖范围（全国/特定区域/跨境）
- 描述服务上线状态（已上线/计划上线及日期）

**深度合成算法/技术说明**：
- 描述技术基本原理（深度学习模型架构：GAN/扩散模型/Transformer 等）
- 说明算法运行机制（输入-处理-输出流程）
- 描述主要技术参数（模型规模·精度指标·推理速度等，不涉及核心商业秘密的部分）
- 描述训练数据概况（数据来源类型·数据规模·数据处理方式），注意不涉及商业秘密
- 描述技术更新迭代计划

**深度合成服务流程图**：
- 绘制从用户输入到合成输出的完整流程图
- 标注各环节的数据处理节点
- 标注内容审核节点与人工干预节点

### 步骤 4：管理制度材料准备

**安全管理制度总则**：
- 建立安全管理组织架构（安全负责人·安全管理部门·各岗位职责）
- 制定安全管理制度体系框架
- 明确安全管理制度的制定·审批·发布·修订流程

**内容安全管理制度**：
- 内容审核标准：明确禁止生成的内容类型（违法违规·暴力恐怖·色情低俗·虚假信息·侵犯他人合法权益 等）
- 内容审核流程：机器初审（关键词过滤·图像识别·语义分析）→ 人工复审 → 终审
- 审核记录保存：审核日志保存期限与方式
- 违规内容处置：发现违规内容后的处置流程（删除·屏蔽·报告）

**数据安全管理制度**：
- 数据分类分级：按数据敏感程度分级（公开·内部·敏感·高度敏感）
- 数据存储保护：加密存储·访问控制·备份策略
- 数据传输安全：传输加密·接口安全
- 数据生命周期管理：收集·存储·使用·共享·销毁各环节管理
- 特别关注：人脸信息·声纹信息等生物识别信息的特殊保护要求

**用户信息保护制度**：
- 个人信息收集使用规则：收集范围·使用目的·存储期限
- 敏感个人信息处理规则：单独同意机制·个人信息保护影响评估
- 用户权利保障：查询·更正·删除·撤回同意·注销账户的渠道与流程

**深度合成内容标识方案**：
- 标识方式：可见水印·文字提示·元数据标识（如C2PA标准）·隐写标识
- 标识位置：生成内容显著位置（如图片角落·视频片头片尾·音频开头）
- 标识内容：「本内容由 AI 深度合成技术生成」或类似明确表述
- 分场景标识策略：不同服务类型（换脸·语音合成·图像生成 等）的差异化标识方案
- 标识不可轻易移除的技术措施

### 步骤 5：技术措施材料准备

**技术保障措施说明**：
- 内容审核系统：
  - 自动审核能力：文本审核（敏感词库·语义分析）·图像审核（违规内容识别）·音频审核·视频审核
  - 人工审核能力：审核团队规模·审核培训·审核质量管控
  - 审核覆盖率与准确率指标
- 风险监测系统：
  - 异常行为监测：批量生成·高频调用·异常用户行为识别
  - 违规内容识别：实时监测·定期巡查
  - 深度合成内容检测能力：是否具备检测深度合成内容真伪的技术能力
- 应急处置机制：
  - 违规内容处置流程：发现→评估→处置→报告→记录
  - 安全事件应急响应：响应时间·处置流程·上报机制
  - 用户投诉处理：投诉渠道·响应时限·处理流程·结果反馈
  - 监管配合机制：配合监管机关查询·处置的工作流程

**安全评估报告**（详见 security-assessment-filing 子技能）：
- 安全风险评估范围与方法
- 风险识别与评估结果
- 风险控制措施
- 评估结论

### 步骤 6：用户保护材料准备

**用户协议更新**：
- 增加深度合成服务专项条款：告知用户使用深度合成技术的情况
- 明确用户不得将深度合成内容用于违法违规用途
- 明确用户对深度合成内容的使用限制
- 约定深度合成内容的知识产权归属（如适用）

**隐私政策更新**：
- 增加深度合成相关个人信息处理告知（处理类型·目的·方式·存储期限）
- 如涉及人脸信息·声纹信息等生物识别信息，须增加敏感个人信息处理专项告知
- 说明是否使用用户输入数据进行模型训练
- 用户撤回同意与数据删除的机制

### 步骤 7：材料完整性自查与质量检查

## 常见错误与避免方法

| 常见错误 | 后果 | 避免方法 |
|---------|------|---------|
| 算法说明过于简略 | 备案审核退回或要求补充材料 | 算法说明应覆盖原理·架构·运行机制·参数·训练数据概况，在保护商业秘密的前提下尽量详细 |
| 内容标识方案不明确 | 被认定未履行标识义务 | 标识方案须具体到标识方式·位置·内容·不同场景策略，并提供示例截图 |
| 安全管理制度仅有框架无细则 | 被认定安全管理制度不健全 | 须有具体可执行的制度文本，而非仅有目录或概述 |
| 用户协议/隐私政策未更新 | 被认定未履行告知义务 | 须在备案前完成用户协议和隐私政策的更新，增加深度合成专项条款 |
| 训练数据说明涉及商业秘密泄露 | 企业核心利益受损 | 在满足备案要求的前提下，对核心算法参数·训练数据具体来源等做适当脱敏处理 |
| 备案主体与实际运营主体不一致 | 备案无效或被要求重新备案 | 确认备案主体与实际提供服务的主体一致，集团内多主体须分别备案 |
| 安全评估报告缺乏实质性内容 | 被认定未完成安全评估 | 安全评估须覆盖内容安全·数据安全·算法安全·用户保护·应急处置五个维度 |
| 材料间信息不一致 | 审核退回或质疑真实性 | 各材料中的服务描述·技术说明·数据规模等信息须保持一致 |

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 部分材料无法准备 | 标注缺失原因，给出补充建议与预计完成时间；区分「可暂缓」与「备案前必须完成」 |
| 涉及商业秘密 | 在材料中做脱敏处理，保留必要的技术说明；标注 `[商业秘密 — 已脱敏]` |
| 服务尚未上线 | 可先行准备备案材料，在服务上线前完成备案；材料中标注「计划上线日期」 |
| 企业为首次备案，无历史材料 | 从零开始准备全部材料，建议预留更长的准备时间 |
| 企业已有算法备案/生成式AI备案 | 评估已有备案材料中可复用的部分（如安全管理制度·技术措施说明），减少重复工作 |
| 多服务须同时备案 | 按服务类型分别准备材料，共享通用部分（主体信息·通用管理制度），差异化部分（服务说明·算法说明·标识方案）分别准备 |
| 材料准备过程中发现合规缺口 | 标注 `[合规缺口 — 须先整改]`，如内容审核机制缺失·用户协议未更新等，须先完成整改再提交备案 |

## 错误处理

| 错误/敏感情形 | 处理方式 |
|------------|---------|
| 虚假备案需求 | 拒绝，标 `[合规边界 — 拒绝]`；如要求伪造安全评估结论·虚报技术措施·编造管理制度即拒绝 |
| 信息不足 | 列出需补充的信息清单，按材料类别分组，标注各项的优先级 |
| 行刑衔接 | 标 `[强制升级 — 行刑衔接]`；如识别到服务涉及换脸用于欺诈等场景 |
| 企业要求跳过某些必要材料 | 标注 `[材料缺失 — 备案可能被退回]`，说明缺失材料的必要性，建议补充 |
| 材料准备时间超过备案时限 | 标 `[时限风险 — 材料准备可能无法在备案截止日前完成]`，建议优先准备核心材料 |
