# AI 治理实践配置文件契约（Practice Profile Schema）

> 本文件定义 ai-governance 插件的实践配置文件：**是什么、在哪、技能怎么读**。
> AI 治理版改编自 regulatory-legal/skills/_shared/practice-profile-schema.md，结合 AI 治理语境特殊化（AI 治理合规立场护栏、不提供规避备案方法护栏、行刑衔接强制升级契约、AI 治理合规边界契约）。

---

## 一、配置文件是什么 / 真相来源

**唯一真相来源：`ai-governance/CLAUDE.md`（散文模板）**

- 本阶段**首次新建**，由 `/ai-governance:cold-start-interview` 冷启动访谈填充和更新
- 律师/合规官/法务可直接编辑该文件，无需重新运行访谈
- 所有 ai-governance 技能从 `ai-governance/CLAUDE.md` 读取实践配置

**架构决策**：沿用 CLAUDE.md 散文模板作为单一真相来源，不引入独立 JSON/YAML 格式。

## 三、技能读取契约

### 3.1 标准读取流程

每个 ai-governance 技能在执行实质性工作前，**必须**按以下顺序读取配置：

1. **读取 `ai-governance/CLAUDE.md`**
2. **检查关键字段是否含 `[PLACEHOLDER]`**：
   - `## 我们是谁` 的「执业角色」「所属行业」「是否提供AI相关服务」字段
   - `## 合规与监管立场` 的「整体服务立场」字段
   - `## 谁在使用` 的「角色」字段
3. **据此选择运行模式**（正常模式 or 临时模式）

### 3.2 临时模式默认值表

| 配置字段 | 临时模式默认值 |
|---------|-------------|
| 整体服务立场 | 按用户任务判断，默认 AI 使用案例分类 |
| 执业角色 | 律师（最高权限） |
| 合规严格度 | 保守型 |
| 所属行业与AI服务类型 | 按用户陈述逐案判断 |
| 监管关系 | 默认主动合规 |

## 四、行刑衔接与强制升级契约

### 4.1 强制升级触发情形

| 情形 | 识别方式 | 输出标记 |
|------|---------|---------|
| 被监管机关责令下架·关停服务 | 监管通知、用户陈述 | `[强制升级 — 重大违规]` |
| 未备案且已被通知整改或处罚 | 监管通知、用户陈述 | `[强制升级 — 重大违规]` |
| 行刑衔接（AI 服务违规可能构成刑事犯罪） | 用户陈述、文书材料 | `[强制升级 — 行刑衔接]` |
| 用户重大权益损害 | 用户陈述、投诉材料 | `[强制升级 — 用户权益损害]` |
| 责任人人身自由风险 | 用户陈述 | `[强制升级 — 人身自由风险]` |

### 4.2 强制升级契约的优先级

**本契约凌驾于服务立场配置之上**，且本插件不替代刑事辩护工作。

## 五、AI 治理合规边界契约

### 5.1 合规边界触发情形

| 需求类型 | 识别关键词 | 处置方式 |
|---------|----------|---------|
| 规避备案义务 | 「不想备案」「怎么绕过备案」 | 拒绝，标 `[合规边界 — 拒绝]` |
| 虚假备案·伪造安全评估 | 「帮忙编数据」「做个假评估」 | 拒绝，标 `[合规边界 — 拒绝]` |
| 绕过内容安全审查 | 「怎么跳过审核」 | 拒绝，标 `[合规边界 — 拒绝]` |
| 隐匿训练数据侵权 | 「不用管版权」 | 拒绝，标 `[合规边界 — 拒绝]` |

### 5.2 合规边界契约的优先级

**本契约不可被任何立场或配置覆盖**。
