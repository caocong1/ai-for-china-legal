# 全面覆盖原则（all-elements-rule）

> 前置读取：`ip-legal/skills/_shared/legal-basis-conventions.md`（引用规范）、`ip-legal/skills/_shared/ip-law-citations.md`（引用库）、`ip-legal/skills/_shared/practice-profile-schema.md`（配置读取契约）、`ip-legal/CLAUDE.md`（实践配置）

---

## 目的

基于 claim-construction 子技能输出的技术特征比对表，运用全面覆盖原则（字面侵权判断方法），逐一核查被诉技术方案是否覆盖权利要求中**每一个**技术特征，得出字面侵权初步结论。

全面覆盖原则是专利侵权判断的第一层分析，字面上相同时直接得出侵权结论；若字面不相同（即被诉方案缺少某技术特征或存在差异），则字面侵权不成立，须进入等同原则分析（doctrine-of-equivalents）。

**本子技能可由律师/IP 经理/专利代理师单独触发：`/ip-legal:all-elements-rule`**

本技能依赖 claim-construction 子技能的输出：

| 前置输入 | 说明 |

| 子技能 | 分工说明 |
|-------|---------|
| claim-construction | 提供权利要求技术特征拆解与被诉方案技术特征提取（本技能前置） |

## 详细步骤

### 第一步：确认分析基础

核对技术特征比对表的完整性：

| 核对项 | 检查要点 |
|-------|---------|
| 独立权利要求来源 | 确认使用的是独立权利要求（非从属权利要求），且已选定保护范围最宽的独立权利要求 |
| 技术特征列表完整 | A、B、C、D……各特征均已列出，无遗漏 |
| 被诉方案特征完整 | 被诉方案中对应特征有明确来源（说明书/样品/公开信息） |
| 不确定特征已标注 | 存在信息不完整的特征已在 claim-construction 中标注待补充 |

### 第二步：技术特征「相同性」逐一判断

对每个权利要求技术特征，判断被诉方案对应特征是否「相同」：

**相同性判断标准（依据权利要求解读规则，描述规则，不写裸条号）**：

> 技术特征相同，是指被诉方案的对应技术特征与权利要求的技术特征**字面上相同**（或技术含义相同），覆盖了权利要求技术特征所记载的全部内容。

**相同性判断分级：**

| 判断结论 | 含义 | 后续处理 |
|---------|------|---------|
| **字面相同** | 技术特征的文字描述或技术含义一致 | 该特征满足全面覆盖，继续判断下一特征 |
| **字面不同（可能等同）** | 文字描述不同，但功能/手段/效果可能相似 | 该特征字面侵权不成立，标注并移交 doctrine-of-equivalents |
| **缺少该特征** | 被诉方案中不存在对应于该权利要求技术特征的技术特征 | 全面覆盖原则下不构成侵权（缺一不可原则），但仍须分析等同 |

**特别注意事项：**

> **「多余指定原则」已在司法实践中被废止（描述规则，不写裸条号）**：不得随意忽略权利要求中的任何一个技术特征，即使认为某特征「多余」或「非必要」，在全面覆盖判断中仍须逐一比对，不能省略。

> **被诉方案增加额外特征的处理**：若被诉方案在覆盖权利要求全部技术特征的基础上，**增加了额外的技术特征**，不影响全面覆盖原则的侵权认定——额外特征的存在不能免除侵权责任（描述规则，不写裸条号）。

### 第三步：全面覆盖判断矩阵（技术特征逐一比对）

制作逐一比对判断表：

| 特征编号 | 权利要求技术特征 | 被诉方案对应特征 | 相同/不同/缺少 | 说明 |
|--------|--------------|--------------|------------|------|
| A | [权利要求特征] | [被诉方案特征] | 字面相同 / 字面不同（转等同分析）/ 缺少 | [说明理由] |
| B | [权利要求特征] | [被诉方案特征] | 字面相同 / 字面不同（转等同分析）/ 缺少 | [说明理由] |
| C | [权利要求特征] | [被诉方案特征] | 字面相同 / 字面不同（转等同分析）/ 缺少 | [说明理由] |
| D | [权利要求特征] | [被诉方案特征] | 字面相同 / 字面不同（转等同分析）/ 缺少 | [说明理由] |

### 第四步：字面侵权结论

依据上表的全部技术特征比对结果，得出字面侵权初步结论：

**全面覆盖原则字面侵权判定规则（来自专利侵权司法解释，描述规则，不写裸条号）**：

> 被诉侵权技术方案的技术特征须覆盖权利要求的**每一个**技术特征（全面覆盖原则/全部技术特征原则）。若被诉方案**缺少权利要求中任一技术特征**，则不落入保护范围，不构成字面侵权。

**字面侵权结论决策树：**

```
全部技术特征均「字面相同」？
├── 是 → 构成字面侵权 → 进入 prior-art-defense（抗辩分析）
└── 否 → 存在字面不相同或缺少的特征 → 字面侵权不成立
         └── 是否有字面不同（可能等同）的特征？
             ├── 是 → 进入 doctrine-of-equivalents（等同原则分析）
             └── 否（有技术特征明确缺少且无等同可能）→ 不侵权结论初步形成
                    → 进入 prior-art-defense（最终综合结论）
```

**字面侵权风险等级：**

| 等级 | 条件 | 说明 |
|-----|------|------|
| 🔴 字面侵权高风险 | 全部技术特征均字面相同 | 字面侵权结论初步成立，须进入抗辩分析 |
| 🟠 字面侵权中风险 | 大部分特征字面相同，少量特征不同但等同可能性高 | 字面侵权不成立，等同侵权风险较高，须进入等同分析 |
| 🟡 侵权风险中等 | 部分特征字面相同，部分特征差异较大，等同可能性待判 | 须深入等同分析，整体侵权结论存在较大不确定性 |
| 🟢 侵权风险低 | 多个关键特征字面不同且等同可能性低；或存在明确缺失特征 | 字面侵权不成立，等同侵权风险也较低，综合侵权可能性低 |

## 检查清单

在完成全面覆盖判断后，按以下清单验证：

- [ ] 确认使用的是独立权利要求（而非从属权利要求）
- [ ] 权利要求每一个技术特征均已被逐一比对（无遗漏）
- [ ] 未使用「多余指定原则」——每个技术特征均参与比对
- [ ] 字面相同、字面不同、缺少三类分别作出明确判断
- [ ] 被诉方案增加额外特征的情形已按「不影响全面覆盖认定」处理
- [ ] 字面侵权结论已明确（构成/不构成），并说明理由
- [ ] 字面不同或缺少的特征已标注「转等同分析（doctrine-of-equivalents）」
- [ ] 风险等级已评定（🔴/🟠/🟡/🟢）
- [ ] 实践配置已读取，维权立场已确认（或临时模式标注）

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 被诉方案增加了额外技术特征 | 不影响全面覆盖判断；覆盖全部权利要求特征即可构成字面侵权，额外特征不构成免责理由 |
| 被诉方案的技术特征信息存疑 | 在结论中明确标注信息局限性，建议通过取样检测或更完整的技术文档补充后重新判断 |
| 外观设计专利 | 不适用本技能的技术特征逐一比对方法；应使用整体视觉效果比对，应由 claim-construction 已切换处理 |
| 方法权利要求中部分步骤顺序争议 | 若权利要求明确规定步骤顺序，被诉方案须以相同或等同顺序实施；若未明确规定，顺序通常不影响覆盖判断（描述规则，不写裸条号） |
| 独立权利要求有并列技术方案 | 逐一分析每个并列技术方案与被诉方案的覆盖关系；任一技术方案被完全覆盖即构成侵权 |
| 从属权利要求是否需要分析 | 独立权利要求构成侵权时，从属权利要求自动一并被侵犯；独立权利要求不侵权时，无需再分析从属权利要求 |

## 错误处理

| 错误情形 | 处理方式 |
|---------|---------|
| 未提供 claim-construction 输出即触发本技能 | 停止判断，要求用户提供权利要求技术特征表与被诉方案技术特征表；或建议先运行 /ip-legal:claim-construction |
| 被诉方案技术特征信息缺失 | 针对缺失特征标注「信息不足，无法判断」，不作出「字面相同」或「缺少」的虚假结论 |
| 从属权利要求被错误地作为分析基础 | 纠正为独立权利要求；说明须先判断独立权利要求 |
| 比对表中有空白行（某技术特征无对应分析） | 要求补充，不得遗漏任何一个权利要求技术特征 |
