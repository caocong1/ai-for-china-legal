# 监控台账与预警

> **与 matter-intake/key-dates-limitation 的分工**：`matter-intake/key-dates-limitation` 在案件录入时完成单次初判和标记。本子技能维护**持续性监控台账**：跨案件聚合、剩余天数动态计算、分级预警和行动建议生成。
>
> **单独触发说明**：本子技能可通过 `/litigation-legal:monitoring-ledger` 单独触发，建立新台账或更新现有台账。

## 目的

建立和维护诉讼时效监控台账，确保律师不因疏漏导致时效届满而丧失胜诉权：

1. **台账建立**：汇总当前事项中所有请求权的时效届满日
2. **动态预警**：根据剩余天数动态触发黄色/红色预警
3. **行动建议**：对临近届满请求权给出具体行动建议
4. **台账更新**：新案件录入、中止中断事由发生、案件解决后更新台账

## 步骤一：建立/更新时效监控台账

### 1.1 台账结构

台账按事项（案件）→ 请求权两级组织：

```
事项：[案件名称/slug]
  └── 请求权一：[名称] | 届满日：YYYY-MM-DD | 剩余[X]天 | 状态：[预警等级]
  └── 请求权二：[名称] | 届满日：YYYY-MM-DD | 剩余[X]天 | 状态：[预警等级]
```

### 1.2 收集台账录入信息

对每个请求权，收集以下字段（从前序子技能加载，或向律师确认）：

| 字段 | 来源 | 说明 |
|------|------|------|
| 请求权名称/描述 | 律师/事项工作区 | 简要描述（如：要求返还货款） |
| 当事人信息 | 事项工作区 | 义务人名称 |
| 案件类型 | 事项工作区 | 合同/侵权/劳动等 |
| 请求权金额（如有）| 律师 | 用于与风险校准阈值联动 |
| 时效期间 | limitation-period-identification | 3年/特殊时效 |
| 起算日 | accrual-determination | 已确认或待确认 |
| 调整后届满日 | tolling-interruption | 含中止/中断修正 |
| 最近中断事由日期 | tolling-interruption | 用于验证台账最新性 |
| 起算点确定性 | accrual-determination | 确定/存疑 |

## 步骤二：计算剩余天数与预警等级

### 2.1 剩余天数计算

```
剩余天数 = 届满日 - 今日
```

计算时使用实际日历天数（含节假日），不用工作日。

### 2.2 预警等级体系

| 预警等级 | 剩余天数 | 颜色标识 | 处理要求 |
|---------|---------|---------|---------|
| **正常** | >180天 | 绿色 ✓ | 常规监控，按季度更新 |
| **关注** | 91–180天 | 蓝色 ℹ | 开始准备中断措施，下季度前行动 |
| **黄色预警** | 31–90天 | 黄色 ⚠ | 需在本月内采取中断时效措施 |
| **红色预警** | 1–30天 | 红色 🔴 | **紧急——立即行动** |
| **极度紧急** | ≤7天 | 深红色 🔴🔴 | **今日或明日必须行动** |
| **起算不确定** | — | 橙色 ❓ | 需紧急确认起算点，不得拖延 |
| **疑似届满** | <0天（估算）| 灰色（时效可能已过）| **紧急 — 立即核查中断事由** |

### 2.3 与风险校准联动

如 `litigation-legal/CLAUDE.md` 已配置重大性阈值，对金额超过阈值的请求权，在对应预警等级后加注：

```
[重大性标记：请求权金额 ¥[X] 超过已披露阈值 ¥[Y]，须同步通知 [升级联系人]]
```

## 步骤三：生成行动建议

### 3.1 行动建议模板

对每个黄色/红色预警请求权，生成具体行动建议：

**情形一：剩余≥30天（黄色预警区间）**

> 建议行动（请二选一或组合）：
>
> (A) **发出书面催款函/要求履行通知**——可产生时效中断效果，使时效从发送日（须有送达证明）重新起算。需准备：函件正文 + 送达证明（快递/公证）。
>
> (B) **提起诉讼或申请仲裁**——提起即发生中断，无需等待判决。如已准备起诉，可在届满日前提起。
>
> (C) **取得义务人书面确认**——如可能，取得义务人签字确认债务，同样产生中断效果。

**情形二：剩余<30天（红色预警区间）**

> **紧急行动（立即处理）**：
>
> 🔴 **优先方案：立即起诉或申请仲裁**——提起诉讼/仲裁申请即产生时效中断，最为稳妥。
>
> 🔴 **备选方案：同日发出公证催款函**——公证送达产生中断效果，效力强于普通快递；但须今日联系公证处安排紧急公证。
>
> ⚠ **注意**：时效届满后义务人可主动提出时效抗辩，法院不主动适用。请务必在届满日前采取行动。

**情形三：起算点不确定**

> **紧急确认起算点**：台账中 [请求权名称] 的起算点尚未确认。在不确定届满日的情况下，建议立即采取保守措施（发出催款函）以保留权利，同时尽快确认起算点。

### 3.2 优先中断方式排序

| 方式 | 中断效力 | 证据强度 | 操作便利 |
|------|---------|---------|---------|
| 提起诉讼/申请仲裁 | 最强 | 最强（立案回执）| 较复杂（准备起诉材料）|
| 公证发函（公证送达）| 强 | 强（公证书）| 中等（需联系公证处）|
| 挂号信/快递函（有签收回执）| 中 | 中（快递单+签收）| 便捷 |
| 电子邮件催告（对方已回复确认）| 中等 | 中等 | 便捷但证明较难 |
| 普通邮件/微信（对方无回复）| 存疑 | 弱 | 最便捷但效力最弱 |

## 步骤四：台账更新机制

### 4.1 触发台账更新的事件

| 触发事件 | 更新内容 |
|---------|---------|
| 新案件录入（matter-intake 完成）| 新增台账行（来自 key-dates-limitation 初判，本子技能完善）|
| 中断事由发生（发函/起诉/对方确认）| 更新届满日（届满日 = 中断日 + 时效期间）|
| 案件已解决（判决/和解/撤案）| 将请求权标记为「已解决」，移出预警列表 |
| 时效期间识别更新（确认特殊时效）| 更新时效期间和届满日 |
| 起算点确认（原先不确定）| 更新起算日和届满日 |

### 4.2 台账写入位置

台账建议写入：

1. 案件事项工作区文件（`matter.md` 或独立台账文件），方便按案件查看
2. 如配置文件指定工作产物位置，写入对应文件夹

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 时效届满（剩余天数<0）且无中断记录 | 标记灰色「疑似届满」，提示紧急核查中断事由；不得定论时效已届满（需律师最终判断）|
| 起算点不确定（多个请求权）| 分别标注每个请求权的确定性；不确定的标橙色并建议立即采取保守措施 |
| 仲裁案件 | 向仲裁机构申请仲裁同样产生中断效力，与诉讼等同处理 |
| 台账覆盖案件过多（如>10个）| 优先显示黄色/红色预警案件，绿色/蓝色案件折叠至附录 |

---

## 错误处理

| 错误情形 | 处理方式 |
|---------|---------|
| 届满日已过但律师刚发现 | 不下「时效已届满」定论；建议立即核查所有中止/中断记录（律师函/对账单/诉讼记录），标注「[疑似届满 — 需紧急核查中断事由]」 |
| 台账与事项工作区数据不一致 | 提示不一致处，列出两个版本，请律师确认哪个版本正确后更新 |
