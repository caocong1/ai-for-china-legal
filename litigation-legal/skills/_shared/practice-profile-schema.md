# 诉讼版实践配置文件契约（Practice Profile Schema）

> 本文件定义 litigation-legal 插件的实践配置文件契约：字段定义、存储位置、技能读取规则，
> 以及与 `litigation-legal/CLAUDE.md` 实践配置模板的映射关系。
>
> **所有诉讼仲裁技能通过本契约读取个性化配置。** 冷启动访谈按本契约填充配置文件。
> 本文件以 `commercial-legal/skills/_shared/practice-profile-schema.md` 为骨架，针对诉讼仲裁语境改编。

---

## 一、配置文件是什么

### 真相来源

实践配置文件的真相来源是 **`litigation-legal/CLAUDE.md`**（散文形式）。该文件：

- 由 `/litigation-legal:cold-start-interview` 冷启动访谈生成和更新
- 可由律师直接编辑（无需重新运行访谈）
- 为所有诉讼仲裁技能提供个性化设置的读取来源

技能从以下路径读取：`~/.claude/plugins/config/ai-for-china-legal/litigation-legal/CLAUDE.md`

### 架构决策

**决策：不引入与现有体系冲突的新配置文件格式。**

背景：Context 中提到配置可为 `.claude-plugin/profile.json` 或类似格式。但现有体系已用 `litigation-legal/CLAUDE.md`（散文模板）作为真相来源，与 `commercial-legal` 体系保持一致。

**选择理由：**

1. `litigation-legal/CLAUDE.md` 已是律师可直接阅读和编辑的人类友好格式，散文胜于 JSON/YAML 用于表达「角色倾向」「和解策略」「风险校准」等判断性立场
2. Claude 的实践配置文件读取本质是文本理解，不需要机器解析的结构化格式
3. 引入 `profile.json` 会产生双重真相来源，维护成本高且容易产生不一致
4. 现有 `matter-intake` 编排入口已从 `litigation-legal/CLAUDE.md` 读取配置，模式已验证可行
5. 与 `commercial-legal` 体系对称，降低跨插件学习成本

**结论：** `litigation-legal/CLAUDE.md` 是唯一真相来源。冷启动访谈实际填充该文件的占位符字段。

## 三、技能读取契约

### 3.1 技能读取时机

**在执行任何实质性工作之前**，每个技能（或其编排入口）必须：

1. 读取 `litigation-legal/CLAUDE.md`（配置文件路径：`~/.claude/plugins/config/ai-for-china-legal/litigation-legal/CLAUDE.md`）
2. 检查关键字段是否已填充（不含 `[PLACEHOLDER]`）
3. 根据检查结果选择运行模式（正常模式 / 临时模式）

### 3.2 配置缺失或含占位符时的处理

当技能检测到 `litigation-legal/CLAUDE.md` 缺失或关键字段含 `[PLACEHOLDER]` 时，**不得静默使用猜测值**，须按以下逻辑处理：

**提示词（标准格式）：**

> 我注意到你还没有配置你的诉讼实践档案——这是让我根据你的角色倾向、风险校准和文书风格来定制输出的关键。
>
> **两个选择：**
> - 运行 `/litigation-legal:cold-start-interview`（约 5 分钟）来配置你的档案，然后我会根据你的真实执业立场进行工作。
> - 说 **「临时模式」**，我会使用通用默认值并在每个输出上标记 `[临时模式 — 配置你的档案以获得定制输出]`。

**临时模式默认值：**

| 字段 | 临时模式默认值 |
|-----|-------------|
| 风险偏好 | 平衡型 |
| 执业角色 | 律师 |
| 管辖法律 | 中国大陆法律 |
| 角色倾向 | 因案而异（直接询问，不预填）|
| 重大性阈值 | 不自动触发，提示律师判断 |
| 和解倾向 | 视情况（不预设立场）|
| 文书风格 | 规范正式，含法条引用 |
| 利益冲突清除方法 | 要求律师在录入前确认 |

### 3.3 角色倾向（原告/被告/两者）立场选择

技能在录入/起草前须确认我方角色，依据 `litigation-legal/CLAUDE.md` → `## 谁在使用` → `角色倾向`：

1. **配置为「原告」**：matter-intake 预填角色为原告，向律师确认；风险分级导向案件价值/代理经济
2. **配置为「被告」**：matter-intake 预填角色为被告，向律师确认；风险分级导向敞口/准备金
3. **配置为「两者—默认X」**：按默认方预填，向律师确认当前案件是否与默认一致
4. **配置为「因案而异」或 `[PLACEHOLDER]`**：直接询问，不预填

**立场确认后**，matter-intake 和下游技能（defense-drafting/representation-drafting）加载对应策略视角。

## 四、重跑 / 编辑 / 版本控制

### 4.1 重跑访谈

律师可随时重新运行完整访谈或任一子技能：

- **完整重跑**：`/litigation-legal:cold-start-interview` — 按顺序填充全部字段
- **部分更新**：单独触发某一子技能，只更新配置文件的对应章节
  - `/litigation-legal:identity-team` — 仅更新「我们是谁/谁在使用/可用集成」
  - `/litigation-legal:risk-calibration` — 仅更新「风险校准」
  - `/litigation-legal:case-management` — 仅更新「利益冲突清除/外部律师合作/证据保全/和解策略」
  - `/litigation-legal:document-style` — 仅更新「文书风格/输出」

### 4.2 直接编辑

律师可直接编辑 `litigation-legal/CLAUDE.md`，无需重跑访谈。编辑后即时生效，下次技能调用自动读取新值。

### 4.3 版本控制

- `litigation-legal/CLAUDE.md` 纳入 git 版本控制（与项目同仓库）
- 支持通过 git 查看历史版本：`git log --oneline litigation-legal/CLAUDE.md`
- 支持通过 git diff 查看变更：`git diff HEAD~1 litigation-legal/CLAUDE.md`
- **回滚方式**：`git checkout <commit-hash> -- litigation-legal/CLAUDE.md`

## 五、法律事实合理性检查

本节复用 `litigation-legal/CLAUDE.md` 的「验证用户陈述的法律事实」与「时效性触发」规则。

### 5.1 适用范围

在访谈/录入将字段写入配置文件**之前**，对以下类型的用户口述内容进行合理性检查：

- 具体法规条款号（如「民诉法第某某条」）
- 具体时效期间（如「我们的案件诉讼时效是一年」）
- 具体阈值金额（如「准备金阈值是¥100万」）
- 引用司法解释条文或地方法院规定

### 5.2 合理性检查规则

**规则 1：条文号核查**

当用户口述条文号时，与训练知识中的该条文内容交叉核对。若发现潜在不一致：

> 你提到「[条文号]」规定「[规则]」；我的理解是该条款规定的是「[另一规则]」。请确认哪个写入配置文件？`[前提 flagged — 需验证]`

**规则 2：时效期间核查**

一般诉讼时效为三年（《民法典》第188条，已核实）。若用户口述特殊时效（如一年、五年），须说明适用的特殊法律：

> 你说这类纠纷诉讼时效是「[X年]」；一般诉讼时效为三年（《民法典》第188条）。如果这是特殊时效，请说明依据的具体法律。`[前提 flagged — 需验证]`

**规则 3：时效性标记**

对于任何涉及具体法规适用的配置字段，若距最后更新日期超过 6 个月，在字段值后附加：

> `[时效性 flagged — 需验证最新法规]`

**规则 4：矛盾回答澄清**

若用户在同一访谈会话中提供矛盾立场（如先说「进取型」后对所有案件都要求立即和解），在写入前澄清：

> 你的风险偏好设置为「进取型」，但和解策略为「早期和解优先」——这两者可能冲突。请确认组合是否准确。

### 5.3 写入前确认流程

对于所有 `[前提 flagged — 需验证]` 标记的字段：

1. 显示检测到的潜在问题
2. 提供两个选项：(a) 按用户口述写入（保留 `[前提 flagged — 需验证]` 标注）；(b) 按建议值修改后写入
3. 不得静默写入未经确认的存疑值
