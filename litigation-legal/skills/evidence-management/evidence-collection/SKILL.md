# 证据收集与保管链

> 引用规范：`litigation-legal/skills/_shared/legal-basis-conventions.md`
> 引用库：`litigation-legal/skills/_shared/civil-procedure-citations.md`（重点：C.4节电子数据 + C.1节举证责任）
> 结构模板：`shared/references/document-structures.md`（证据目录三大类模板）
> 本子技能可被 `litigation-legal/skills/evidence-management/SKILL.md` 编排入口调用，
> 也可通过 `/litigation-legal:evidence-collection` 单独触发。

---

## 目的

系统化收集案件相关证据，建立证据保管链（chain of custody），确保证据的可追溯性和取证合法性，为三性审查和证据目录编制提供基础。

## 步骤一：案件证据需求分析

根据对方诉讼请求和我方答辩路线，分析需要收集的证据类型：

| 证明目标 | 需要的证据类型 | 目前状态 |
|---------|------------|---------|
| 基础交易关系存在 | 合同文本、采购订单、招投标文件 | [已有/缺失/待查] |
| 我方已履行义务 | 付款凭证、验收单、交货记录 | [已有/缺失/待查] |
| 对方违约事实 | 催告函、拒绝履行证据、损失凭证 | [已有/缺失/待查] |
| 损失金额 | 财务凭证、评估报告、发票 | [已有/缺失/待查] |
| 通知/送达情况 | EMS快递回执、当面签收记录 | [已有/缺失/待查] |
| 其他（视案件）| [视情况] | [已有/缺失/待查] |

## 步骤二：证据类型清单与收集指引

### 2.1 书证类

| 证据类型 | 收集重点 | 原件/复印件要求 |
|---------|---------|--------------|
| 合同/协议文本 | 含所有附件、补充协议、变更函 | 须提供原件核对 |
| 往来函件/通知 | 送达凭证（快递单、签收记录）| 原件 + 送达凭证 |
| 财务凭证 | 发票、银行转账记录、付款截图 | 原件或银行加盖印章的流水单 |
| 会议纪要 | 双方签字确认版 | 原件 |
| 法律文件 | 起诉状、判决书、仲裁裁决（如有）| 原件 |

**原件核对标记**：若只有复印件，标记 `[原件待核对 — 复印件证明力较弱，建议提供原件]`。

### 2.2 电子数据类

> **电子数据取证合法性原则**：不得通过非法手段获取（如未经授权入侵系统、窃取他人数据）。非法取得的电子数据合法性存疑，不得提交，否则可能被认定为非法证据 `[待验证]`。

| 电子数据类型 | 保管链要求 | 公证/存证建议 |
|------------|---------|-------------|
| **微信/钉钉聊天记录** | 截图保存（含聊天对象名称+时间戳）+ 手机型号和操作系统版本 | 建议公证（公证效力优先认定）`[待验证]` |
| **电子邮件** | 完整邮件（含邮件头：发件人/收件人/时间/邮件服务器信息）| 建议导出EML格式存档 |
| **短信记录** | 截图 + 手机号信息 | — |
| **网页/平台信息** | 截图（含URL + 访问时间）+ 第三方存证（如可信时间戳）`[待验证]` | 建议可信时间戳或公证 |
| **OA/ERP系统记录** | 系统导出（含系统元数据：操作人/时间/IP）| 须经IT部门出具说明 |
| **录音录像** | 原始文件（不得剪辑）+ 取证时间 + 录制设备 | 如涉及对话录音，须评估合法性 |

**电子数据保管链记录表**（每条电子数据单独填写）：

| 字段 | 内容 |
|------|------|
| 数据名称/描述 | [如：微信聊天记录——关于货款确认] |
| 数据来源 | [如：我方法务手机 iPhoneXX，微信账号XXX] |
| 提取方式 | [截图/手机端直接提取/第三方公证提取] |
| 提取人 | [姓名] |
| 提取时间 | [YYYY-MM-DD HH:MM] |
| 完整性校验 | [如：已截图全部相关对话，无删减] |
| 存证方式 | [本地备份/云端备份/公证/区块链存证] |
| 原始性 | [是否为原始载体数据，还是二次截图] |
| 合法性声明 | [数据来源是否合法，有无侵犯他人合法权益] |

### 2.3 证人证言类

| 证人证言要素 | 说明 |
|------------|------|
| 证人资格 | 了解案件情况的自然人（须与本案无直接利益关系，利益关系须说明 `[待验证]`）|
| 书面证人证言 | 证人本人签字确认，注明日期 |
| 证人出庭 | 建议安排证人出庭接受质询（书面证言证明力弱于出庭 `[待验证]`）|
| 证人的感知来源 | 是亲历（亲眼所见/亲耳所闻）还是转述 |

### 2.4 鉴定意见类

| 鉴定要素 | 说明 |
|---------|------|
| 申请时机 | 须在举证期限内申请，逾期申请不予准许（除非有正当理由 `[待验证]`）|
| 鉴定机构资质 | 须由具有相应资质的鉴定机构出具 `[待验证]` |
| 鉴定事项范围 | 仅鉴定专业技术问题（如损失金额、技术缺陷、文件真实性等）|
| 鉴定人出庭 | 鉴定意见须在庭审中质证，鉴定人须出庭接受质询（除非双方同意不出庭）`[待验证]` |

## 步骤三：证据分类预标记（三大类初步归类）

> **重要**：在收集阶段即按三大类预标记，便于后续 evidence-catalog 子技能编制目录。
> 三大类强制分组规则在 evidence-catalog 子技能中执行，本步骤仅作初步归类。

| 初步归类 | 说明 | 典型证据 |
|---------|------|---------|
| **第一类：基础交易关系类** | 证明基础交易关系的存在和内容 | 合同文本、招投标文件、订单 |
| **第二类：法律关系及索赔事实类** | 证明法律关系有效成立及权利发生 | 催告函及送达凭证、违约通知 |
| **第三类：被告违约事实类** | 证明对方未履行义务的具体事实 | 拒付证据、逾期记录、损失凭证 |

## 步骤四：证据缺口分析与补强方向

| 证明目标 | 现有证据 | 证据缺口 | 建议补强方式 |
|---------|---------|---------|------------|
| [如：基础交易关系] | [如：已有合同] | [如：无补充协议] | [如：向对方调取/申请法院调查取证] |
| [证明目标N] | [现有证据] | [缺口描述] | [补强建议] |

**证据获取途径**：

| 途径 | 适用情形 | 注意事项 |
|------|---------|---------|
| 自行收集 | 我方持有或可合法获取的证据 | 注意取证合法性 |
| 申请法院调查取证 | 我方无法自行收集（如对方持有且拒绝提供）`[待验证]` | 须在举证期限内申请 |
| 证据保全 | 证据有灭失风险或将来难以提取（如工地现场/系统数据）`[待验证]` | 须向法院申请 |
| 申请鉴定 | 需要专业技术鉴定的事项 | 须在举证期限内申请 |
| 向相关机构调取 | 如银行流水（申请法院协助调取）、工商登记信息 | 须通过法院调取令 |

- [ ] 案件证据需求已基于诉讼请求和答辩路线分析

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 电子数据来源存在合法性疑问（如偷录、截取他人通信）| 标记 `[合法性存疑 — 以侵害他人合法权益方式获取的证据可能被认定为非法证据 `[待验证]`]`，不建议提交，提示律师评估 |
| 只有复印件无原件 | 标记 `[原件待核对]`，评估原件核对可能性（联系对方、公证机构等）|
| 证据因灭失风险需紧急保全 | 标记 `[证据保全紧急]`，提示向法院申请诉前/诉中证据保全 |
| 缺少某一大类证据（如无第二类证据）| 不伪造，标记缺类并提出补强方向，提示 evidence-catalog 时会触发质量闸门 |
| 证人与案件有利益关系 | 如实记录利益关系，说明证明力可能受影响 |

## 错误处理

| 错误情形 | 处理方式 |
|---------|---------|
| 律师提供的电子数据无法判断来源合法性 | 要求律师说明取证方式，不自行推断合法性 |
| 证据数量极多，难以逐一分析 | 提示先处理与主要争议焦点直接相关的证据，其余标记「待补充分析」|
| 研究闸门未完成 | 标记 `[研究闸门未完成]`，说明电子数据取证规则以引用库为参考，建议核验 |
