# 主体资格预检与利益冲突门槛

> 引用规范：`litigation-legal/skills/_shared/legal-basis-conventions.md`
> 本子技能可被 `litigation-legal/skills/matter-intake/SKILL.md` 编排入口调用，
> 也可通过 `/litigation-legal:subject-conflict-precheck` 单独触发。

## 目的

在录入任何新案件之前，从两个维度把守硬门槛：

1. **主体资格预检**：识别案件涉及的主体类型是否存在资格限制或程序瑕疵，避免案由识别错误、立案被驳或诉讼请求无法支持。
2. **利益冲突检查**：确保本律所/团队在本案中不存在利益冲突，或确认冲突状态已明确处理并记录在案。

**未完成本子技能检查之前，禁止创建任何案件文件（matter.md、history.md、_log.yaml 条目）。**

---

## 前置读取

执行本子技能前，读取以下文件：

1. `litigation-legal/CLAUDE.md` — 读取「利益冲突清除」章节（清除方法 + 负责人 + 并行录入政策）
2. `litigation-legal/skills/_shared/legal-basis-conventions.md` — 引用规范（来源分类 + 待验证规则）
3. `litigation-legal/skills/_shared/practice-profile-schema.md` — 配置读取契约（冲突门槛字段映射）
4. `shared/references/subject-qualification-traps.md` — 主体资格陷阱清单（全量读取）

## 步骤一：主体资格预检

### 1.1 识别涉案主体类型

收集以下信息：
- **我方当事人**：律所/公司名称、实体类型（有限责任公司/股份公司/律师事务所/政府机关/自然人等）
- **对方当事人**：名称、实体类型
- **其他相关主体**：担保人、保证人、第三方等

### 1.2 特殊主体资格触发检查

对照 `shared/references/subject-qualification-traps.md`，如果案件涉及以下任一主体类型，**必须先核查主体资格**：

| 主体类型 | 核查重点 | 案由影响 |
|---------|---------|---------|
| 融资担保公司/担保公司/地方金融组织 | 是否有权出具独立保函（一般无权） | 独立保函纠纷 → 应为保证合同纠纷 |
| 银行分支机构（无总行授权） | 是否有对外担保权限 | 担保可能无效，需核验授权文件 |
| 保险公司（保证保险）| 性质争议：保险责任 vs 保证责任 | 案由不同：保险纠纷 vs 担保纠纷 |
| 上市公司（担保方）| 是否有董事会/股东会决议 | 无决议 → 担保可能无效 |
| 国企/事业单位（担保方）| 是否有国资委审批 | 程序瑕疵，担保效力待定 |
| 地方政府/政府融资平台 | 担保是否合法（预算法限制）| 担保可能无效 |
| 无行为能力人/限制行为能力人 | 是否有法定代理人 | 合同效力问题，代理权需核验 |

### 1.3 主体资格问题处理规则

- **无主体资格问题**：记录「主体资格预检：无异常」，继续执行。
- **发现主体资格问题**：
  1. 在初步记录中**显著标记**（用 `[主体资格异常 — 需复核]` 标签）
  2. 说明具体问题和对案由/请求权基础的影响
  3. 告知律师须在继续录入前确认调整后的案由
  4. 在后续 `matter.md` 的「待决问题」中保留标记，直到律师确认

## 步骤二：利益冲突检查

### 2.1 依据实践配置文件读取冲突清除策略

从 `litigation-legal/CLAUDE.md` → `## 利益冲突清除` 读取：
- **清除方法**：律所系统 / 人工检查 / 外部律师 / 其他
- **清除负责人**：姓名/团队/律所
- **并行录入政策**：配置文件是否声明「待处理状态下可并行录入」

### 2.2 收集冲突检查信息

请律师提供冲突检查的结果和元数据：

| 字段 | 说明 |
|------|------|
| **检查状态** | 已清除 / 待处理 / 未执行 / 豁免 |
| **检查对象** | 已检查的实体名称（至少包括：对方当事人、已知关联公司、对方代理律师（如已知）、关键证人）|
| **清除方法** | 与配置文件声明的方法一致 |
| **清除人** | 执行检查的人员姓名/团队 |
| **清除日期** | YYYY-MM-DD |

### 2.3 按状态的三路径门槛

**状态：已清除** → 记录元数据，继续。

**状态：待处理** → 在 `matter.md` 和 `_log.yaml` 日志行中**显著标记**冲突未决；仅在实践配置文件声明「并行录入可接受」时允许继续录入；在每次 `/matter-update` 和 `/portfolio-status` 中持续标记，直到解决。

**状态：豁免** → 需要书面豁免文件（记录豁免的存在、签署人、存放位置）。

**状态：未执行** → **停止。这是硬门槛。** 本技能不会创建 matter.md、history.md 或 _log.yaml 条目，直到冲突状态得到解决。三条可接受的路径：

#### 路径 1 — 立即运行冲突检查

暂停本次录入。按照实践配置文件中的利益冲突清除方法进行清除。带着 `状态: 已清除` 或 `状态: 豁免` 返回。

#### 路径 2 — 标记待处理并指定负责人和截止日期

仅在实践配置文件声明并行录入可接受时允许。须捕获：谁在运行冲突检查、预计何时返回、检查的具体实体。录入继续；案件行携带 `conflicts.status: 待处理`；`/portfolio-status` 每次运行都标记；`/matter-update` 重新提示直到解决。

#### 路径 3 — 跳过并记录理由

仅在用户**明确确认跳过**时允许。记录在 `conflicts.override` 中：

```yaml
conflicts:
  status: 未执行
  override:
    by: [用户姓名]
    date: [YYYY-MM-DD]
    rationale: [为什么跳过冲突检查 — 永久记录；不会自动过期]
```

此字段在每次 `/portfolio-status`、每次 `/matter` 简报和每次 `/matter-update` 中**永远可见**，直到被用户通过显式编辑 `_log.yaml` 移除。

> **不要默默进行。** 「我稍后做」不是可接受的回答。必须选择路径 1/2/3 之一，且选择被记录在案。

### 2.4 冲突检查的边界

本子技能**不运行冲突检查本身**——实际清除发生在实践配置文件声明的系统或判断中。本子技能的职责是：

- 记录冲突检查已执行的事实（状态/方法/检查对象/清除人/日期）
- 若未执行：按三路径门槛阻止录入
- 若存在冲突标记：确保它在所有后续工作产物中保持可见

## 检查清单

完成本子技能时，确认以下所有项：

- [ ] 已识别案件所有主体类型（我方 + 对方 + 第三方）
- [ ] 已对照 `subject-qualification-traps.md` 检查特殊主体类型
- [ ] 主体资格问题（如有）已显著标记并告知律师
- [ ] 已从 `litigation-legal/CLAUDE.md` 读取利益冲突清除策略
- [ ] 冲突检查状态已明确：已清除 / 待处理 / 未执行 / 豁免
- [ ] 如果「未执行」：律师已选择路径 1/2/3 并记录在案
- [ ] 如果「已清除」：检查对象列表、清除人、日期已记录

## 边界条件

| 情形 | 处理方式 |
|------|---------|
| 主体资格不明确，无法判断 | 标记 `[主体资格待确认]`，列出待核查信息，允许继续但在 matter.md 中标记为待决问题 |
| 多方当事人各有不同主体问题 | 逐一列出每个主体的问题，不合并处理 |
| 律师坚持跳过主体资格检查 | 记录用户豁免决定（等同于路径 3），在 matter.md「待决问题」中保留标记 |
| 冲突检查刚刚在前一案件完成，本案涉及相同对方 | 不假设延续有效，须确认本案是否单独运行了冲突检查 |
| 配置文件 `## 利益冲突清除` 为 `[PLACEHOLDER]` | 使用临时模式：要求律师手动确认冲突状态；不假设已清除 |

## 错误处理

| 错误情形 | 处理方式 |
|---------|---------|
| 无法读取 `litigation-legal/CLAUDE.md` | 切换临时模式，显示临时模式标记，并告知律师配置文件缺失 |
| 主体类型模糊（如「某某公司」无详细信息）| 询问律师补充主体工商信息，不依据猜测分类 |
| 律师拒绝选择三路径之一 | 明确告知：不选择路径则不能进入下一步，记录对话并等待 |
| `subject-qualification-traps.md` 无法读取 | 列出已知主体风险类型（见上表），提示律师手动核查陷阱清单 |
