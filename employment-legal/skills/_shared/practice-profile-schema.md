# 劳动人事实践配置文件契约（Practice Profile Schema）

> 本文件定义 employment-legal 插件的实践配置文件契约：字段定义、存储位置、技能读取规则，以及与 `employment-legal/CLAUDE.md` 实践配置模板的映射关系。
>
> **所有劳动人事技能通过本契约读取个性化配置。** 冷启动访谈子技能按本契约填充配置文件。

---

## 一、配置文件是什么

### 真相来源

实践配置文件的真相来源是 **`employment-legal/CLAUDE.md`**（散文形式）。该文件：

- 由 `/employment-legal:cold-start-interview` 冷启动访谈生成和更新
- 可由律师/HR 直接编辑（无需重新运行访谈）
- 为所有劳动人事技能提供个性化设置的读取来源

### 架构决策（Discretion 选择）

**决策：不引入与现有体系冲突的新配置文件格式。**

背景：现有体系已用 `employment-legal/CLAUDE.md`（散文模板）作为律师/HR 可读可编辑的真相来源，沿用该格式保持一致性。

**选择理由：**

1. `employment-legal/CLAUDE.md` 已是律师/HR 可直接阅读和编辑的人类友好格式，散文胜于 JSON/YAML 用于表达「审查立场」与「为什么」
2. Claude 的实践配置文件读取本质是文本理解，不需要机器解析的结构化格式
3. 引入新的 `profile.json` 或独立 YAML 文件会产生双重真相来源，维护成本高且容易产生不一致
4. 现有 `hiring-review` 编排入口已从 `employment-legal/CLAUDE.md` 读取配置，模式已验证可行

**关于现有访谈「生成 YAML 格式」的表述纠正：**

现有 `employment-legal/skills/cold-start-interview/SKILL.md` 末尾提到「生成 YAML 格式」的表述，应理解为**填充 `employment-legal/CLAUDE.md` 对应章节的占位符**，而非另存独立 YAML 文件。访谈子技能的产出是更新后的 `employment-legal/CLAUDE.md` 内容，不引入新格式文件。

**结论：** `employment-legal/CLAUDE.md` 是唯一真相来源。

## 三、技能读取契约

### 3.1 技能读取时机

**在执行任何实质性工作之前**，每个技能（或其编排入口）必须：

1. 读取 `employment-legal/CLAUDE.md`
2. 检查关键字段是否已填充（不含 `[PLACEHOLDER]`）
3. 根据检查结果选择运行模式

### 3.2 配置缺失或含占位符时的处理

当技能检测到 `employment-legal/CLAUDE.md` 缺失或含 `[PLACEHOLDER]` 时，**不得静默使用猜测值**，须按以下逻辑处理：

**提示词（标准格式）：**

> 我注意到你还没有配置劳动人事实践档案——这是让我根据你的审查立场（用人单位方/劳动者方）、风险校准、升级矩阵和地方性规定来定制输出的关键。
>
> **两个选择：**
> - 运行 `/employment-legal:cold-start-interview`（约5分钟）来配置你的档案，然后我会根据你的真实审查立场进行工作。
> - 说 **「临时模式」**，我会使用通用默认值——并在每个输出上标记 `[临时模式 — 配置你的档案以获得定制输出]`。

### 3.3 临时模式默认值

| 字段 | 临时模式默认值 |
|-----|-------------|
| 风险偏好 | 平衡型（🟠 高风险问题须说明） |
| 执业角色 | 律师（工作产物标题用「律师工作产物」） |
| 管辖法律 | 中国大陆法律（劳动合同法2012修正版本） |
| 审查立场（服务对象） | 用人单位方（合规与风险预防视角），若文件明显显示另一方则据实调整 |
| 地方性规定 | 按工作地点逐案确认，无默认地区（须询问工作地点后标注） |
| 经济补偿 | 按法定 N 标准（《劳动合同法》第47条），不预设 N+1 或 2N |
| 竞业限制补偿 | 按法定下限（须支付，补偿金额标「临时模式，按双方约定或地方指引确认」） |
| 竞业人员范围 | 限于高管、高技、其他负有保密义务人员（按第23-24条） |

## 四、审查方立场（用人单位方/劳动者方）选择

### 4.1 用人单位方与劳动者方的区别

劳动审查不同于商事合同甲方/乙方——劳动关系中双方利益对立鲜明：

| 立场 | 视角 | 重点 |
|------|------|------|
| **用人单位方** | 合规与风险预防 | 条款是否暴露用人单位于劳动争议风险；如何在合法前提下保护用人单位利益（工时设计、竞业保护、解雇风险控制） |
| **劳动者方** | 权益主张 | 条款是否侵犯劳动者法定权利；哪些条款无效；劳动者可主张的权利和救济路径（二倍工资、违法解除赔偿金、竞业补偿不足） |

### 4.2 立场确认规则

1. **从配置文件读取**：读取 `## 谁在使用` → 服务对象；若含 `[PLACEHOLDER]` 或不明确，执行步骤 2
2. **不明显时询问**：「在本次审查中，你代表的是用人单位方（雇主）还是劳动者方（员工）？」
3. **确认后加载对应视角**：
   - 用人单位方 → 合规修改建议、降低争议风险
   - 劳动者方 → 权利提示、无效条款标记、救济路径

### 4.3 立场确认时机

- **hiring-review 编排入口**：在加载各子技能前确认立场
- **各子技能**：读取立场标注于前置说明，不再重复询问（除非编排入口未传入）

## 五、重跑 / 编辑 / 版本控制

### 5.1 重跑访谈

律师/HR 可随时重新运行完整访谈或任一子技能：

- **完整重跑**：`/employment-legal:cold-start-interview` — 按身份与团队 → 审查立场 → 风险与升级 → 文书风格与地方性规定 顺序填充全部字段
- **部分更新**：单独触发某一子技能，只更新 `employment-legal/CLAUDE.md` 的对应章节（04-03 拆分后可按子访谈触发）

### 5.2 直接编辑

律师/HR 可直接编辑 `employment-legal/CLAUDE.md`，无需重跑访谈。编辑后即时生效，下次技能调用自动读取新值。

### 5.3 版本控制

- `employment-legal/CLAUDE.md` 纳入 git 版本控制（与代码同仓库）
- 支持通过 git 查看历史版本：`git log --oneline employment-legal/CLAUDE.md`
- 支持通过 git diff 查看变更：`git diff HEAD~1 employment-legal/CLAUDE.md`
- **回滚方式**：`git checkout <commit-hash> -- employment-legal/CLAUDE.md`

## 六、法律事实合理性检查

本节定义访谈子技能将字段写入配置文件前的合理性检查规则（复用并适配 employment-legal 劳动法语境）。

### 6.1 适用范围

在访谈子技能将字段写入 `employment-legal/CLAUDE.md` **之前**，对以下类型的用户口述内容进行合理性检查：

- 具体法规条款号（如「第39条是过失辞退吧」）
- 具体金额阈值（如「经济补偿上限是 12 个月月薪吧」）
- 具体期限或比例（如「竞业限制期限 3 年可以吧」「竞业补偿按工资的 10% 吧」）
- 用户口述的合同条款或内部规定标准

### 6.2 合理性检查规则

**规则 1：条文号核查**

当用户口述条文号时，与已核实锚点（见 labor-law-citations.md 锚点汇总）交叉核对。若发现潜在不一致：

> 你提到 X 条款要求 Y；我的理解是该条款规定的是 Z。请确认哪个写入配置文件？`[前提 flagged — 需验证]`

**规则 2：阈值合理性（劳动法特有）**

| 口述场景 | 检查规则 |
|---------|---------|
| 竞业限制期限超过2年 | 提示：「《劳动合同法》第24条规定竞业限制期限不得超过两年，3年约定超出法定上限，无效部分视为2年」`[前提 flagged — 需验证]` |
| 竞业补偿比例极低（如5%以下） | 提示：「竞业补偿过低可能被认定为无效或导致劳动者有权单方解除竞业协议（具体比例参照地方指引 `[待验证]`）」`[前提 flagged — 需验证]` |
| 试用期月数超过法定上限 | 提示：「根据合同期限，法定试用期上限为 X 个月（《劳动合同法》第19条），用户口述的 Y 个月超出上限，超出部分视为试用期届满后的劳动期限」`[前提 flagged — 需验证]` |
| 经济补偿年限主张超过12年 | 提示：「月工资高于当地上年度月平均工资三倍的，年限封顶12年（第47条）；月工资不超三倍时，无年限上限」`[前提 flagged — 需验证]` |

**规则 3：时效性标记**

对于任何涉及具体法规适用的配置字段，若法规距最后更新日期超过 6 个月，在字段值后附加：

> `[时效性 flagged — 需验证最新法规]`

**规则 4：矛盾回答澄清**

若用户在同一访谈会话中提供矛盾立场（如先说「用人单位方审查」后要求「主张员工权益」），在写入前澄清服务对象立场。

### 6.3 写入前确认流程

对于所有 `[前提 flagged — 需验证]` 标记的字段：

1. 显示检测到的潜在问题
2. 提供两个选项：(a) 按用户口述写入（保留 `[前提 flagged — 需验证]` 标注）；(b) 按建议值修改后写入
3. 不得静默写入未经确认的存疑值
